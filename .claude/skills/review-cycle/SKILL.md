# レビューサイクル

コード品質を担保するための2レビュアー並列レビューサイクル。

## 概要

1. サブエージェント（アーキテクチャ・設計観点）とCodex（実装フィージビリティ観点）を並列起動
2. 両方のフィードバックを統合して修正
3. 再レビューを実行し、両方からLGTMが出るまで繰り返す

## 手順

### Phase 1: レビュー対象の確認

1. **ディレクトリの存在を必ずGlobで確認する**（Exploreエージェントの報告を鵜呑みにしない）
2. レビュー対象のファイル一覧を取得
3. ユーザーにレビュー対象を確認

### Phase 2: 並列レビュー実行

2つのレビュアーを同時に起動する:

#### レビュアーA: サブエージェント（Explore型）
プロンプトテンプレート:
```
以下のディレクトリのコードをレビューしてください。

対象: <ディレクトリパス>

観点:
- アーキテクチャ・設計の一貫性
- 型安全性（asキャスト、Record<string, unknown>の乱用）
- エラーハンドリングの適切さ
- confirm()やalert()の使用（禁止）
- コンポーネントの責務分離
- DRY原則違反
- プロジェクト規約違反（CLAUDE.mdの規約を参照）
- 定義されているが呼び出し元が存在しない関数（dead code検出）

## 信頼度スコアリング
各指摘に0-100の信頼度スコアを付与すること:
- 0: 偽陽性の可能性が高い / 既存の問題
- 25: 問題かもしれないが偽陽性の可能性もある
- 50: 実際の問題だがnitpickレベル
- 75: 高確信。実際に問題を引き起こす / プロジェクト規約で明示的に禁止されている
- 100: 確実。頻繁に問題を引き起こす

**信頼度80以上の指摘のみ報告すること。** 量より質を重視。

結果を以下の形式で出力:
## Critical（必ず修正）[各指摘に confidence: XX を付記]
## Warning（修正推奨）[各指摘に confidence: XX を付記]
## Info（検討事項）[各指摘に confidence: XX を付記]
## 総合判定: LGTM / NOT LGTM
```

#### レビュアーB: Codex

**`-o` は使わず stdout で結果を受け取る**（ファイル出力だと前セッションの古い出力を誤読するリスクがある）:
```bash
codex exec --full-auto --sandbox read-only -C "D:\workspace\hono-practice" "以下のディレクトリのコードレビューをしてください。<ディレクトリパス> を対象に、実装の正確性・エッジケース・パフォーマンス・セキュリティの観点でレビューしてください。各指摘に信頼度スコア(0-100)を付与し、80以上の指摘のみCritical/Warning/Infoに分類して報告してください。最後にLGTM/NOT LGTMの判定を出してください。"
```
Bashツールの出力からそのまま結果を読み取る。バックグラウンド実行時は output_file から読む。

**Codexが構造化された結果を返さなかった場合（ファイル読み取りだけで終了等）、stdoutを再確認してレビュー結果を探す。結果が空ならリトライする。片方のレビュアーだけで完了判定しない。** 実績として、Codex結果をスキップした結果confidence 97の実バグを見逃した事例がある。

### Phase 3: フィードバック統合と修正

1. 両レビュアーの指摘を統合（重複排除）
2. Critical → Warning → Infoの優先順で修正
3. **修正時のファイル分割ルール**: 並列修正する場合は編集対象ファイルが重複しないよう排他的にグループ分け

### Phase 4: 再レビュー

1. 修正内容を含めて再度Phase 2を実行
2. 両方からLGTMが出たら完了
3. NOT LGTMが出たら指摘を修正してPhase 4を繰り返す

## レビュー結果の判断

- **信頼度スコア80未満の指摘はフィルタ済み**: レビュアーは信頼度80以上のみ報告する。それでも偽陽性は混じるため以下で判断する
- **信頼度はラウンドで逓減する**: Round 1-2の指摘は概ね正確。Round 3以降はCriticalの偽陽性率が上がる傾向がある
- **Critical指摘は実コードと突き合わせて裏取り必須**: レビュアーはプロジェクトコンテキスト（DB実カラム名、ランタイム環境特性、プラグイン動作等）を持たない。`tsc`結果や実行結果など客観的証拠で判断する
- 基本は厳しい方に合わせるが、**偽陽性と判断したら根拠を示してスキップする**。正しいコードを壊すリスクの方が高い

## レビュアーの特性（使い分け）

- **サブエージェント（Explore型）**: 設計の一貫性、DRY違反、規約違反、責務分離を拾う。コード読み取り精度は高いが、指摘の重要度判定が甘い傾向がある
- **Codex**: クロスリファレンス（frontendにあるのにmobileにない等）、分岐漏れ、エッジケースに強い。ただしコード読み取り精度が不安定で、存在するコードを「ない」と言うことがある。行番号やコード引用を伴う指摘は、引用が実際のコードと一致しているか特に注意して確認する
- 2つは補完関係にあり、どちらか単体では見つからない問題が組み合わせで見つかる

## 注意事項

- レビュー対象ディレクトリを間違えない（正しいパスは `apps/frontend`）
- Codexの `-C` パスは `D:\workspace\hono-practice` を使う
- Codexは `-o` を使わず stdout で結果を受け取る。ファイル出力は前セッションの古い出力を誤読するリスクがある
- 修正後の再レビューでは、前回の指摘が解消されたことを明示的に確認させる
- **計画書レビューと実装レビューは別物**: 計画書のレビュー済み≠実装コードのレビュー済み。大規模リファクタリングでは実装後にも必ず回す
- Warningでもプロジェクト規約違反（`as`キャスト等）は修正すべきと提案する。「スコープ外」で先送りしない
