# 並列エージェント運用

## エージェントのプロンプトに含める共通ルール
```
- 作業ディレクトリ: <worktreeパスまたはリポジトリルートの絶対パスを必ず明示>
- APIパス末尾スラッシュなし / DB: Neon Postgres
- 1ファイル200行以内 / バレルindex.tsでエクスポート
- 古いファイルは削除（再エクスポートとして残さない）
- Repository命名: ドメイン名を含める / 型定義: type（interfaceではない）
```

## worktree環境でのエージェント起動
- **作業ディレクトリは絶対パスで明示必須**（`.worktrees/<name>/` のフルパス）。省略するとエージェントがリポジトリルートに書き込み、rootに差分が漏れる（3/29の教訓）
- プロンプト先頭に `作業ディレクトリ: D:/workspace/hono-practice/.worktrees/<name>` を含める
- `pnpm run tsc` 等の検証コマンドもworktreeパスで `cd` してから実行するよう指示する

## 計画時のチェック
- **「Webでやることは Mobile でもやる必要があるか？」を必ず自問する**（計画段階でmobileのUI・sync・リポジトリへの影響を確認。漏れが4セッション連続で発生した教訓）
- 提案・比較テーブルを出す際は、Web/Mobile両方の影響をデフォルトで含める

## Explore → Implement 分離
Medium以上のタスク（2ファイル以上の変更、新機能、リファクタ）では、実装前にExploreエージェントを起動する。

### Exploreエージェントの制約
- **使用可能ツール**: Read, Glob, Grep, Bash（read-onlyコマンドのみ）
- **禁止**: Write, Edit, MultiEdit
- **目的**: 実装せず調査のみ。ターン数削減に直結する

### Explore出力テンプレート
```
## 影響分析
- 変更対象ファイル一覧（パス + 現在の行数 + 変更種別: 新規/修正/削除）

## 既存パターン
- 類似機能の実装パターン（参照ファイルパス:行番号）
- 使用すべきファクトリ関数・ユーティリティ

## 依存関係
- import/exportの依存グラフ（変更が波及するファイル）
- テストファイルの対応関係

## リスク
- 200行超えになりそうなファイル
- 型の破壊的変更が必要な箇所
- Web/Mobile両方への影響
```

### Implementエージェントへの引き継ぎ
Explore結果をプロンプト先頭に含める: 「以下はExploreエージェントの分析結果です。これに基づいて実装してください。」

## エージェント起動設定
- **mode**: 編集を伴うエージェントは `bypassPermissions` を指定する（権限ダイアログでブロックされて失敗する教訓。3/28）
- **model選択基準**:
  - **sonnet**: デフォルト。ほとんどのタスクはsonnetで十分
  - **opus**: 複雑なロジック設計、アーキテクチャ判断、思考を要求するタスク
  - **haiku**: 単純作業（文字列抽出、パターン置換等）を並列で大量実行する場合

## 競合防止
- 編集対象ファイルが重複しないよう事前にGlobで排他的にグループ分け
- **インターフェース変更は先に直列で実施**し、結果を後続エージェントのプロンプトに含める
- ファイル移動時は**テストファイルのimportパス更新**もプロンプトに含める
- frontendの作業では `apps/frontend/CLAUDE.md` のルールも含める

## プロンプト作成時の注意（日記3/25, 3/26の教訓）
- **DB enum値・ドメイン固有の定数値はプロンプトに明示的に含める**（planカラムの値が`"premium"`なのにエージェントが`"pro"`を使ったバグ。自分が実装した情報でもプロンプトに書かなければエージェントは知らない）
- **セキュリティクリティカルなコード（署名検証・トークン処理等）はエージェントに丸投げしない**。自分で書くか、プロンプトで具体的な実装仕様を指定する（Webhook署名検証でタイミングアタック脆弱性・層構造違反等が発生した教訓）

## エージェント出力の扱い（日記3/16, 3/19の教訓）
- **エージェントの完了報告は信用しない**。必ず自分で `test-once` → `tsc` → `fix` を回す（エージェントが「全パス」と報告しても型エラー残存の実績あり）
- **調査は委譲、編集は自分**: パターンが明確な一括修正（`as any` → `as unknown as X` 等）は `replace_all` の方が確実。noExplicitAnyで3回やり直した教訓
- **型が通るか1ファイルで確認してからファイル一括差し替え**: zValidatorラッパーで15ファイル差し替え→型エラー→全revertの教訓
- **自分が詰まった問題も別エージェントに委譲してみる**: 同じアプローチに固執しがち。別エージェントは別の解法を試す（premiumMiddleware型修正で実証）
- **キャスト・型エラーの修正は「宣言箇所」で解決できないか先に検討する**: サブエージェントは指摘箇所をピンポイントで直す傾向がある。親は「なぜこのキャストが必要か」という上位の問いを立て、型宣言の修正で根本解決できないか確認してから委譲する（`as unknown as` → ref型パラメータ拡張で利用側キャスト全廃できた教訓。3/28）
