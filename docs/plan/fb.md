# Actiko レビューまとめ

## 前提の再整理

- Actiko は純粋な小規模収益アプリとしてだけでなく、**ポートフォリオ**、特に**AI 開発やコーディングエージェント（CA）を用いた持続開発の素振り**としての意味合いが強い
- そのため、構成の大きさや APM / エラー収集 / subscription gateway / 課金テーブル設計などには、**事業実装と同時に技術検証の意味**がある
- この文脈では、以前の「オーバーエンジニアでは」という指摘は一部当たるが、**欠点というより意図された設計判断**として理解すべき
- また、以下はユーザー側から補足済み
  - Web / frontend の型安全性は改善済み
  - モバイル配布まわりは現在進行形で整備中
  - `.env.local` は credentials を保持しておらず、運用事故リスクは実態上低い
  - 課金導線は未完成だが、subscription 実装自体はポートフォリオ上の技術検証を目的としている

---

## 1. アーキテクチャレビュー

### 総評

**このアーキテクチャで持続開発は可能。**
ただし、これは「小さく早く売るために最適」という意味ではなく、 **"継続的に開発・検証・改善していくための強い土台がある"** という意味で評価している。

### 良い点

- `apps/backend`, `apps/frontend`, `apps/mobile`, `apps/tail-worker`, `packages/*` の責務分離が明確
- `packages/domain`, `packages/frontend-shared`, `packages/sync-engine`, `packages/platform` への共通化方針が一貫している
- ADR や設計ドキュメントがあり、**設計意図がコード外に残っている**
- オフライン前提 + 同期という難しいテーマに対して、Web/Mobile の両方で構造的に向き合っている
- subscription を「ただの画面」ではなく、**FE / BE / DB をまたぐ関心事**として扱っている

### 懸念点

- 技術構成はかなり本格的で、**月2万円規模の収益目標に対しては重い**
- Web / Mobile の UI 実装にはまだ重複が残っており、 **機能追加時に両側へ修正が飛びやすい**
- モバイルリリースまわりは進行中で、**技術土台はあるがプロダクション運用としてはまだ仕上げ段階**
- 綺麗な設計を維持しようとするほど、**プロダクト完成よりアーキテクチャ改善に時間を使いやすい**

### この文脈での結論

Actiko の構成は、一般的な「小さな個人アプリ」に対しては重めだが、
**"CA を前提にした中長期開発の訓練場" としてはむしろ筋が良い**。

したがって、

- **構成の大きさ自体を改善対象にする必要はない**
- 改善対象はむしろ **UI 重複の削減** と **課金・販売導線の実装順の整理**

---

## 2. 収益性レビュー（月2万円を持続的に稼げるか）

### 結論

**月2万円は十分に現実的なライン。**
ただし現状は、**「売れるプロダクト」ではなく「売れる可能性がある土台」**の段階。

### ポジティブな理由

- 金額目標そのものは大きくない
- Actiko には「活動ごとに最適化された記録モード」「オフライン前提」「多様な入力 UI」という**独自性の芯**がある
- 広く浅く刺すより、**狭いが強いニーズのある少数ユーザー**を取る方向と相性が良い
- 技術基盤は十分にあり、販売導線さえ整えば実験を回せる

### 現状の不足点

- 課金導線そのものは未完成
- subscription はあるが、**実際にお金を払う end-to-end の体験**は未実装
- 何に対して払うのか、つまり **paid reason** がまだ薄い
- 課金分析、転換分析、paywall 最適化など、**売るための計測と実験系**はこれから

### 解釈

「稼げるか？」に対する答えは Yes だが、
その Yes は **現状のままでも売れる**という意味ではなく、
**いまのコードベースなら、正しい順番で monetization を実装すれば月2万円は十分狙える**という意味。

---

## 3. `docs/plan/monetization-strategy.md` はそのまま進めてよいか

## 結論

**思想としては採用してよいが、そのまま実行計画として使うのは非推奨。**

言い換えると、

- **戦略文書としては良い**
- **実行順を決める backlog としては修正が必要**

---

## 4. 計画書の良い点

### 4.1 無料と有料の境界線が良い

計画書では、

- レイヤー 1-2: 記録体験（無料）
- レイヤー 3-5: 増幅価値（有料）

という整理になっている。

これは妥当。

Actiko の価値の芯は「記録そのもの」にあるため、ここを削るとアプリの個性が消える。
一方で、Goal / 可視化 / AI などは「記録を増幅する価値」なので課金対象として自然。

### 4.2 ブロックよりプレビューを重視している

「使えない」ではなく「見えるが深い部分が閉じている」という売り方は、Actiko の性質に合っている。
このアプリは記録後に価値が立ち上がるため、**無料で体験してから欲しくなる導線**の方が強い。

### 4.3 entitlement 中心の設計思想が正しい

subscription を UI だけで閉じず、
FE / BE / DB / plan matrix / premium middleware まで跨って扱う設計は正しい。

これはポートフォリオとしても強く、将来の実運用にもつながる。

---

## 5. 計画書をそのまま実行しない方がいい理由

### 5.1 課金開始を AI に依存しすぎている

現在の計画書は、実質的に

- 記録機能を整える
- AI や高度分析を整える
- その後に課金基盤を育てる

という流れになっている。

これは、**monetization の開始タイミングを最も重い機能群に依存させている**。

しかし現状コードにはすでに、

- Goal
- Heatmap
- Stats
- CSV
- Subscription schema
- useSubscription

など、有料化しやすい資産がある。

したがって、**AI 完成待ちで課金を始める必要はない**。

### 5.2 Activity Kinds を最初の有料要素にするのはやや重い

Activity Kinds はアプリ全体に入り込む概念なので、ここを最初に gate すると、

- Web / Mobile 双方で影響が広い
- 記録体験の根幹を削ったように感じられやすい
- 実装や説明のコストが高い

最初の paid feature としては、**制限のかけやすい "量" や "履歴" の方が向いている**。

### 5.3 KPI が今の目標に対して大きすぎる

計画書は MRR 100k などを見ているが、今回の目標はまず **月2万円の持続**。

この段階では、
大きな KPI よりも **「このアプリに払う 30〜40 人前後の継続ユーザーが成立するか」** を検証する方が重要。

---

## 6. 推奨方針

## 結論

**戦略は維持し、実行順だけ変える。**

### 6.1 まず無料版として出すのはあり

初回リリースを free-only にするのは合理的。

ただし、中途半端に

- Premium の匂いだけある
- upgrade CTA だけある
- でも買えない

という状態にはしない方がよい。

**売らないなら free-only として出す。**
**売るなら最初から billing を通す。**

### 6.2 先に作るべきなのは paywall ではなく entitlement ロジック

UI より前に、shared な関数として plan 依存の挙動を切り出すべき。

例えば：

- `canCreateGoal(plan, currentGoalCount)`
- `visibleHeatmapDays(plan)`
- `availableStatsGranularities(plan)`
- `canUseCsv(plan)`
- `voiceDailyLimit(plan)`

こうすると、Web / Mobile の UI 重複改善とも相性が良い。

### 6.3 最初の paid bundle は既存価値の拡張に絞る

v1 の paid bundle は以下が良い。

- 2つ目以降の Goal
- ヒートマップ全履歴
- 長期統計（週次 / 四半期 / 年次）
- CSV export
- API Key は補助的価値

逆に、以下は後回しが妥当。

- Activity Kinds の本格ゲート
- AI 週次レビュー
- 音声モード無制限
- パターン検出
- Lifetime プラン

### 6.4 AI は monetization の核ではなく第2段階に置く

AI は差別化には強いが、最初の課金成立条件にすると重い。

最初の月2万円ラインは、

- Goal 数制限
- 履歴可視化
- 長期統計

で十分狙える。

AI はその後、

- 単価の正当化
- 継続率向上
- 年額プラン誘導
- 差別化の強化

に使う方が良い。

## 7. Trial / プラン構成について

### Trial

計画書の「登録時 14 日無料」は思想としてはよいが、
Web と Mobile で同一概念として扱うと設計が重くなる。

したがって、

- Web: account-based trial
- Mobile: store 側の trial / offer に寄せる

というふうに、**体験は揃えても実装概念は分けた方が良い**。

### プラン構成

最初から

- Monthly
- Annual
- Lifetime
- Enterprise

まで公開 UI に乗せる必要は薄い。

初期はシンプルに

- Free
- Pro Monthly
- Pro Annual

で十分。

Lifetime はポートフォリオ上は schema に残してもよいが、
**公開商品としては後回し**が妥当。

---

## 8. 今の Actiko で最初に切るべき課金ポイント

### 最優先

- Goal 数制限
- ヒートマップの履歴期間
- 統計の長期表示

### 次点

- CSV export
- API Key

### 後回し

- Activity Kinds の課金
- AI 系全部
- 音声無制限
- Lifetime

---

## 9. 最終まとめ

### アーキテクチャについて

- Actiko は一般的な小規模アプリに対しては重い
- ただし、**ポートフォリオ兼 CA 開発訓練場**という文脈ではその重さは合理的
- 改善対象は構成そのものではなく、**Web / Mobile UI 重複の圧縮**

### 収益化について

- 月2万円は十分狙える
- ただし、現状は「売れる土台」がある段階であり、「売れる導線」がある段階ではない
- したがって、これから必要なのはアーキテクチャ改善より **paid reason の明確化と billing の end-to-end 実装**

### monetization-strategy について

- **思想は正しい**
- **そのままの順番では進めない方がいい**
- AI 完成待ちで monetization を始めず、まず既存価値を課金可能な形に切り出すべき
- 戦略は維持し、実行順を変えるべき

### 一言で要約すると

**`docs/plan/monetization-strategy.md` は「何で稼ぐか」を考える文書としては良い。
ただし「どの順番で実装して公開するか」の文書としては、AI 側に寄りすぎている。
Actiko の現在地なら、最初の月2万円は Goal / Heatmap / Stats を中心とした軽い paid bundle で取りにいくのが最も合理的。**
