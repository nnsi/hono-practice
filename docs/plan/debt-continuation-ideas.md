# Debt システムを軸にした「記録が続く」設計アイデア

> 作成: 2026-03-12
>
> 前提: `product-vision.md` の Debt システム定義 + 以下の議論に基づく
>
> - 入力導線は現状維持（自分のペインに最適化）
> - 継続の仕組みは万人向けに広げる（自分も「続かない人」）
> - 記録モード・音声AI・ウィジェットは競合にもある。Debt システムが Actiko 唯一の明確な差別化

---

## 「記録が続かない」要因の整理

入力の面倒さは記録モードで解消済み。残る要因:

| # | 要因 | 状態 |
|---|------|------|
| 1 | 記録しても何も返ってこない（報酬ゼロ） | **未解決** |
| 2 | 途切れたら「もういいや」（完璧主義の罠） | Debt の概念が解決するが、設計詳細が未定 |
| 3 | 負債が溜まりすぎて絶望する | **未解決**（Debt が逆効果になるリスク） |
| 4 | 今日何をすればいいかわからない | **未解決** |
| 5 | 長期的に飽きる・意味を見失う | **未解決** |

以下、各要因に対する Debt ベースのアイデア。

---

## 要因1: 記録しても何も返ってこない

### 1-A. 記録直後の Debt フィードバック

記録を保存した瞬間に、Debt の変動を1行で表示する。

```
記録しました: 腕立て 30回
─────────────
  目標 30回 → 達成 ✓
  負債: 15回 → 0回（完済！）
```

```
記録しました: 読書 15分
─────────────
  目標 30分 → 残り 15分
  負債: 45分 → 30分に減少
```

**なぜ効くか**: 記録行為に即時報酬を与える。「数値を入力した」→「負債が減った」の因果が毎回見えるので、記録すること自体に意味が生まれる。

### 1-B. Surplus（貯金）の可視化

目標を超過した分を「貯金」として表示する。

```
今日の読書: 45分（目標 30分）
  → +15分の貯金！通算貯金: 2時間30分
```

**なぜ効くか**: 負債がない状態でも「もうちょっとやろう」のインセンティブになる。借金返済だけでなく、蓄財する楽しさ。

### 1-C. 完済の瞬間を祝う

特定のアクティビティの Debt がゼロになった瞬間に、専用の演出を出す。

- ヒートマップがパッと緑に光る
- 「読書の負債を完済しました」のトースト
- 完済までにかかった日数を表示（「5日間で返済」）

**なぜ効くか**: ストリーク達成は「維持のプレッシャー」だが、完済は「終わりのある目標の達成」。達成感の質が違う。

---

## 要因2: 途切れたら「もういいや」

### 2-A. 回復ストリーク

従来のストリーク（連続達成日数）に加え、**連続 Debt 返済日数**を追跡する。

```
通常ストリーク: 🔥 0日（昨日途切れた）
回復ストリーク: 💪 3日連続で負債を減らしています
```

**なぜ効くか**: 通常ストリークが切れても、別のストリークが始まる。「ゼロからのやり直し」ではなく「回復中」という状態が存在するので、復帰の心理的コストが下がる。

### 2-B. 部分達成の承認

目標未達でも、やった分だけ Debt が軽減されることを明示する。

```
目標: 腕立て 30回
今日: 10回
─────────────
  未達だけど、負債は 30回ではなく 20回だけ
  （やらなかった場合より 10回分マシ）
```

**なぜ効くか**: 「全部やらないなら意味がない」という心理を崩す。10回でも記録する価値がある、という認知を作る。

---

## 要因3: 負債が溜まりすぎて絶望する

Debt システム最大のリスク。旅行や病気で1週間空けたら、負債が爆発して二度と開かなくなる。

### 3-A. Debt キャップ（負債上限）

アクティビティごとに Debt の上限を設定する。上限を超えた古い Debt は自動消滅する。

```
設定: 読書の最大負債 = 7日分（210分）
─────────────
10日間放置しても、負債は 210分のまま。
3日分（90分）は自動的に消える。
```

**デフォルト値の提案**: 7日分。「1週間で取り返せる」と思える範囲。ユーザーがカスタマイズ可能。

**なぜ効くか**: 「どれだけサボっても最大これだけ」という天井があると、復帰のハードルが下がる。借金に上限がある安心感。

**実装方針**: `activity_goal` テーブルに `debt_cap numeric` カラムを追加。既存の sync フローにそのまま乗る。

### 3-A'. 曜日別目標（day_targets）

曜日ごとに目標値を変える。基本は `daily_target_quantity` だけで運用し、必要な人だけ `day_targets` で曜日別に上書きする。

```
基本: daily_target_quantity: 30, day_targets: null
  → 全曜日 30分

詳細: daily_target_quantity: 30, day_targets: { 6: 60, 7: 0 }
  → 月〜金 30分、土 60分、日 休み
```

Debt 計算: `その日の目標 = day_targets[曜日] ?? daily_target_quantity`

フリーズ（例外的な不在）とは別の概念。フリーズが「旅行で来週休む」なら、day_targets は「毎週土日はやらない」。`day_targets` に `0` を指定すれば休みになるため、`active_days` は不要。

**実装方針**: `activity_goal` テーブルに `day_targets jsonb` カラムを追加。NULL の場合は全曜日 `daily_target_quantity` が適用される。sync はカラム追加のみで既存フローに乗る。UIはゴール設定画面に「曜日別に設定」の詳細導線を用意し、大多数のユーザーは `daily_target_quantity` だけで完結する手軽さを維持する。

**スコープ外**: 「筋トレ後2日休む」のような動的パターンは曜日スケジュールでは扱わない。日次目標との相性が悪い活動は、ゴールを設定せず記録モード + Stats で振り返る使い方が適切。

### 3-B. 復帰モード

一定期間（例: 3日以上）記録がないアクティビティを開いた時、通常の Debt 表示ではなく復帰用UIを出す。

```
┌─────────────────────────┐
│  おかえりなさい           │
│                         │
│  読書: 5日間の負債       │
│                         │
│  [ 今日から再開する ]     │
│  [ 負債をリセットする ]   │
│                         │
│  ※ リセットしてもログは   │
│    消えません             │
└─────────────────────────┘
```

**なぜ効くか**: 久しぶりにアプリを開いた人に「お前サボったな」ではなく「おかえり」と言う。リセットの選択肢を明示的に与えることで、復帰の意思決定を軽くする。

### 3-C. Debt フリーズ（一時停止）

旅行・病気・繁忙期などに、特定アクティビティの Debt 蓄積を一時停止する。

```
読書: ⏸ 一時停止中（3/10〜3/14）
  → この期間は負債が増えない
  → 記録した場合は貯金として加算される
```

**なぜ効くか**: 「やれない期間がある」ことを計画的に扱える。罪悪感なく休める仕組み。

**設計方針**:

- ゴール設定画面に常時フリーズ設定欄を配置する（「一時停止中」の時だけ出すのではなく常にアクセス可能）
- 開始日・終了日を事前に設定可能（例: 来週の旅行を今から登録）
- 終了日未定も許可（病気など終わりが読めないケース）
- フリーズ中はホーム画面に ⏸ アイコンが表示されるため、再開忘れに気づける
- フリーズ期間中は「目標がない」扱い。ヒートマップでは達成でも未達でもない状態（目標なし）として表示

**実装方針**: `activity_goal_freeze_period` テーブルを新設（`id`, `goal_id`, `user_id`, `start_date`, `end_date`, `created_at`, `updated_at`, `deleted_at`）。フリーズ履歴を複数保持でき、ヒートマップで過去のフリーズ期間を正しく表示できる。sync は `syncGoalFreezePeriods.ts` を新設し、goals の後に同期する依存順序で組み込む。

## 要因4: 今日何をすればいいかわからない

### 4-A. Debt 優先ダッシュボード

ホーム画面で、Debt が大きい順にアクティビティを並べる。

```
📊 今日の優先度

1. 読書      負債 90分  │████████░░│
2. 筋トレ    負債 20回  │███░░░░░░░│
3. 音ゲー    貯金 +500  │達成済み ✓ │
```

**なぜ効くか**: 「全部やらなきゃ」ではなく「上から順にやれば最大効果」という指針を与える。意思決定の負荷を下げる。

### 4-B. ワンアクション提案

アプリ起動時に、最も効果的な1アクションを提案する。

```
💡 今日これだけやれば
   読書 30分 → 負債が半分に
```

**なぜ効くか**: 選択肢を1つに絞ることで行動の閾値を下げる。「とりあえずこれだけ」で始められる。

## 要因5: 長期的に飽きる・意味を見失う

### 5-A. Debt トレンドの方向表示

現在の Debt 量だけでなく、**方向（増加中/減少中/安定）** を表示する。

```
読書: 負債 60分 ↘ 減少中（先週比 -30分）
筋トレ: 負債 10回 → 安定
音ゲー: 貯金 200 ↗ 増加中
```

**なぜ効くか**: 負債があっても「減っている」なら肯定的に捉えられる。絶対値より変化の方向がモチベーションに効く。

### 5-B. 節目の振り返り

一定の記録量に到達した時点で、自動で振り返りを出す。

```
🎉 読書の累計が 100時間に到達しました

  最初の記録: 2026-01-15
  ここまで: 57日間
  1日平均: 約105分
  最大負債だった日: 2/3（180分）→ 2/10に完済
```

**なぜ効くか**: 数値の積み上がりを実感する機会を作る。「続けてきた」という事実が見えると、やめるのがもったいなくなる（サンクコスト効果の正の活用）。

### 5-C. 目標の自動提案

Debt が慢性的に溜まるアクティビティに対し、目標値の調整を提案する。

```
読書の目標: 30分/日
直近30日の達成率: 40%

💡 提案: 目標を 20分/日 に下げませんか？
   → 達成率の予測: 75%
   → 負債の蓄積ペースが半分に

   [ 20分にする ] [ このまま ]
```

**なぜ効くか**: 慢性的な Debt は「目標が高すぎる」サイン。Debt システムが救済として機能するだけでなく、目標設定自体を最適化するフィードバックループになる。

## 実装優先度の提案

Debt システムの基本実装（日次目標・負債蓄積・超過返済）が前提。その上で:

### Tier 1: Debt の基本体験を「続く仕組み」にする

| アイデア | 理由 |
|---------|------|
| 1-A. 記録直後の Debt フィードバック | 最小実装コストで毎回の記録に報酬を与える。Debt 体験の根幹 |
| 2-B. 部分達成の承認 | 表示の工夫だけで実装可能。「やらないよりマシ」の認知を作る |
| 3-A. Debt キャップ | 負債が破綻しない安全装置。これがないと Debt が逆効果になる |
| 3-C. Debt フリーズ | 旅行・体調不良など予測可能な不在に対する予防策。キャップが「事後の救済」なのに対し、フリーズは「事前の予防」。両方揃って初めて Debt が安全に機能する |

### Tier 2: 一般ユーザーへの訴求力を上げる（重点3機能）

スクリーンショット・App Store 説明文・初回起動で価値が伝わる機能。

| アイデア | 訴求力の根拠 |
|---------|-------------|
| 2-A. 回復ストリーク | 「ストリーク切れても回復できる」→ 競合との違いが一文で説明できる。最大の訴求メッセージ |
| 4-A. Debt 優先ダッシュボード | ホーム画面のスクショで「今日何をすべきか一目でわかる」が伝わる |
| 1-C. 完済演出 | Debt を「追い詰められる体験」ではなく「返し切る達成感」にする。スクショ映え |

### Tier 2.5: 訴求力はあるが次点

| アイデア | 理由 |
|---------|------|
| 4-B. ワンアクション提案 | 4-A と組み合わせると自然だが、4-A 単体でも機能する |
| 1-B. Surplus の可視化 | Debt がない状態の人向け。訴求の優先度は下がる |
| 3-B. 復帰モード | 離脱後にしか体験できず、訴求材料としては使いにくい |
| 5-B. 節目の振り返り | 長期ユーザー向け。初期の訴求より口コミフェーズで活きる |

### Tier 3: データ蓄積後に効く長期施策

使い込んで初めて価値が出る。retention には効くが acquisition には効きにくい。

| アイデア | 理由 |
|---------|------|
| 5-A. Debt トレンド方向表示 | 数週間分のデータが必要。既存ユーザーの継続率向上 |
| 5-C. 目標の自動提案 | 慢性的な Debt パターンの検出にデータ蓄積が必要 |

## マネタイズとの関係

`monetization-strategy.md` では Debt バランス / ストリークは Free に含まれている。

上記アイデアの Free / Pro 配置案:

| Free | Pro |
|------|-----|
| 記録直後の Debt フィードバック (1-A) | Surplus 可視化 (1-B) |
| 部分達成の承認 (2-B) | 回復ストリーク (2-A) |
| Debt キャップ (3-A) | Debt トレンド方向表示 (5-A) |
| 復帰モード (3-B) | 目標の自動提案 (5-C) |
| Debt 優先ダッシュボード (4-A) | 節目の振り返り (5-B) |
| ワンアクション提案 (4-B) | |
| 完済演出 (1-C) | |
| Debt フリーズ (3-C) | |

**原則**: 「続けるための仕組み」は Free。「続けた結果を深く理解する仕組み」は Pro。
