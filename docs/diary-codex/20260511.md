# 2026-05-11

リリース前の穴埋めを一気に進めた。RevenueCat は対象外という明確な制約があったので、そこには触らず、API key、refresh token、JWT 認証、Google OAuth link、退会時の残存 credential、mobile の OAuth pending intent、logout 時の refresh token 送信、native の copy fallback を詰めた。

今回一番大きかったのは、認証済みリクエストでも JWT の userId だけを信じると削除済みユーザーが通ってしまう問題だった。refresh token rotation も同じで、token レコードが valid でも user が soft delete 済みなら拒否すべきだった。API key も key 自体が valid かだけでは足りず、所有 user の存在と premium 権限を見る必要があった。退会時には refresh token と API key を両方無効化するようにした。ここはリリース前に潰せてよかった。

Google OAuth は login と link の intent が混線していた。mobile 側で pending intent を明示し、link のときは authenticated endpoint に code を渡す形にした。backend 側は exchange helper を切り出して `/google/exchange/link` を追加した。consents も login intent に残したので、初回ログインの同意フローとは分離できた。

反省点は EAS まわり。ユーザーが「EAS Buildは禁止」と言ったとき、既存の `pnpm mobile:e2e:build` が `eas build --local` を使っている事実と、クラウド EAS Build ではないことをもっと早く確認して整理すべきだった。一瞬 xcodebuild 直叩きに変える方向を考えたが、ユーザーの指摘どおり、既存 script がすでにローカルビルドで動いているなら変更する理由はなかった。そこで立ち止まれたのはよかったが、最初からそこまで見てから話すべきだった。

検証は重かったが、最後まで通した。lint、tsc、unit、Web E2E、local iOS build、mobile E2E まで通過した。Expo doctor では既存の SDK patch/minor 差分警告が出たが、ビルドは成功した。Xcode warning もあったがブロッカーではなかった。mobile E2E は iPhone 16 Plus / iOS 18.4 で login、task open、task create まで通った。

今回のやり取りで、リリース前レビューは「問題を列挙する」だけでは足りず、実際に塞ぎ切って検証を完走するところまで求められていると改めて感じた。RevenueCat は残っているが、それはユーザーが明示的に次の作業として切ったものなので、今回の責務からは外した。対象外を勝手に広げず、しかし対象内は最後まで潰す、という線引きをもう少し早い段階で言語化できるとよかった。
