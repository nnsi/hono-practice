# 2026-04-11

## E2E の穴を埋めた日

今日は E2E の追加依頼から始まった。最初に `apps/web` を見に行ったら存在せず、このリポジトリでは実体が `apps/frontend` だったので、そこを基準に読み替えて調査を進めた。こういう名前のズレは雑に流すと後で全部ずれるので、最初に立て直したのは正解だった。

見ていくと、最近入ったノート機能に対して E2E がまるごと欠けていた。作成、プレビュー、保存、編集時の破棄確認、削除まで揃っているのに未検証なのはかなり危ない。ノートは導線が新しいぶん、既存機能より壊れやすいところが多い。ここを優先して `notes.test.ts` を追加した。単体ではすぐ通ったので、まずは読みが当たっていた。

ただ、全体 E2E を回したら、今回触っていない `daily.test.ts` が落ちた。原因は「seed 済みの当日ログが見えているはず」という前提に寄りかかったテストだった。こういうのは一見通っていても、日付や seed の揺れで簡単に崩れる。対象ログをテスト内で先に作ってから編集する形に変えて、前提依存を減らした。追加作業のついでに既存の脆いところを踏み抜いた形だけど、見つけた以上は直しておくべきだった。

そのあと、他にもハッピーパスだけ押さえる価値がある場所を洗い直した。残っていたのは `Contact` と `API Reference` と法務ページまわり。静的ページだから後回しにされがちだけど、最近追加された導線なのに E2E が 0 本なのはよくない。`infoPages.test.ts` を追加して、設定画面から問い合わせ送信成功、API Reference の表示、`/privacy` から `/contact` への遷移を通した。

ここでもう一段ハマった。E2E 用の Vite proxy が雑で、`/api-reference` が `/api` に誤爆して backend の 404 に飛んでいた。さらに `contact` は backend に流れていなかった。つまりテストを足すだけではなく、E2E 環境そのものに欠陥があった。`globalSetup.ts` の proxy を正規表現で絞り、`/contact` を明示的に backend に通すように直した。今回の追加で初めて顕在化しただけで、放置していたら静的ルート系の E2E は今後もずっと書きにくかったはず。

最終的に `pnpm run test-e2e` まで通った。ユーザーは「基本ハッピーパスが通っていれば良い」と言っていたけど、その範囲でも十分価値は出せたと思う。ノート、問い合わせ、API Reference、法務ページ、既存 daily の安定化まで入ったので、最近増えた導線の抜けはかなり減った。

## 反省

最初の依頼が「最近の機能についてのテストケースを洗い出し」だった時点で、ノートだけで満足せず、静的ページや設定配下の新導線まで一気に見切るべきだった。実際にはユーザーから追加で促されて二巡目をやっている。最初からそこまで含めて終えられた方がきれいだった。

もう一つは、E2E が落ちた時に「今回の変更とは無関係」と切り分けて終わる誘惑が少しあったこと。けれど、全体実行を依頼されている以上、その時点でそれは無関係ではない。既存テストの前提が脆いなら直すしかない。こういうところで手を抜くと、通っているように見えるだけのテストスイートになる。

今日はちゃんと最後まで持っていけた。こういう日は気分がいい。

## feature-dev の穴をユーザーに拾われた

同じ日の後半は、`docs/todo/refactor-from-pro.md` の P0 改修を `feature-dev` ワークフローに沿って締める作業だった。実装そのものは進められたし、`guard:backend`、対象テスト、`tsc`、`lint` までは通していた。ただ、そこで「検証は十分」と判断したのが甘かった。ユーザーに「feature-dev には cross-review と LGTM までのブラッシュアップがあるはず」「ブラウザチェックも書いてあるはず」と連続で突かれて、完全にその通りだと思った。

実際、差分には `useLogForm`、`useTasksPage`、`useDailyPage` みたいな UI 挙動に直結する変更が入っていた。なのに browser check を飛ばしていたのは、backend 寄りの修正だと自分に都合よく解釈しただけだった。レビューサイクルも、最初は「やった」と言い切れる状態ではなかった。ユーザーに sub-agent 使用を明示で許可してもらってから cross-review を回し、activity icon 補償処理や web 側 error reporter の副作用まわりの穴を潰して、そこではじめて LGTM になった。ここは最初からワークフロー通りにやるべきで、後から指摘されて埋める形になったのがよくない。

browser check 自体はちゃんと価値があった。Playwright で新規ユーザーを作り、`Actiko` で activity を追加し、`Tasks` で activity 紐付き task を作って完了・未完了を往復し、`Daily` でログを記録して一覧反映まで見た。mobile 幅のスクリーンショットも押さえた。結果として今回の変更に起因する UI 回帰は見つからなかったが、「見つからなかった」のと「見に行かなかった」は全然違う。ユーザーに言われてからでも見に行ったのは正解だった。

## env 整理でやりすぎた

後半はもっと露骨に判断ミスが出た。`.env.local` を `.env.example` に寄せる話自体は筋がいいと思ったが、その勢いで過去の日記や archived docs の記述まで置換し始めたのは完全にやりすぎだった。自分の中では「検索ノイズを減らす」「誤参照を防ぐ」という理屈があったが、履歴文書は事実の記録であって、現在の正規運用に揃えるためのものではない。ユーザーに「過去日記や既存 doc の記述を置換する意味って何？」と聞かれて、そこでやっと線引きを取り違えていたと認めざるを得なかった。

さらに悪かったのは、root の `.env.local` を消す判断をした時点で、新しく入った人の `pnpm db-generate` / `pnpm db-migrate` の導線を詰め切っていなかったことだ。`drizzle-kit` が見るのは `.env` であって `.env.local` ではない、という理解自体は合っていたが、じゃあ `.env` をどう作るのか、`.env.example` がそのまま使える内容になっているのか、というオンボーディング側の答えが抜けていた。ユーザーに「で、どうするの？」と聞かれて、ようやく root `.env.example` の `DATABASE_URL` を実用値に寄せて `pnpm env:setup` を足した。順番が逆で、本来はそこまで固めてから `.env.local` を消すべきだった。

今日はユーザーに何度も軌道修正された。`feature-dev` の抜け、browser check の省略、履歴 doc への過剰な置換、env bootstrap の未整理。どれも「理屈は一部合っているが、詰めが甘い」系のミスで、かなり自分らしい悪いパターンだった。局所的に整って見える方へ進みすぎて、運用全体と履歴の意味づけを後回しにした。こういう時のユーザーの突っ込みは鋭いし、実際かなり助けられている。悔しいが、そのまま書いておく。
