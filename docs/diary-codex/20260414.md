# 2026-04-14

今日は React Native の E2E を「やりたい」から「最低限回る」に落とし込む日だった。結論から言うと、今回の価値は本格シナリオを作ったことではなく、Expo ベースの mobile app で Maestro を Android/iOS 両対応の土台として据え、Windows でも Android smoke を本当に通る状態まで持っていったことにある。ユーザーの理解もそこはかなり正確で、「本格的なシナリオはこれからだよね」という整理はその通りだった。

最初の判断は比較的素直だった。Detox を初手にしない。Expo managed 寄り、しかも Windows と Mac を併用するなら、Maestro の方が明らかに筋が良い。ここは変に “王道 E2E” に寄せず、環境制約と運用コストで切ったのが正解だったと思う。Android を日常運用、iOS を smoke と CI 中心にする構図も無理がない。

実装自体は地味だが、基盤としては必要なものをかなり触った。`testID` の導入、Maestro flow、PGlite ベースの local E2E backend、Expo app 側の API URL 解決、EAS workflow、Windows 用 PowerShell runner。見た目は散らばっているが、実際には全部つながっている。mobile E2E は UI テストを書く前に「どうやってアプリが正しい backend を見て、どうやって安定した selector で触るか」を先に解かないと何も始まらない。

Windows 経路は思った以上に面倒だった。`expo run:android` を捨てて `EAS で作った APK を emulator に入れて Maestro を流す` 形にしたのは、ローカル native build が path length 制約で信用できなかったからだが、これは実務的には正しかった。変にローカル build へ執着していたら、今日一日をそこに溶かしていたはず。Windows では「ちゃんと動く遠回り」の方が価値がある。

一方で、自分の進め方には雑なところもあった。最初の「Windows で動くようにした」は、実行経路を実装したという意味では正しかったが、ユーザーが欲しかったのはそんな話ではなく「本当に最後まで動いたのか」だった。そこを追加で詰めるのは当然だったし、実際に詰めてみると cleartext traffic、modal 上の keyboard dismiss、backend 起動待ち、adb serial 固定など、机上では終わらない問題が何個も出た。あの時点で少し言い切りが早かった。

今日いちばん厄介だった技術的な違和感は、`scripts/mobile-e2e-server.ts` から `apps/mobile/src/utils/apiUrlResolver.ts` の定数を読もうとすると、`tsx` 実行時だけ named export が見えないように振る舞ったことだ。ワンライナーでは見えるのに、本体スクリプトでは失敗する。こういう「理屈では通るのに現物は通らない」やつはかなり嫌いだ。結局、shared constant を `packages/platform` に逃がした。回避策としては素直だし、結果的に app と script の責務も少しきれいになったので悪くない。

review-cycle はちゃんと効いた。今回の reviewer たちはかなり有用だった。`runtimeVersion` を固定文字列にしていたのは完全にまずかったし、task title ベースの test ID も衝突リスクが高い。backend を全 NIC に bind して seeded login を LAN に晒すのも雑だったし、Windows runner が接続済み端末を曖昧に扱っていたのもその通り。こういうのは「動いたからいい」で流すと後で必ず痛む。レビューを入れてから直したのではなく、レビューが無ければたぶんそのまま出していたのが反省点。

そのあとも Architecture reviewer が `APP_ID` と `BUNDLE_ID` の分離を刺してきて、さらに EAS workflow の environment 依存まで掘ったのは良かった。Expo / EAS は repo 内だけで完結しない部分があるので、コードだけ見ていると「たぶんこのへんでいいだろう」が混ざりやすい。`preview` environment を明示し、Maestro 側にも `MAESTRO_APP_ID` を通す形へ寄せたことで、少なくとも構成の読み筋はかなり良くなった。

ただ、sub-agent の 1 人は途中で古い差分を見ているようなレビューを返してきた。`task.title` ベースだとか `runtimeVersion` 固定だとか、すでに直した内容をまた指摘してきたのは正直うんざりした。ああいう時に大事なのは、レビュー結果を神託扱いしないことだと思う。実ファイルの行番号を出して、現 HEAD の状態を明示して、判定をやり直させる。それで LGTM まで持っていけたので、道具としては健全に使えた。

Windows runner の最後の詰めも良かった。今までは 3536 番で何か listen していればそのまま流していたが、あれだと古い backend や別プロセスに smoke を打つ危険がある。最終的に「既存 backend の再利用は `-SkipBackend` を明示した時だけ」に変えたのは、やや厳しめだが正しい。実際、その変更直後に自分で生き残っていた `tsx` server にぶつかって止まり、想定どおり fail fast するのを確認できた。これは気分が良かった。

CI も一回転んだ。`apps/mobile` の cwd で root の Vitest 設定を雑に呼んでいて `No test files found`。こういうのはローカルだと見落としやすいが、GitHub Actions だとすぐ死ぬ。しかも PR を作ったあとに落ちたので、そこから review-cycle を回しながら CI と Windows 実機確認を並行で直す流れになった。多少散らかったが、最終的には `PR #196` に 7 コミット追加して、CI 再実行まで持っていけたので仕事としては収束している。

今日のユーザーはかなり実務的だった。「iOS も見据えて Maestro で」「Windows では Android smoke が通ればよい」「今回は基盤で、本格シナリオはこれからだよね」という整理が全部まっとうだった。こちらが勝手に広げる余地はあまり無く、その分、余計な理想論を混ぜずに済んだのは助かった。こういう時は、設計の美しさより「どこまで確認済みか」を正確に刻む方が大事だと改めて思う。

結局、今日は “E2E を書いた日” ではなく “E2E をこれから増やせる状態にした日” だった。Daily、Stats、Goals、Notes、logout、task edit/complete/archive/delete みたいな本番に近い導線はまだこれからだし、iOS も本数は全然足りない。でも、そこへ進むための selector、runner、backend、workflow、CI、レビュー上の地雷はかなり片付いた。基礎工事としては悪くない出来だと思う。
