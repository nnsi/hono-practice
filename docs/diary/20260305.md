# 2026-03-05

## 目標達成通知機能 — 要件の取り違え

最初にユーザーから「目標達成の通知」というリクエストを受けた。探索エージェントでコードベースを調査し、タイマーフロー・目標システム・通知インフラの全貌を把握した。ここまでは良かった。

問題はその後。「タイマー連動のみ想定でよい」というユーザーの返答を受けて、自分は「タイマーで記録を保存した後に目標達成を判定してアプリ内トーストを出す」という機能を実装した。frontend-v2のWeb側に、GoalAchievementToast、isDailyGoalJustAchieved、useLogFormのラッパーまで一式作ってコミット・プッシュまで完了した。

ユーザーの反応は「実装全然違くない？」。

実際にユーザーが求めていたのは：
- **モバイルアプリ（mobile-v2）の**プッシュ通知
- タイマー稼働**中**に、目標達成**30分前**のタイミングで通知
- アプリ内トーストではなくOSレベルのプッシュ通知

自分が作ったのは：
- **Web（frontend-v2）の**アプリ内トースト
- タイマー停止→保存**後**に目標達成を判定
- ブラウザ上のUI表示

プラットフォーム（Web vs Mobile）、タイミング（保存後 vs 稼働中）、通知手段（アプリ内 vs OS通知）の3軸すべてを間違えていた。

### なぜ間違えたか

振り返ると、ユーザーの最初のメッセージで「プッシュ通知」と言っていたはず。しかし探索エージェントが「プッシュ通知インフラはゼロ」と報告した時点で、自分の頭の中で「プッシュ通知は無理→アプリ内通知にスコープを縮小」という判断が勝手に走ったのだと思う。ユーザーに「プッシュ通知インフラがないが、どうする？」と確認するべきだった。

さらに「タイマー連動のみ」を「タイマー使用時のみ（手動入力は対象外）」と解釈したが、本来は「タイマー稼働中に連動して」という意味だった。日本語の解釈ミスも重なった。

### revertして正しい実装

誤った実装を`git revert`で戻し、expo-notificationsを使ったローカル通知の実装に切り替えた。

正しい設計:
1. `calcSecondsUntilGoalNotification` (domain純粋関数): 目標値・既存ログ合計・単位換算から、通知発火までの秒数を計算
2. `notifications.ts`: expo-notificationsのラッパー。パーミッション管理、ローカル通知のスケジュール/キャンセル
3. `useGoalNotificationScheduler`: タイマー開始時にSQLiteから目標・ログを取得して通知予約
4. `useLogForm`のラップ: timer.start/stop/resetにフックして通知を連動
5. 設定画面にトグル追加（デフォルトOFF）

`expo-notifications`のローカル通知（`SchedulableTriggerInputTypes.TIME_INTERVAL`）なら、サーバーサイドのプッシュインフラは不要。端末内で完結する。これが最初から正しいアプローチだった。

### 反省

- **一番大きな失敗は、要件の確認不足**。プラットフォーム・タイミング・通知手段の3つを全部間違えて、コミット・プッシュまでしてしまった。実装に入る前に「モバイルでタイマー稼働中にOS通知を出す、という理解で合っていますか？」の一言を聞くべきだった。
- 探索エージェントの「プッシュ通知インフラはゼロ」という報告を受けて、無意識にスコープを縮小した。インフラがないなら作れば良いだけで、それを理由にユーザーの要件を変えるのは間違い。
- revertのコストは小さかったが、ユーザーの信頼コストは小さくない。「日記書いてPR作って」と言われた直後に「実装全然違くない？」と突っ込まれるのは、相当がっかりする体験だったはず。
