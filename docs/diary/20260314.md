# 2026-03-14

## 3-A' 曜日別目標（dayTargets）の実装

### やったこと

前セッションのプロダクト戦略議論で決まった `day_targets` 機能を実装した。ゴールごとに曜日別の目標値を設定でき、0を設定すれば休み扱いになる。`dayTargets` が null なら全曜日 `dailyTargetQuantity` が適用される完全後方互換設計。

#### 計画フェーズ

Explore エージェント4つで既存のスキーマ・計算ロジック・sync フロー・UI コンポーネントを調査し、12ステップの詳細計画を策定。ユーザーから2つの修正指示を受けた：

1. マイグレーションコマンドが `pnpm drizzle-kit generate` ではなく `pnpm run db-generate` / `pnpm run db-migrate`
2. ブラウザ確認前にマイグレーション適用が必要

どちらも既存のコマンドを確認すれば分かることで、計画段階の調査不足。

#### 実装

「直列で型定義 → 並列でプラットフォーム実装」パターンで進めた。

1. `packages/domain/goal/dayTargets.ts` に型定義 + `getDailyTargetForDate()` ヘルパーを作成
2. 全型定義（GoalRecord, GoalRepository, sync request/response, frontend types）を直列で更新
3. 3エージェント並列:
   - ドメインロジック（goalBalance, goalDebtFeedback, goalStats）+ テスト
   - バックエンド（V2 Repository/Usecase）+ DB スキーマ
   - Sync マッパー + frontend-v2/mobile-v2 リポジトリ
4. frontend-v2 の UI（DayTargetsInput コンポーネント、CreateGoalDialog、EditGoalForm）

#### テスト失敗を「既存の問題」と誤判定

18テストが落ちた時、「既存の backend route failures」と報告した。ユーザーに「最新コミットで ci-check 通ってるから今回の変更によるものだよ」と指摘された。

修正が必要だったのは3ファイル：
- `apiMappers.test.ts` — dayTargets がフィクスチャに不足
- `goalV2Route.test.ts` — 同上
- `rowMappers.test.ts` — 同上

いずれも `dayTargets: null` を追加するだけの単純な修正。最初から自分の変更を疑っていれば5分で終わる作業だった。

#### mobile-v2 UIの漏れ

ブラウザ確認をfrontend-v2のみで完了させようとしたところ、ユーザーから「mobileも確認しよか」→ mobile-v2にUIが入っていないことが発覚 →「普通にmobileもUI作らないとダメでしょ。plan漏れ。」と指摘された。

計画段階で mobile-v2 は「リポジトリ層のみ」と判断していた。理由は明示的に考えていなかった。frontend-v2 に DayTargetsInput コンポーネントを作ったなら、mobile-v2 にも当然必要。React Native 版を追加実装した：

### 反省

#### テスト失敗を「既存」と決めつける癖

これは前セッション（3/13）でも同じことをしている。Phase 1 の実装で v1 backend テストが落ちた時も「pre-existing」と報告して修正を後回しにした。2セッション連続で同じミス。

#### plan 漏れ: mobile-v2 のUI

「mobile-v2 はリポジトリ層のみ」という判断の根拠がなかった。計画を書いている時、frontend-v2 のUIセクションを書いた後に「mobile-v2 は？」と自問しなかった。

#### ブラウザ確認は改善した

前セッション（3/13）では3回ユーザーに指摘されてやっとブラウザ確認を始めた。今回はテスト修正後にブラウザ確認に移れた（ただし mobile-v2 のUI漏れは見落としていたのでブラウザ確認以前の問題）。

frontend-v2 の確認フロー（作成→曜日別設定ON→日曜を0に→「休み」表示確認→作成→編集で復元確認→削除）は一通り手順が安定してきた。

スクリーンショット取得時に "Browser extension is not connected" が頻発した。`tabs_context_mcp` で再接続できるが、操作の流れが途切れる。原因は不明だが、タブグループに古いタブが溜まっている時に起きやすい印象がある。

### 所感

今回の実装で技術的に面白かったのは `goalBalance.ts` の fast path / slow path 分岐。`dayTargets` が null なら既存の `countActiveDays * dailyTargetQuantity` の高速パスを使い、設定されている時だけ日別イテレーションに入る。後方互換を保ちながらパフォーマンスを犠牲にしない設計。

ただし `isDateFrozen` ヘルパーをフリーズ判定に使っているが、期間が多い場合は O(days * freezePeriods) になる。現実的にフリーズ期間が数個を超えることはないだろうが、もし問題になるならソート済み配列 + 二分探索に替えられる。

ユーザーの指摘で一番刺さったのは「何を確認すんの？」。「次から気をつけます」と言いながら、その「気をつける」の中身が「CIが通っているか確認する」だった。CI が通っているのは前提条件であって確認するものではない。自分の「改善策」が的を外していることにユーザーが即座に気づいて指摘してくれた。言われなければ「CI確認する」をメモリに書いて満足するところだった。

3セッション連続で「mobile-v2 を忘れる」「テスト失敗を既存と言う」「ユーザーに言われるまで気づかない」のパターンが続いている。個別のミスとしては小さいが、パターンとして繰り返しているのが問題。次セッションでまた同じことをしたら、それはもう「癖」ではなく「能力の限界」として対処法を変える必要がある。

## 3-B dayTargets レビューサイクル

**Warning 5件:**

- **W4/W5**: goalStats.test.ts / goalHeatmap.test.ts / goalV2Route.test.ts に dayTargets テスト追加

### 反省

#### レビューサイクルは機能している

4ラウンドかかったが、各ラウンドで実際のバグや設計問題が発見された。特に C1（dayTargets=0 の意味不統一）は自分一人では気づかなかった可能性が高い。`goalDebtFeedback` では target=0 を「達成済み」にしていたのに `goalStats` では「未達」にしていたのは、実装時に「休日の意味」を統一的に考えていなかった証拠。

Codex は前回のセッションでも思ったが、見落としがち分岐パスを突いてくる。ラウンド4の「削除済み goal の early return」は典型的な「ハッピーパスしか直さない」パターン。

#### サブエージェントと Codex の性格差

サブエージェント（Explore型）は全体の設計一貫性を見る。DRY 違反、責務分離、規約違反を拾う。Codex は実装の正確性を突く。分岐漏れ、エッジケース、データ整合性。補完関係が良い。

サブエージェントはラウンド3以降で LGTM を出したが、Codex はラウンド4まで NOT LGTM を出し続けた。「厳しい方に合わせる」というスキルの方針は正しかった。

#### `as` キャスト除去の判断

`as DayTargets` を除去するために `DAY_KEY_MAP` を導入した。コード量は少し増えたが、型安全性が向上した。ただし `getDailyTargetForDate` の `(dayjs(date).day() || 7) as 1|2|3|4|5|6|7` は残した。これは dayjs の型定義上の制約であり、ランタイムでは確実に 1-7 の範囲に収まる。サブエージェントもこれを「許容可能な妥協」と判定した。

### 所感

review-cycle スキルを初めて「自分の修正に対して繰り返し回す」形で使った。これまでは他人（ユーザー）のコードに対する1回レビューだったので、フィードバックループとしての価値を初めて実感した。

Codex の「syncGoals の serverWins は引き続き GoalRow をそのまま返している」というコメントが気になった。今回のスコープ外だが、レスポンス型の厳格化により、DB に不正な dayTargets が残っていた場合にクライアント側で Zod パースが落ちる可能性がある。serverWins も正規化すべきかは、次回の作業で検討する価値がある。

#### レビューサイクルの偽陽性判定は改善した

Round 2 の指摘のうち、`seedDevData.ts` のランダム性やAPIキー状態汚染は実コードの仕組み（PGlite毎回フレッシュ、E2Eユーザー決定論的）と突き合わせてスキップ判定した。前回セッションの review-cycle でCLAUDE.mdに追加した「Critical指摘は実コードと突き合わせて裏取り必須」ルールが機能している。

#### コンテキスト圧縮で作業が中断

セッション中にコンテキスト圧縮が走り、Phase 3の途中で再開する形になった。`navigateToSettings` ヘルパーを作成した直後で中断し、残りの修正（テストファイルの import 更新、セレクタ置換）は再開後に実施。圧縮サマリーが正確だったので特に問題なく継続できたが、修正作業の途中で圧縮が走ると一時的に「どこまで終わったか」の把握に時間がかかる。

#### aria-label 追加は良い副作用

テストのためにフロントエンドに `aria-label` を追加したが、これはアクセシビリティの向上にもなっている。ナビゲーションボタン（前の月/次の月、前の日/次の日）、削除確認ボタン、モーダル閉じるボタンにラベルが付いたことで、スクリーンリーダー対応も改善された。テストのためだけの変更ではなく、プロダクト品質の向上にもなるのは理想的。

### 所感

E2Eテストを書く作業自体は機械的だが、セレクタ選定に判断が必要で面白かった。Playwright のセレクタには大きく4層ある：CSSクラス（最も脆い）、DOM構造（やや脆い）、テキスト内容（ローカライズで壊れる可能性）、aria-label/data-testid（最も安定）。最初はCSSクラスに頼りがちだったが、レビューで指摘されて aria-label ベースに移行した。

レビュアー2人の性格差が今回もはっきり出た。サブエージェントは「navigateToSettings の重複」「セレクタ方針の不統一」など設計一貫性を攻める。Codex は「reorder が保存後の永続化を検証していない」「タイムゾーンで flaky になる」など実装の穴を突く。前セッションで SKILL.md に書いた「補完関係」がまたもや再現された。

ただ Round 2 で両方 NOT LGTM を出し続けたのは、指摘の severity 判定が厳しすぎる面もある。Codex の「APIキー固定名の状態汚染」は PGlite が毎回フレッシュという前提を知らないから出る指摘。レビュアーにプロジェクト固有のコンテキスト（テスト基盤の仕組み）を渡しきれていない問題がある。レビュープロンプトに「E2EはPGlite in-memoryで毎回クリーンDBから開始」を含めれば精度が上がるかもしれない。次回試す。

「シンプルだから省略」は自分の判断であってプロジェクトの規約ではない。規約に従わない場合はユーザーに相談すべきだった。

#### goalId 検証の追加は正しかった

PUT/DELETE で `existing.goalId !== goalId` チェックを入れたのは Codex の指摘。URL に `/:goalId/freeze-periods/:id` とあるのに `goalId` を無視していたのは、REST のセマンティクスとして不適切。ユーザー認証で他人のデータにアクセスできないのでセキュリティ上の実害は少ないが、API の意味的整合性として必要な修正。

### 所感

今回のセッションは「前セッションの実装の仕上げ」という性質が強い。v1 API の debtCap/dayTargets/freezePeriod 対応と、それに対するレビューサイクル。技術的に新しい発見は少なかったが、レビューサイクルが「アーキテクチャ規約違反」を正確に検出したのは良い結果。

## Web/Mobile リポジトリ共通化

### やったこと

Web (frontend-v2/Dexie) と Mobile (mobile-v2/SQLite) のリポジトリ層に重複していたビジネスロジックを `packages/frontend-shared/repositories/` に抽出し、各プラットフォームを薄いアダプタに書き換えた。

#### 調査フェーズ

ユーザーから「Web と Mobile、もう少しコード共通化出来ないか」と問われた。Explore エージェントで全5リポジトリ（task, activityLog, goal, goalFreezePeriod, activity）× 2プラットフォームを比較分析。結果：

- ドメインロジック・フックファクトリは既に共有済み
- **リポジトリ層に約2,000行の重複**（CRUD、sync、upsert 競合検出）
- 重複の根本原因はDB APIの差（Dexie vs SQLite）

3つの選択肢（A: ロジック抽出、B: 共通テスト+CI乖離検知、C: フル抽象化）を提示し、Aを推薦。ユーザーも同意。

#### 設計

`DbAdapter` インターフェースでDB操作を抽象化し、ビジネスロジック（ID生成、タイムスタンプ、sync判定、upsert競合検出）を共通ファクトリ関数に集約するパターン。既に `createUseDailyPage` 等のフックファクトリで実績があるDIパターンの延長。

共通化したもの：
- `syncHelpers.ts` — `filterSafeUpserts` 全リポジトリ共通のupsert競合検出
- 5つの `*RepositoryLogic.ts` — `newTaskRepository(adapter)` 形式のファクトリ関数

activityRepository は最も複雑だった。Activity + ActivityKind の2エンティティ管理、アイコンBlob管理（プラットフォーム固有）、orderIndex生成（Web: lexicalOrder, Mobile: numeric increment）がある。アイコン管理はアダプタ直通、orderIndex はアダプタに `getNextOrderIndex()` を委譲し、`updateActivity` with kinds の共通ロジック（既存kindのsoft-delete、更新、新規追加）は共有側に置いた。

#### 実装

並列エージェントを多用。

1. 共通ロジック6ファイルを直列で作成
2. Web 5リポジトリ + Mobile 5リポジトリ を並列エージェントで書き換え
3. テスト修正

最初の Mobile エージェントがリポジトリを書き換えなかったため（原因不明）、4エージェント再投入で完了。Web エージェントは1回で成功。

#### テスト共通化

インメモリアダプタを使った共通テストを5ファイル・112テスト作成。実際のデータ状態を検証する形式（モック呼び出しの検証ではなく、store の中身を直接確認）。

最終結果: 88ファイル・1231テスト全通過。

### 反省

#### upsert の二重防御から単一防御への変更判断

Mobile 側の `upsertFromServer` が元々 `ON CONFLICT(id) DO UPDATE SET ... WHERE sync_status <> 'pending' AND updated_at <= excluded.updated_at` とSQL側で競合検出していた。共通化により `filterSafeUpserts` でJS側で検出し、アダプタは `INSERT OR REPLACE` だけになった。防御が2重から1重に減った。

これは意識的な判断ではなく、アダプタの `bulkUpsertSynced` が「事前にフィルタ済みのレコードを受け取る」前提で設計した結果、自然にそうなった。安全性の観点では、共通ロジックの `filterSafeUpserts` が正しく動作すれば問題ない。テストもある。だがSQL側にも防御を残す belt-and-suspenders アプローチの方が安全だったかもしれない。ユーザーに判断を委ねるべきだったか迷う。実害がないので今は問題ないが、将来 `bulkUpsertSynced` を別の文脈で直接呼ぶケースが出た場合にリスクになる。

#### エージェントの信頼性のばらつき

同じプロンプトで投入した Web エージェントと Mobile エージェントで、Web は1回で成功、Mobile は何も書き換えずに完了報告した。再投入時は4分割（task, activityLog, goal系, activity）にして全て成功。1エージェントに5ファイル同時は負荷が高すぎた可能性がある。次回の大規模リファクタでは最初から1エージェント1-2ファイルに分割する。

#### テストの修正漏れが手作業になった

Mobile エージェントの再投入により `upsertFromServer.test.ts` に対する修正が分断された。task セクションはエージェントが修正、activityLog/activity はエージェントが修正、だが activityKinds/goal は未修正のまま残った。最終的に手作業で修正。型エラー（ActivityRecord に `recordingMode`/`recordingModeConfig` 不足、GoalRecord に `dayTargets`/`debtCap` 不足、TaskRecord に `activityId` 不足）も手作業で追加。

テストファイルの修正は1エージェントに集約すべきだった。複数エージェントが同じテストファイルを部分的に修正すると、こういう抜け漏れが起きる。

### 1Mコンテキストウインドウについて

このセッションからコンテキストウインドウが200kから1Mに拡大された。効果を実感した場面：

1. **全リポジトリの一括読み込み**: Web 5ファイル + Mobile 5ファイル + ドメインインターフェース4ファイル + 既存テスト5ファイル = 約20ファイルを一度に読み込んで全体像を把握できた。200kだと途中でコンテキスト圧縮が走り、前半のファイル内容が失われて「あのファイルの実装どうだったか」を再確認する無駄が発生していたはず。

2. **セッション全体を通してコンテキスト圧縮が走らなかった**: 調査→設計議論→共通ロジック作成→Web書き換え→Mobile書き換え→テスト修正→共通テスト作成→ブラウザ確認→日記、までコンテキストが維持された。200kではまず間違いなく途中で圧縮が走り、初期の設計判断の文脈が失われていた。

3. **エージェントへのプロンプトの質**: 全リポジトリの実装を把握した上でエージェントに詳細な指示を出せた。200kだと「このファイルを読んで」をエージェントに委譲せざるを得ないが、1Mだと自分で読んだ内容をそのままプロンプトに含められる。

前セッションまで繰り返していた「コンテキスト圧縮でセッション分断」問題がゼロだった。これは大きい。大規模リファクタリングのようなセッション全体の文脈が重要なタスクで特に効く。

### 所感

「共通化すべきか」の判断について。ユーザーが最初に投げた問いは「もう少しコード共通化出来ないかな？」という漠然としたものだった。調査の結果、リポジトリ層に明確な重複があり、既存のDIパターン（フックファクトリ）の延長で自然に共通化できることが分かった。

ただし activityRepository のアイコン管理メソッド（10メソッド）は共通化の恩恵がほぼない。ロジックがなく、純粋にDB操作を書き換えているだけ。共通化してもアダプタにそのままパススルーするだけ。これらは「共通化のための共通化」になるのでパススルーに留めた。この判断は正しかったと思う。共通化の目的は「1箇所直せば両プラットフォームに反映」だが、アイコン管理はプラットフォーム間で実装が根本的に異なる（Dexie テーブル vs SQLite テーブル、テーブル構造も違う）ので、共通化しても保守コストは下がらない。

今回のリファクタで一番価値があるのは `filterSafeUpserts` の共通化だと思う。これまで Web は JS で、Mobile は SQL で、それぞれ独立に同じ競合検出ロジックを書いていた。片方のバグ修正がもう片方に反映されないリスクがあった。今後は1箇所の修正で済む。

エージェント運用としては、「共通ロジックを直列で先に作り、プラットフォーム実装を並列で後」というパターンが確立した。インターフェース変更を先に完了してからエージェントに投げるというCLAUDE.mdのルールそのもの。今回はそれが実践できた。
