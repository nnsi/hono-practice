2. **スキル分割**: `/multi-review`（レビュー+集約） と `/review-cycle`（修正ループ） に分離

3. **`/codex-multi-review` を新規作成**: codexプラグインの `codex-companion.mjs task` を使い、Codex 4体に専門レビュアーロールを割り当てる軽量版。トークン消費を大幅に削減

4. **スコアリングルール設計**: 75+報告、80+修正、2人以上75+も修正対象に格上げ

5. ハードコードされた `D:\workspace\hono-practice` を `$PROJECT_ROOT` に統一

### 設計判断の背景

前セッションで `/review-cycle` を実戦投入した結果、2つの問題が見えた:
- **レビュアーにプロダクトコンテキストが乗っていない**: 汎用的な観点でしかレビューできず、Actiko固有の規約違反（オフラインファースト原則、層構造等）を見逃していた
- **トークン消費が激しい**: サブエージェント4体がそれぞれコンテキストを消費し、再レビューラウンドでさらに膨らむ

前者は各レビュアーにプロダクトコンテキストを埋め込むことで対処。後者はCodexプラグインを使った外部実行に切り替えることで対処。両方を一度にやれた。

### 所感

ユーザーの「レビュアーにコンテキストが乗ってない」という問題提起は的確だった。自分もRound 1のレビュー結果を見て「指摘が表面的だな」と思っていたが、原因を「レビュアーの能力限界」と捉えていた。実際はプロンプトにプロダクト固有情報を入れていなかっただけで、仕組みの問題だった。

スキル分割の提案は良い判断だった。148行の1スキルはLLMが後半の指示を遵守しなくなるリスクがある。ユーザーが「途中で忘れられるか勝手にレビュアーを削られる」と懸念したのは、前セッションで実際にそれが起きかけたからだろう。
