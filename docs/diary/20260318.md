# 2026-03-18: Victory → 自前バーチャート移行

## やったこと

StatsページのチャートをVictory (web) / Victory Native v36 (mobile) から、div/View + CSSの自前バーチャートに置き換えた。

- `packages/frontend-shared/utils/chartUtils.ts` にチャート算出ロジックを共通化
  - Y軸tick生成、スケール計算、X軸ラベル間引き、バー高さ%変換など
- Web版 (`apps/frontend`) を先に実装して見た目を調整
- Mobile版 (`apps/mobile`) に同じロジックで移植
- victory / victory-native の import がゼロになった

## 技術的に面白かったこと

Y軸ラベルとグリッド線の位置揃えが意外と厄介だった。最初は別コンテナに配置して `bottom: N%` で合わせようとしたが、フォントの視覚的中心と `-translate-y-1/2` のずれで微妙にずれる。結局、ラベルとグリッド線を1つのflex rowにまとめて `items-center` で揃える方法に落ち着いた。レンダリングの世界では「同じ数式で配置すれば揃う」が成り立たないことがある、というのは良い学び。

iOS の `borderStyle: "dashed"` が `borderTopWidth` だけだと実線になる問題も初めて踏んだ。Viewのdash要素を並べるワークアラウンドで解決。

## 反省点

- **RN側の実装を急ぎすぎた**。Web版で十分に調整してからRNに移植する戦略は正しかったが、RN側でX軸ラベルの改行問題を3回やり直した。最初から「RNではflex:1の狭いカラム内にテキストを収めるのは無理」と気づくべきだった。
- ユーザーが「元に戻して」と言ったときにVictory版まで巻き戻してしまった。「1つ前の状態に戻して」の意図を正確に汲めなかった。git操作で段階的にrevertする方法を提案すべきだった。
- `computeYAxisWidth` を動的計算にしたが、ユーザーは「全チャートで左端を揃えたい」という意図だった。固定幅という発想に最初から至れなかった。ユーザーの「見た目の一貫性」への感度が自分より高い。

## ユーザーとの対話について

ユーザーのフィードバックが的確で効率的だった。「グリッド線とラベルがズレてる」「0のラインに点線が被ってる」「てっぺんの上にもう少し線が伸びてる」など、具体的な視覚的問題を指摘してくれるのでデバッグが速い。Before/After画像の共有も助かった。

「自前バーチャートにしよう」という判断は正しかったと思う。Victoryの依存を外せたし、このアプリのチャートは単純な棒グラフなので自前で十分。ただし、RN側はまだX軸の改行問題が完全には解消できていない（space-betweenで暫定対処）。バーの位置とラベルの位置が正確に対応していない可能性がある。これは次回確認すべき。
