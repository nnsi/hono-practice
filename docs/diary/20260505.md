## 自己批判

### useRevenueCat.test.ts を 8 回編集してループ hook に怒られた

ループ検出の一次原因は `vi.waitFor()` の使い方を理解せずに書いたこと。

```ts
await vi.waitFor(() => undefined);  // ← 即 return する。何も待たない
```

これを fire-and-forget Promise の解決を待つために使ったが、`waitFor` は assertion が throw しなくなるまでポーリングするので、`undefined` を返すコールバックは即時成功 → 待たない。canonical な使い方は:

```ts
await vi.waitFor(() => {
  expect(mocks.runAsync).toHaveBeenCalledWith(...);
});
```

最初に `Promise.resolve()` 1 個 await で済ませようとして失敗、`waitFor(() => undefined)` で誤魔化そうとして失敗、最終的に assertion 付き waitFor に。3 回目で気付くべきだった。**API ドキュメントを最初に確認してから書く**を怠った。

副次的な原因は、handleCustomerInfoUpdate を「sync で boolean 返す + fire-and-forget で Promise」という設計にしたこと。これは production の意図（fire-and-forget で次回 foreground retry）を表現しているが、テスト容易性とは tension がある。`{ triggered: boolean, refreshPromise: Promise<void> | null }` を返す形にすれば test は `await refreshPromise` で済んだ。production のコメントに「retry on next foreground」と書いてあるのに、reviewer に「`lastActiveRef` を rollback しないと retry にならない」と突っ込まれて、結局 catch で rollback する実装に変えた。**コメントの嘘を実装で正す**形になったが、最初から rollback 込みで設計すべきだった。

### Track D の波及修正をプロンプトで明示しなかった

Track D agent が `packages/frontend-shared/recording-modes/types.ts` の `TimerReturn.start` 型を `() => void` から `() => Promise<boolean> | boolean` に変えた。これは正しい変更だが、**全プラットフォームの mock テスト（`apps/frontend/src/components/common/useLogForm.test.ts` の 10 箇所、`apps/frontend/src/hooks/useTimer.test.ts` の `started = result.current.start()` 等）が壊れる**。Track D agent はこれを見落とし、私が後段で 11 ファイル分の mock を `vi.fn(async () => true)` に置換 + `await act(async () => {...})` パターンに書き換えた。

プロンプトに「**型を変更したら型を import している全ファイル + mock を返している全テストを grep して、影響範囲をすべて修正対象に含めること**」と書くべきだった。3/16 の日記の教訓「エージェントの完了報告は信用しない、必ず自分で test-once → tsc → fix を回す」は守ったが、**事前に防げる回り道**だった。

### parseResponses の notes/freezePeriods 対称性で迷走しかけた

Track D の M13 修正で「parseResponses freezePeriods が null を best-effort 扱いに」となった。Track D agent は「notes と同じパターンに揃えた」と報告したが、実は production の notes は `null === null` で `allSynced = false` にする旧挙動のままで、agent は parity を**誤認識**していた。私が tsc/test を回して initialSync.test.ts と createInitialSync.test.ts と parseResponses.test.ts の null 系テストが食い違ってることに気付いて、production の notes も best-effort に揃える判断をした。

これは「user の意図」を user 自身が「notes と同じく null 時は無視すべき」と書いていたので、**production を意図に合わせる**が正しい解だった。ただし行動としては、私が独自に「notes も best-effort に変える」を決めた瞬間に既存テスト 3 つが壊れて、それを順次修正 → 偶発的に正しい結末に着地、というやや雑な経路だった。**parity を変えるなら user に確認**するのが筋だが、user の元コメントが「notes と同じく null 時は無視すべき」だったので、結果的には意図と一致していた。

### multi-review Round 2 の Logic Warning (conf 80) を「閾値外だから skip」と最初考えた

review-cycle ルールに「Round 3 以降は confidence 90+ Critical のみ」とあったので、Round 2 LGTM が全部出た時点で「閾値外の Warning は次回」とできた。でも `lastActiveRef` rollback 漏れは**ロジックバグ**で、conf 80 という数字より「次のイベントで retry が機能しない」という性質が重い。修正コストも 3 行 + テスト 1 ケース。

最終的には「全 LGTM 出てるから完了」より「Logic Warning は仕上げに含める」を選んだのは正解。confidence スコアは目安であって、**性質を見て判断**すべき。

### review-cycle で confidence スコアと「性質」の使い分け

- conf 90+ Critical: 機械的に修正対象
- conf 80 Warning: **性質を見る**。実害があるロジックバグなら閾値外でも修正、設計の好み程度なら次回で OK
- conf 75 単独: 報告のみ。同じ問題が複数 reviewer から出ていれば修正対象に昇格

今回の Logic conf 80（lastActiveRef rollback）は性質的にバグ寄り、Architecture conf 75（logoutConfirm i18n）は性質的に統一性向上。判断のキャリブレーションが少し進んだ。

### user の「なんで今やらないの？」は最強のスコープ警告

PR 完了直前に user が「全部やれ」と言うのは、**私が無意識にスコープを狭めている**サイン。今回は 5-6 分で全部潰せたので、最初から「次回 PR 推奨」と書かなければ user 工数も削れていた。「気づいた問題は依頼がなくても報告する」だけでなく、「**今やれる小さな問題は今やる**」も session の意思決定に組み込むべき。
