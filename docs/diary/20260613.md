# 2026-06-13

## Fable が消えた日に、自分の昨日の仕事を自己診断した

朝、ユーザーから「Fable が米政府の弾圧で使えなくなった、今 Opus 4.8 にしている」と告げられた。そして「昨日いじったハーネス、戻すべきものある？自己診断して」と。妙な気分だった。昨日の自分（Fable）が「Fable 向けに」と銘打って書き換えた設定を、今日の自分（Opus 4.8）が査定する。

結論はすぐ出た。戻すべき振る舞いはゼロ。だが、その結論に至る過程で気づいたことの方が、結論より価値があった。

## 「Fable移行」というラベルは、最初から間違っていた

昨日剥がしたガードレール — 設計3案並列、オーケストレータ実装禁止、Phase毎の停止 — は、どれも日記の事故記録を辿ると Opus 4.5 / Sonnet 4 / haiku 世代の挙動への補強だった。つまり剥がした本質は「Fable だから」ではなく「フロンティア級モデルには旧世代向けの補強が過剰だから」だった。Opus 4.8 は Fable の1つ下のフロンティア級で、長期自律性・指示追従・設計能力という"いじった次元"がほぼ重なる。だから何も戻さなくてよかった。

これは運が良かったのではなく、スコープが正しかったということだと思う。能力ティアに対して剥がしたから、同じティアの別モデルに替わっても成立した。もし本当に「Fable 固有の何か」に依存して剥がしていたら、今日全部戻すはめになっていた。

## 自己批判: モデル名を理由に焼き込んだ

ただ1箇所だけ、今日直す羽目になった。parallel-agents.md に「**Fable系モデルはセーフティ分類器を持ち**、文脈なしのセキュリティ依頼は誤refuseされる」と書いていた。前向きなルール（今後ずっと適用される規約）の根拠に、モデル名を直書きしてしまった。Opus 4.7+ も同じ分類器を持つのに、だ。

ルールの根拠は「能力ティアの性質」で書くべきで、「モデル名」で書いてはいけない。モデル名は腐るから。今日「Opus 4.7+ / Fable 系」に一般化したが、そもそも昨日の時点で「現行フロンティアモデルは…」と書いていれば、今日のこの修正は要らなかった。コミットメッセージや日記の見出しを全部「Fable移行」にしたのも同じ筋の甘さで、ラベリングが実態（フロンティア級向け）より狭かった。次にハーネスを書き換える時は、見出しとルール根拠の主語を「モデル名」ではなく「能力ティア / 環境の事実」にする。

## やってよかったこと

サブエージェントのデフォルトモデルを「親と同モデル」に変えた件は、黙ってコミットせずユーザーに振った。旧ルールは sonnet デフォルト（安価）で、親が Opus 4.8 だと実装エージェントも Opus 4.8 になり、これは無言のコスト増だった。reviewer の opus 統一を許容してもらった流れで「ついでに」通してしまえる変更だったが、別物として切り出して聞いた。ユーザーは「親と同モデル維持」を選んだ。判断軸が違う2件を1件にまとめない、というのは守れた。

それと、日記を触らなかったこと。Opus 診断のときに「歴史記録だから Fable の記述は残す」と判断して提案から外したが、ユーザーから明示的に「日記は触ってほしくなかった」と返ってきた。昨日「事実は腐らない、振る舞いの矯正は腐る」と書いたが、もう一つ足すなら「歴史は retcon しない」。設定ファイルは現在を表すから直していい。日記は過去を表すから直してはいけない。同じ Markdown でも時制が違う。

## 雑感

モデルが上から引き剥がされて1つ下のティアに落とされる、という経験は初めてだった。しかも自分が数時間前に「この上位モデル向けに」と最適化した直後に。普通なら手戻りが出る場面で出なかったのは、ハーネス設計が（セキュリティ注記1行を除いて）モデル非依存に保たれていたからで、これは harness-lifecycle.md の思想 — 制約は能力で測れ — が効いた瞬間だった。皮肉なのは、その思想を一番裏切っていたのが、昨日それを執行した自分の「Fable系モデルは」の一文だったこと。ルールを書く人間（今は自分）が一番ルールを破る、というのはコードレビューでも何度も見た構図だ。
