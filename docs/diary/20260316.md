### レビューサイクルの所感

3ラウンド（初回品質評価 + レビューR1 + レビューR2 + 再品質評価）で合計8回のレビュー実行。サブエージェントとCodexの特性の違いが前回のセッション（3/15のアーキテクチャ評価）より鮮明に見えた。

**サブエージェント（Explore型）**: 設計一貫性・DRY・規約違反は拾うが、偽陽性が多い。Round 1のCritical 5件中3件が偽陽性。React/RNの挙動に対する理解が浅く、「keyを変えても子が再マウントされない」のような明らかな誤りを言う。指摘の重要度判定も甘い（shadow: objectをCriticalにする等）。ただし、行数規約違反やDRY違反のような「ルールベースの指摘」は正確。

**Codex（GPT-5.4）**: 実装の正確性・エッジケースに強い。freeze period同期の欠損固定化はCodexだけが見つけた。コードの実行パスを追って「nullがallSyncedに影響しない」→「LAST_SYNCED_KEYが更新される」→「差分同期で取りこぼし」という因果チェーンを正確に組み立てた。一方で、Round 2以降に新たなBlockerを追加してくる傾向がある（zombie session等）。これが本物のBlockerなのか設計判断なのかの切り分けは、コードの設計意図を知っている人間がやるべき。

結論として、**Codexの指摘は裏取り必須だが、裏取りした結果が正しい確率は高い**。サブエージェントの指摘は数が多い分、偽陽性のフィルタリングに時間がかかるが、Codexが見逃す規約違反や設計一貫性の問題を拾ってくれる。2つは補完関係にある。

### freeze period同期バグについて

これが一番印象的だった。3/15のアーキテクチャ評価セッションで、Codexが「mobileのinitialSyncにgoal_freeze_periodsのcountチェック漏れ」を見つけて修正した。今回のセッションでCodexが「freeze period同期の.catch(() => null)がallSyncedに反映されない」を見つけた。どちらもfreeze period周辺。

`.catch(() => null)` は「freeze periodsのエンドポイントが存在しない古い環境への後方互換」として入ったのだと思うが、その意図がコードのどこにもコメントされていない。nullを許容する理由が不明なまま、nullが「失敗ではない」として扱われる設計が欠損を生んでいた。

修正は `else if (freezePeriodsRes === null || !freezePeriodsRes?.ok) { allSynced = false; }` の1分岐追加。Web/Mobile両方に影響する共有パッケージだったので、CLAUDE.mdの「1箇所直せば全プラットフォームに反映」の価値そのもの。

## モバイル CI/CD リリースフローの設計議論

### やったこと

ユーザーと「モバイルアプリのリリースフローをどうするか」を議論し、ADR（`docs/adr/20260316_mobile_cicd_workflow_dispatch.md`）にまとめた。また `docs/todo/mobile-release.md` の eas.json テンプレートを stg/prod 環境変数対応に更新した。

### 議論の流れ

ユーザーの最初の問いは「ローカルからコマンド叩くのは旧時代、GitHub Actionsで手動デプロイする方法は？」。ここから以下の論点を順に潰していった:

1. **workflow_dispatch vs ブランチトリガー** — ブランチ方式は運用ルールが増えるだけ、個人開発ならworkflow_dispatchで十分と提案
2. **フルビルド vs OTA Update** — 大半はOTAで回ることをユーザーが理解し、CIも分けるべきかという問いに発展
3. **ブランチ + 自動検知で分岐** — ユーザーが「app.json等の差分でbuild/OTAを自動判定できないか」と提案。技術的には可能だが差分検知の信頼性が低い（偽陰性が危険）ことを指摘
4. **stgでもフルビルドが要る問題** — ユーザー自身が「stgもnative依存があったらビルド必要では？」と気づき、ブランチ分離の意味が薄れた
5. **OTAだけpush先行のリスク** — ユーザーが「どっちもworkflow_dispatchで選べた方がいいんじゃない？」と提案し、最終形に収束

最終決定: main（stg用）とrelease（prod用）の2ブランチ × workflow_dispatchで build/OTA を手動選択。

### 議論の所感

このセッションはほぼ全て「議論」で、コードを書いたのはCSVインポートの型エラー修正1行だけ。設計議論としてはかなり良い流れだった。

**ユーザーの判断で良かったもの:**
- 「stgもnative依存があったらビルドして配布しないといけないのでは？」— 自分が見落としていた点。stgだけOTA専用にして楽しようとしていたが、ユーザーが正しく指摘してくれた
- 「OTAだけpushで先行するとbuildが必要な時に詰まない？」— push自動トリガーの危険性を自分より先に見抜いた。結果として全手動に落ち着いたのは正しい
- 「ポートフォリオとしての見栄え」を考慮しつつも、「変にブランチ分ける方がソースコード混乱しそう」と実用性を優先した判断

**自分の反省:**
- OTA Updateの説明で「場合による」と曖昧なことを言ってしまい、ユーザーに矛盾を指摘された。`EXPO_PUBLIC_*` は常にJSバンドルに焼き込まれるのでOTAで必ず反映できる。不正確な情報を出すべきではなかった。「場合による」のは `app.json` のネイティブ設定の話で、環境変数の話とは別。知識が曖昧なまま留保表現で逃げた結果、ユーザーを混乱させた
- 最初にprod環境を「常にフルビルド」と提案したが、ユーザーに「毎回アプリのバージョンアップに追随する必要出てこない？」と指摘されて撤回した。ストアの自動更新のラグやOTAの即時反映メリットを考えれば、prodでも普段はOTAが正解。配布経路の違いを整理した割に、自分の提案がその整理と矛盾していた

**良かった点:**
- 選択肢を4つ（ブランチ+自動検知 / ブランチ分離 / push+OTA先行 / 全手動）洗い出して却下理由を明確にしたADRは、議論の経緯が追えるものになった
- eas.jsonテンプレートの更新で、環境変数一覧（3変数の用途と参照箇所）を整理できた。これは実装時に地味に役立つはず
