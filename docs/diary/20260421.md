### Critical バグを実装エージェントもレビュー前の自分も見逃した

`updateAuthState` が `tutorialStatus` を消す問題は、実装としては「register時だけ pending を書く → 動作確認で表示できる」という最小要件を満たすので、**初回ブラウザ確認では完全に見逃した**。Phase 3 の動作確認では「完了後にリロードしたら再表示されない」しか見ていなかった。pending のまま closeして再起動、というシナリオは考えていなかった。

**教訓**: DB 更新系の変更をするときは、その更新がいつ・どこから呼ばれるかを先にトレースする。今回なら「`updateAuthState` の呼び出し元は `performInitialSync` だけ」「`performInitialSync` は起動ごとに走る」まで把握していれば、「起動毎に全置換する」実装への違和感は生まれたはず。

## 学び

- **Dexie の `.put()` も SQLite の `INSERT OR REPLACE` も「全置換」**。追加カラムが増えるたびに既存の update 実装が silent に消す危険がある。authState のような「部分更新したいテーブル」は repository に `setXxxAndYyy` のように明示スコープの関数を生やすか、`existing` を必ず読んで merge する運用にしないと壊れる。mobile 側で repository に寄せて SQL 重複排除+`dbEvents.emit` 統一できたのは良かった
- **factory パターンの DI は Web/Mobile の `useStatus` hook 差異を綺麗に吸収できる**。Web は `useLiveQuery`、Mobile は `useLiveQuery` (カスタム、dbEvents ベース) と形は似ているが内部実装が別。shared factory が `useStatus: () => Status` の 1 point で受けるので、呼び出し側は同じコードで書ける
- **Tailwind v3 のデフォルト z-index スケールは `z-0/10/20/30/40/50/auto` のみ**。`z-60` を書いても CSS が生成されず `z-auto` 相当になる。`CreateActivityDialog` が `ModalOverlay` 経由で `z-50` + DOM 後兄弟で上に来ていたのは偶然。Tailwind config を拡張していないプロジェクトでは arbitrary value `z-[60]` を使うか、そもそも ModalOverlay にレイヤ規約を集約する
- **dev server 起動スクリプトの hardcode ポートには注意**: `apps/frontend/package.json` の `"dev": "vite --host --port 2460"` は .env の `VITE_PORT` を無視して 2460 を使う。worktree 環境で main と競合しなかったのは main の Vite を止めていたから。worktree を複数並行起動する場合、このハードコードは破綻する。worktree-setup script が package.json を書き換えるか、`vite --host` だけにして env 参照させるのが将来的には必要
