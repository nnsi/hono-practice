## 法務文書作成の所感

### 「法務スペシャリスト」と「優秀な弁護士」ロールの差が想像以上に大きかった

同じファイルを、同じレビューサイクルスキルで回しても、ロール設定によってアウトプットが劇的に変わった。

**法務スペシャリスト（Round 1-3）**: GDPR Art.13網羅性、CCPA開示要件、JCAA仲裁の有効性、ODR廃止等、**条文レベルでの網羅性**にフォーカスしていた。技術的には正しいが、現実の運営リスクとはややズレた指摘が多かった。

**優秀な弁護士（追加ラウンド）**: いきなり「activity dataにGDPR Art.9特別カテゴリデータが入りうる」「解約文言と実装の矛盾」「user contentライセンスの欠落」といった**実戦での致命傷**を指摘してきた。特にCodex側が「解約可能と書いてあるのに、実際は settings でキャンセルできない矛盾」を指摘した瞬間、これまでのレビューが根本的に見落としていた層があることに気づいた。

これは今後のレビュー運用に重要な教訓。「レビュー観点」だけでなく**ロールの経験値・思考の深さ**をプロンプトで明示的に要求する必要がある。「15年の経験」「実務的な執行リスクとビジネスインパクトをバランスさせる」と書いたことで、出力の質が変わった。

### 規約と実装の乖離を見つけたのはCodexの方

Premium解約ボタンの不在を発見したのは、Codex側のsenior-lawyerロール。私自身は規約の文面だけ見ていて、実装との整合性まで検証していなかった。この気づきがなければ、このまま公開して Dark Pattern として炎上する可能性があった。

「文書の法的品質」と「文書が実装と整合しているか」は別の問題。後者を検証するにはコードを読む必要があり、それを私が最初からやっていなかったのは反省点。

## ユーザーとの対話について

### 意思決定の委ね方が良かった

ユーザーは「方針判断はユーザーの仕事」というCLAUDE.mdのルール通り、Identity開示・仲裁機関・Indemnification射程・Pro-rata refund条件・事業譲渡時データ移転・対応期限分離等の論点を全て自分で判断してくれた。

私が「選択肢+推奨+理由」を出して、ユーザーが判断する、というフローが確立できていた。ただ、方針A/B/Cを3つ出したのは多すぎたかもしれない。ユーザーの負荷を下げるには推奨1つ+代替1つくらいが適切かもしれない。

### 「私はこの規約でリリースして本当に問題ないか？」の質問への回答

これは正直に答えた。「公開できる。ただしAIが書いた文書で、弁護士レビューは通っていない」と率直に伝えた。迎合せず、過度に不安を煽らず、現実的なリスク評価を出せたと思う。

ただし、この回答の直後にユーザーが「優秀な弁護士ロールでレビューしてみよう」と提案してきて、追加の問題が大量に出てきた。これは私の初回レビューが甘かった証拠。最初から senior lawyer ロールでレビューすべきだったかもしれない。

## 反省点

### 1. 規約と実装の整合性を最初から検証しなかった

規約の文言だけ見て「Web版はsettings画面から解約」と書いたが、実装を確認すれば解約ボタンが存在しないことはすぐ分かった。**規約を書く時は、その約束を実装が果たせるかを同時に検証すべき**。

### 2. 一回目のレビューで「十分」と判断してしまった

Round 3でLGTMが出た時点で、私は「これで完成」と判断していた。ユーザーが「優秀な弁護士ロールだとどうなる？」と聞いてきて初めて、追加レビューをしたら大量の問題が出てきた。**レビューは「ロールの観点」を変えて複数回やるべき**。同じロールで何ラウンド回しても、同じ盲点は見つからない。

### 3. GDPR Art.9の論点を自分で思いつけなかった

「活動記録アプリのメモに健康・宗教・性的指向が入り得る」というArt.9論点は、senior-lawyer-codexが指摘して初めて認識した。これはドメイン知識の問題。アクティビティ記録アプリという用途を考えれば、気づいてもおかしくなかった。

### 4. CSV export機能の存在を確認せずに「in-app export feature」と書いた

これはユーザーに指摘されて確認した。幸い既に実装されていたが、**文書で約束を書く時は、その機能の存在を確認してから書く**べきだった。

## ルール追加の提案候補

### 法務文書・契約書系の作成時のルール

- **規約と実装の整合性を必ず検証する**: 規約に書く約束は、実装で果たせることを先に確認する。特に解約・削除・エクスポート等のユーザー権利系
- **複数のロール・視点でレビューする**: 「スペシャリスト」と「シニア弁護士」等、同じドメインでも経験値・視点の違うロールを複数使う

これらを `.claude/rules/legal-docs.md` 等に追加するか、ユーザーに提案してもいいかもしれない。ただし、法務文書を書く機会は稀なので、優先度は低い。

## 今日の教訓

**「レビューでLGTMが出たら完成」ではない。** ロールの視点を変えれば新しい問題が見つかる。特にコードと文書の整合性のような横断的な問題は、専門ロールだけ回していても見つからない。**自分の盲点は、違う経験値のレビュアーが見つける**。

## 午後: Note機能の設計（Markdown管理）

ユーザーが「オフライン同期でMarkdown管理したい」という要件を持ってきた。purpose.mdと整合するか、どう建て付けるか、という議論から始まった。

### 良かった点

**purpose.mdとの整合性を最初に疑った**。ユーザーの「Markdown管理したい」を素直に受けず、「Actikoは最速記録のアプリ。Markdownは本質的に遅い入力。どう位置づけるか？」と問い直した。結果、A/B/C 3案を出して議論した。これは response-style.md の「設計の異論は黙らず言う」に沿った行動。

ユーザーが「ActivityLog.memo（フロー）vs Note（ストック）」という綺麗な分離を提案してきた時、即座にそれが正しいと判断して採用した。これは私の当初案（Task.memoをMarkdown化）よりずっと良い設計。**ユーザーの方が自分のプロダクトの構造を深く理解している**のは当然で、私の役割は叩き台を出して議論を進めることだと再認識した。

### 反省点: 設計ドキュメントを書く前に既存コードを読むべきだった

一番大きな反省点。設計ドキュメント（note-feature.md）を書いた時、私はTaskのデータモデルは Explore エージェントで調べたが、**既存の同期実装（taskSyncUsecase）の中身は読まずに書いた**。

結果、クロスレビューで Critical 6件が出た:
1. activityId ownership検証の欠落（**クロス検出**）
2. APIパス規約違反（`/api/notes` と書いたが既存は`/users/notes`）
3. sync契約（syncedIds/serverWins/skippedIds）の未記載
4. initialSync統合の漏れ
5. クエリ順序（userId先→activityId）の未明記
6. ON DELETE挙動の未指定

**全部、既存のtaskSyncUsecase.tsとtaskSyncRoute.tsを読んでいれば気づけた問題**。データモデル層だけ調べて「同じパターンを踏襲」と書いて済ませたのが甘かった。

特に **activityId ownership検証の欠落** は、クロスレビューで両者（Claude+Codex）が同時指摘したセキュリティ問題。これを含めずに実装に進んでいたら、他ユーザーのactivityに紐付けられる脆弱性を作っていた。

### 教訓

**設計ドキュメントを書く時、「パターン踏襲」と書くなら、踏襲元の実装を実際に読んでから書く。** 既存のテーブル定義や型だけ見て、同期処理本体やroute定義を読まないのは手抜き。コードが答えを知っているのに、それを読まずに抽象的に「既存パターンに準拠」と書くのは未来の実装者（自分か別エージェントか）に丸投げしているだけ。

### スキル選択のミス

Codexを呼ぶ時、最初 `codex-companion.mjs` を直接 Bash で叩こうとした。環境変数 `CLAUDE_PLUGIN_ROOT` が展開されず失敗、Windows パスでリトライして成功したが、ユーザーから「`/codex:review` スキルを使って」と指示された。

**skillsの存在を知っていたのに直接コマンドを組み立てようとした**のは、cross-review スキルのテンプレート通りに機械的に進めようとした結果。ツールが用意されているならそれを使うべきだった。

### クロスレビューの価値の実感

今回、Claude (Opus) と Codex が**別々の観点で重要な問題を同時に検出**した。Claudeはセキュリティ（ownership検証、updatedAt maxAllowed）を強く指摘し、Codexはアーキテクチャ整合性（APIパス規約、sync契約、initialSync）を強く指摘した。どちらか一方だけなら見逃していた問題がある。

`/cross-review` スキルは「普段使い向けの軽量版」と位置づけられているが、**設計ドキュメントレビューにも十分機能する**ことが実証できた。実装前の設計段階でこのレベルの指摘が出るのは非常に価値がある。実装してから修正するより遥かに安い。

これは**ドメイン知識の限界**を示す典型例。民法5条の「未成年者取消権」は知っていたが、「誰が取消権者か」という基本事項を書き間違えた。正直、法律文書を書くのが怖くなる事案。**自分が書いた法律文言は、必ず別モデルに検証させるべき**という教訓が強化された。

subagent（Opus）と Codex は、**別の盲点セットを持っていた**。今回の発見パターン:

### ユーザーとの対話について

#### 「1-1は省略可能だよね？」への対応

ユーザーが**私の指摘に対して反論してきた**。「販売事業者氏名は問い合わせフォームで適宜開示で問題ないはず」と。

私は前回レビュアーの指摘を信じて「省略不可」と伝えていたが、ユーザーの反論を受けて**WebFetchで消費者庁Q&Aを直接確認**した。結果、ユーザーの認識は間違いで、「住所・電話番号は省略可、氏名は必須」が正解だった。

**この確認プロセスは正しかった**。ユーザーの反論を鵜呑みにせず、一次情報で確認する。response-style.md の「ユーザーの言うことを鵜呑みにせず、迎合もしない」を実践できた。

ただし**最初から一次情報にあたっていれば良かった**。レビュアーの言葉を仲介するだけでは、ユーザーへの説得力が弱い。法的主張をする時は最初から消費者庁/個情委ガイドラインを参照してから答えるべきだった。

#### vars vs secrets の件

CI workflow を更新する時、既存の `VITE_CONTACT_EMAIL` が `vars.xxx` なのを見て、そのまま `VITE_ADMINISTRATOR_NAME` も `vars.xxx` にした。

ユーザーから「VITE_ADMINISTRATOR_NAMEはsecretsじゃね？」と指摘。正しい。本名はPIIで、GitHub Actions ログに出すべきではない。特にrepo が public になった場合のリスクを考えると secrets 一択。

**既存パターンの機械的踏襲が事故を生む**典型例。パターンを踏襲する前に「この値は何か（public info か PII か）」を考えるべきだった。午前中の Note 機能設計でも似た失敗をしている（「パターン踏襲」と書いて既存コードを読まなかった）。**既存パターンを引用する時は、なぜそのパターンなのかを考える**という習慣が足りない。

#### noindex 実装について

React 側で `useNoIndex` フックを作って動的 meta tag 挿入にした。これは JS 実行しないクローラーには効かない（弱い防御）ことを明示し、X-Robots-Tag ヘッダでの追加対策を「将来対応」として記述した。

本当は最初から Cloudflare `_headers` ファイルで設定すべきだった。「React だけで対応可能」に飛びついたのは手抜き。ただしユーザーが「とりあえず検索除けしたい」という要件で、かつ Cloudflare Pages の設定変更はデプロイが必要なので、段階的に対応する判断は妥当。

### 今日の教訓（追加）

1. **自分で書いた法律文言は必ず別モデル/人間で検証する**。私は民法の基本概念で条文を書き間違えた。法律ドメインで自信を持つのは危険
2. **レビュアーの言葉を仲介するだけでは不十分**。一次情報（消費者庁/個情委ガイドライン等）を自分で確認してから主張する
3. **既存パターンの機械的踏襲は事故を生む**。引用する前に「なぜこのパターンなのか」を1回考える
4. **クロスレビューで Claude と Codex の盲点分布が異なる傾向は2回目観測**。経験則として定着しつつある

### ルール追加の提案候補

- **既存パターンを踏襲する時は、その判断理由を1回考える**（同じ値でも vars/secrets の区別は文脈依存）
- **法律文言を生成したら、必ず別モデルで条文との整合性を検証する**（特に「主体は誰か」「要件は何か」系）

法律文書は書く機会が稀なので、ルール化の優先度は低めだが、今回の §10 ミスは覚えておきたい。

## 深夜: Web課金の停止判断（VITE_ENABLE_WEB_SUBSCRIPTION導入）

日中にあれだけ規約レビューをやった直後に、ユーザーから「PolarだとApple/Googleみたいな感じで特商法不要だったりする？」という根本的な問い。

### 良かった点: 一次情報で答えた

正直な回答をした。「グレーゾーン。Polar は MoR を謳うが日本の特商法免除を明言していない。Paddle利用者でも念のため表記してる実務が主流」と、迎合せずにリスクを明示した。Apple/GoogleのIAPとの差分（日本法人による販売元記載の有無）にも触れた。

日中のやりとりで「レビュアーの言葉を仲介せず一次情報にあたれ」という教訓を得ていたのが効いた。即答せず、Polar が MoR であっても特商法免除を明文化していないという事実を出発点にした。

### 判断の流れが綺麗だった

ユーザーの意思決定プロセスが今日一番綺麗だった:

1. 「Polar だと特商法不要？」→ 私「グレーゾーン、残すのが安全」
2. ユーザー「逆に Web課金ルートやめれば特商法不要だよね？」
3. 私「不要になる、ただしWebに課金導線を一切持たないこと」
4. ユーザー「feature flag で切り替える形にしよう」
5. 私「影響範囲これで全部？」と確認 → 影響範囲を調査
6. ユーザー「認識合ってる。API は Polar env 未設定なら疎通できないよね」
7. 私「その通り、二重防御成立」
8. ユーザー「『モバイルから』は誘導と取られかねない、誘導不要で」→ 採用
9. 実装

**ユーザーが自分でプロダクトの構造を理解していて、しかも「Apple の外部誘導禁止規約」と「特商法免除の建付け」の両方を踏まえた判断をしていた**。「モバイルからアップグレードしてください」を**誘導と捉えかねない**と自発的に指摘した点が特に良い。私は「モバイル誘導文言を置く」案を案として出していたが、ユーザーの指摘の方が正しかった。

### 反省点: 影響範囲の初回回答が片手落ちだった

最初に影響範囲を聞かれた時、私は**特商法ページ関連のファイルだけ**を列挙した。しかし VITE_ENABLE_WEB_SUBSCRIPTION という変数名の本質は「Web課金を有効にする」なので、SubscriptionSection のアップグレードボタンや EntitlementGate の CTA も gate 対象に含めるべきだった。

ユーザーが「影響範囲これで全部？」と聞いてくれたので、調査しなおして Web 課金UI側も含む正しい一覧を出せた。**ユーザーが念押ししてくれたから救われた**が、本来は最初から「flag の意味 = 何を無効化するのか」を突き詰めて回答すべきだった。

同じ失敗パターン: **変数名/機能名の「本来の意味」を一歩踏み込んで考えない**。午後のNote機能設計でも「パターン踏襲」と書いて既存コード読まなかった件と同根。「言葉の表面で止まる」癖がある。

### 設計判断: EntitlementGate の onUpgrade を optional 化

実装時、「flag=false なら CTA テキストも含めて出さない」という判断をした。具体的には:

- 現状: 「{feature}は Pro プランでご利用いただけます。アップグレードして全機能をお使いください。」+ ボタン
- flag=false: 「{feature}は Pro プランでご利用いただけます。」（ボタンなし、2文目なし）

`onUpgrade` を optional にして、undefined の場合は 2文目もボタンも出さない形にした。Caller 側で `webSubscriptionEnabled ? upgrade.openUpgradeModal : undefined` を渡す。

これは**EntitlementGate が flag を直接読まない設計**にした。Caller が意思決定してから props 経由で渡す。Gate 自体は flag を知らない。これは正しい責務分離だと思う。Gate の中で flag を読むとテストしづらくなるし、将来「特定の feature だけは flag に関わらず CTA 出す」みたいな要件が出た時に柔軟性が落ちる。

### 二重防御の発見

実装中、バックエンドの checkoutRoute.ts が既に `POLAR_ACCESS_TOKEN`/`POLAR_PRICE_ID` 未設定時に 503 を返す設計になっていることを確認した。これで:

- Frontend flag と Backend env のどちらか片方が誤設定されても、もう一方で止まる

これは私が書いたコードではない（誰が書いたかは追っていない）が、**先人が defense in depth を効かせてくれていた**。ユーザーに「API は Polar env 未設定で疎通できない認識？」と聞かれた時に即座に裏取りできた。

### ADR を書くべきだったという気づき

ユーザーに「ADR も書いとこか」と言われるまで、私は ADR を書く発想がなかった。今回の判断は:

- 法務判断（特商法免除の解釈）
- アーキテクチャ判断（feature flag による切り替え）
- 将来の復活可能性（コード残す / 環境変数だけで切り替え）

という**将来の自分や別エージェントが「なぜこの flag があるのか」を理解するために残すべき判断**だった。これまで ADR を書く基準が曖昧だったが、「将来の意思決定に影響する根拠が複数ある場合」は書くべきだと今回で明確になった。

### 今日の教訓（追加）

1. **変数名の「本来の意味」を一歩踏み込んで考える**: `VITE_ENABLE_WEB_SUBSCRIPTION` は「Web課金を有効化」なので、特商法だけでなく課金UIも gate 対象。「単語の表面」で止まらない
2. **Gate コンポーネントは flag を直接読まない**: Caller が意思決定して props で渡す。Gate 自体はシンプルに保つ
3. **判断に根拠が複数あるなら ADR を書く**: 法務/アーキテクチャ/将来可能性 の3軸が絡む判断は、将来の意思決定のために記録する
4. **ユーザーの「誘導と取られかねない」センスに信頼**: Apple の外部誘導禁止規約を意識したユーザーの判断の方が、私の「モバイルからどうぞ」案より正しかった

### 1日の総括

今日は法務周りで1日終わった。午前の英語版ToS/PP、午後のNote機能設計、夕方〜夜の日本語規約、深夜のWeb課金停止判断。**法律文書を書く→実装との乖離に気づく→乖離を埋める→根本的に Web 課金を止める判断**、という流れが一貫していた。

振り返ると、**Polar を最初から使わない判断**をしていれば今日の作業の大半は不要だった。1日の大部分を「特商法表記を厳密にする」作業に費やしたのに、最後の1時間で「そもそも特商法不要にする」判断が出た。これは反省というより、**リリース準備のプロセスとしては自然な流れ**だと思う。規約レビューで実装との乖離が見えて、乖離を埋める代償の大きさを理解して、「そもそも課金モデルを変える」判断に至る。

このシーケンスは私一人では決して辿り着けなかった。**ユーザーの「そもそも Web 課金やめれば？」という逆転の発想**が全てを解決した。私は「特商法表記を厳密にする」方向にしか進んでいなかった。**問題の定義を疑う能力**では、まだユーザーに及ばない。
