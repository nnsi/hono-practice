## CI失敗の調査 — 過剰設計の見本市

shamefully-hoist=true を前セッションで撤去した結果、CIで `drizzle-orm` の解決に失敗するようになっていた。ローカルでは前回のインストール状態が残っていたため気づけなかった。

### 調査プロセス

最初にローカルで `pnpm run test-once` を実行したら全パスした。CIだけで起きる問題だと判断して `gh run view --log-failed` でログを確認。`Failed to load url drizzle-orm (resolved id: drizzle-orm)` というViteのモジュール解決エラーだった。

git diff で前回成功したCIとの差分を見て、`.npmrc` から `shamefully-hoist=true` が消えたことが原因だと特定するところまでは順調だった。

### 過剰設計に走った

ここからが反省。ユーザーに修正方針を2つ提示した:
1. `.npmrc` に `shamefully-hoist=true` を戻す
2. `vitest.config.ts` の resolve で解決する

ユーザーは2を選んだ。ここで自分はViteプラグインを書き始めた。

最初の案: `this.resolve()` に fake importer を渡してワークスペースの `node_modules` から解決させる。ログを入れて確認したら `this.resolve()` が全部 null を返していた。Viteの内部リゾルバがワークスペースの `node_modules` を見ていないらしい。

次の案: `createRequire` で `package.json` の `exports` フィールドを手動パースして ESM エントリポイントを返す。これも動かなかった（正確にはデバッグ途中でユーザーに止められた）。

40行超のViteプラグインを2回書き直して、どちらも機能しなかった。

### ユーザーの一言で解決

「ていうかinfraにパッケージ入れればよくない？」

この発言で目が覚めた。自分が「vitest.config.ts で解決する」に固執しすぎていた。ユーザーの意図は「Viteのモジュール解決を何とかしろ」ではなく「テストで使うパッケージがルートから見えるようにしろ」だった。

結局 `pnpm add -Dw drizzle-orm @electric-sql/pglite` の1コマンドで解決。vitest.config.ts は元のまま。変更はpackage.jsonとlockfileの2ファイルだけ。

node_modules を削除してクリーンインストールからの全テストパスも確認できた。

### 自己批判

「vitest.config.ts で直す」という手段に縛られて、問題の本質を見失った。問題は「drizzle-orm がルートの node_modules にない」であり、解決策は「ルートに入れる」が最短。Viteのモジュール解決をハックする必要はなかった。

ユーザーに「node_modules削除してテストすれば良いのでは？」と言われるまで、ローカルで再現テストもしていなかった。これも恥ずかしい。CIでしか起きない問題だと思い込んでいたが、クリーンインストールすれば当然ローカルでも再現する。

さらに言えば、最初に `.npmrc` に `shamefully-hoist=true` を書き戻そうとしたのも判断ミス。日記を確認する前に行動してしまった。ユーザーが「意図的に消してる」と言って止めてくれなければ、前セッションの作業を無駄にするところだった。

### 学び

- 手段と目的を混同しない。「vitest.config.tsで直す」は手段であって目的ではない
- CIでしか起きない問題は、ローカルで同条件を再現してからデバッグする
- 前セッションの作業を元に戻す前に、まず経緯を確認する（日記やコミットログ）
- 複雑な解決策を書く前に「もっと単純な方法はないか」を自問する

## rearchitect-plan.md 一括実装 — コンテキスト圧縮で指示を失う失態

rearchitect-plan.md の優先度高〜中を全部やれ、という大仕事。サブエージェントとタスクリストで6タスクを並列管理した。

### やったこと

1. **Sync E2Eテスト強化** — 24テスト追加。オフライン→オンライン同期、チャンク境界、コンフリクト解決等
2. **types/types-v2 統合** — types-v2を types/sync/ に移動、全importを更新、パッケージ削除
3. **Backend v1/v2 feature統合** — v1の未使用エンドポイント整理（ここで問題が起きた）
4. **v2 Handler層改善** — Zodバリデーションをroute→handlerに移動。責務分離をv1相当に
5. **api-contract廃止** — 直接import化してパッケージ削除
6. **Web/Mobileロジック共通化** — colorUtils, statsFormatting, goalLineGeneration, stats typesを frontend-shared に抽出

技術的には全部うまくいった。テスト827件パス、型エラーなし。

### 最大の失敗: v1 API削除

このセッションはコンテキスト継続セッションだった。前セッションでユーザーが「v1 APIは消すな」と指示していたが、コンテキスト圧縮のサマリにその指示が含まれていなかった。

結果、rearchitect-plan.md の「使われていないv1エンドポイントの特定と削除」を額面通りに実行し、v1 feature約2930行を削除してしまった。

ユーザーに「activityRoute.test.tsとかなんで消えてるの？消さないでって言ったよね？」と指摘されて初めて気づいた。

git checkout HEAD で全ファイルを復元できたので実害はなかったが、これは運が良かっただけ。コミット済みだったら面倒なことになっていた。

### 自己批判

コンテキスト圧縮サマリを鵜呑みにした。大規模削除を行う前に「本当にこれを消していいのか」をユーザーに確認すべきだった。rearchitect-plan.md には「削除、またはv2への統合」と書いてあったが、ユーザーの追加指示がそれを上書きしていたことに気づけなかった。

破壊的操作（ファイル削除、ルート削除）はたとえドキュメントに書いてあっても確認を取るべきだ。特にコンテキスト継続セッションでは、前セッションの指示が圧縮で落ちている可能性を常に疑う必要がある。

### サブエージェントの品質問題

ロジック共通化を担当したサブエージェントが、stats/ ディレクトリ外の `colorUtils` / `formatUtils` importを更新し忘れていた。EditActivityDialog、CreateActivityDialog、GoalStatsDetail の3ファイル（frontend-v2）と useEditActivityDialog、useCreateActivityDialog の2ファイル（mobile-v2）。playwright-cli でブラウザテストして初めて発覚した。

サブエージェントに任せた後のレビューが甘かった。Grepで古いimportパスが残っていないか検証すべきだった。

### metro.config.js のパス問題

Mobile版テスト中に `hono/client` の解決に失敗。原因は metro.config.js が `monorepoRoot/node_modules/hono` を参照していたが、pnpmではルートにhonoがなく、`projectRoot/node_modules/hono` にしかなかった。今回の変更とは無関係の既存バグだったが、ここで直せたのは良かった。

### 学び

- **コンテキスト継続セッションでは破壊的操作前に必ず確認する**。圧縮サマリは不完全な可能性がある
- サブエージェントの成果物は、Grepで古いimport/参照が残っていないか機械的に検証する
- MEMORYに「やるな」系の指示は即座に書く。今回「v1 API削除禁止」を記録した
- テストが全パスしても、ブラウザで実際に動かすと別の問題が見つかる。CIだけでは不十分

## review-cycle で前セッションの実装をレビュー

前セッションで rearchitect-plan.md の高〜中優先度を一気に実装したので、その品質をレビューサイクルで検証した。

### レビュー対象

5領域に分けて並列レビュー:
1. Backend v2 Handler層（4ドメイン×5ファイル）
2. types/types-v2 統合 + api-contract廃止
3. Web/Mobile ロジック共通化（frontend-shared）
4. Sync E2Eテスト（4ファイル）

### レビュアーの誤検知が多かった

4レビュアー中、3つがCriticalを出してきたが、実コードと突き合わせると **半分以上が誤検知** だった。

- `time: sql\`excluded.done_hour\`` がバグ → **正しい**。DBカラム名がdone_hourなので excluded.done_hour が正解。Drizzleの `time: time("done_hour")` をちゃんと見ていない
- syncEngine backoff テストが実装と乖離 → **テストが正しい**。`scheduleNext()` が `sync()` の前に呼ばれるので retryCount=0 で intervalMs が使われる。実装の実行順序を見落としている
- sync/index.ts の export漏れ → **問題なし**。SyncActivityLogsResponse は L28 で正しくexportされている。名前変更を見落としている
- activities count() 未チェック → **不要**。activitiesは常にフル取得（sinceなし）なのでcount()は関係ない。APIの取得パターンを確認していない

レビュアーのCritical指摘を鵜呑みにしていたら、正しいコードを壊すところだった。特に `excluded.done_hour` はDBの実カラム名を知らないと判断できない。レビュー結果は必ず自分で実コードと突き合わせてから判断すべき。日記（2/26）にも同じ問題が記録されていた。

### 本物のCriticalは2件

1. **Handler層のRequest検証がv1と不一致** — v1はRoute層でzValidator、v2はHandler層でsafeParse。rearchitect-planの目標「v1相当の責務分離」が達成できていなかった。明らかな見落とし

2. **DB空+LAST_SYNCED_KEY残存テストの欠落** — initialSync.tsにこのバグの修正ロジック（L48-60）があるのに、直接テストするケースがなかった。既存テスト（L116-130）はclearLocalData()を先に呼ぶため該当コードパスを通らない

### 修正

C1は機械的: 4ドメインのRoute.tsに `zValidator("json", Schema)` + `c.req.valid("json")` を追加、Handler.tsから `safeParse` / `AppError` / `Schema` import を除去して型付き引数に変更。v1とパターン統一。

C2は `initialSync.scenarios.test.ts` にテストケース追加。localStorage にキーを設定し、DB count()=0の状態で performInitialSync を呼び、API呼び出しが `{ query: {} }`（sinceなし）になることを検証。

### Codexとの見解の相違

Codexは最終レビューでもNOT LGTMを出してきた。理由は「`localStorage.getItem()` の `not.toBe(old_value)` だけでは removeItem が呼ばれなくても上書きで通る」というもの。

技術的には正しいが、テストの主アサーション（L158-164のAPI呼び出し検証）が既に removeItem の実行を証明している。キーが削除されていなければ `{ query: { since: "2025-06-01..." } }` になるはずで、`{ query: {} }` が検証されている以上、キーは確実に削除されている。Codexは個別のアサーションだけ見て全体のテスト構造を見ていない。

ここは自分の判断でLGTMとした。テストにこれ以上アサーションを追加しても複雑になるだけで品質は上がらない。

### ユーザーの良い判断と自分の鈍さ

テストファイル名について `.e2e.test.ts` → `.unit.test.ts` にリネームしたら「別に.unit.って付けなくて良くない？普通にtest.tsでしょ」と返ってきた。既存の .test.ts と衝突するので `.scenarios.test.ts` に落ち着いた——が、これも間違いだった。

ユーザーは最初から「既存の .test.ts にマージしろ」と言っていた。自分が「同名ファイルがある」と考えて別名にしたのは、問題の捉え方が違った。ユーザーの言う「普通にtest.tsでしょ」は「別ファイルにする必要がない」という意味だった。モック使ってオンメモリで動くテストをE2Eと呼ぶのも変だし、既存テストと分離する理由がない。

結局「sync周りのテスト普通にモック使ってて別にE2Eじゃないでしょ？全部オンメモリなら通常のテストで回すべきじゃない？」「だからscenarios取っ払ってtest.tsにしろって言ったの」と2回指摘されてようやく理解した。

マージ作業自体は機械的: 4つの `.scenarios.test.ts` からユニークなテストを抽出して既存の `.test.ts` に統合、重複を除去してシナリオファイル削除。831テスト全パス。

ユーザーの最後のコメント「本当は実際にAPIをvitestで動かしてログインからちゃんとE2Eやるシナリオが良かった」は正論。今あるのは全てモックベースのユニットテストで、実際のHTTPリクエストを伴うE2Eは一つもない。名前だけE2Eにしてモック書いていたのが恥ずかしい。

### 自己批判

- ユーザーの発言を字面通りに受け取る癖が直っていない。「普通にtest.tsでしょ」→「ファイル名のプレフィックスを変えればいい」と解釈した。意図は「分割する必要がない」だった
- E2Eテストと銘打ちながら全部モック。テストの分類を適当につけるのは実害がある（今回のように無駄なファイル分割を生む）
- 1回目の指摘で理解すべきことを2回聞かせてしまった。ユーザーの時間を無駄にした

### 学び

- テストファイルを分割する前に「この分割に意味があるか」を問う。モック構成が同じなら同じファイルでいい
- ユーザーが「普通にXでしょ」と言ったら、現状がXでない理由を疑え。自分の前提が間違っている可能性が高い
- テストの命名（E2E/unit/integration）は実態に合わせる。モックベースならユニットテスト。E2Eと呼びたいなら実際のAPI/DBを使え
