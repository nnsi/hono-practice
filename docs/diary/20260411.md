## 反省

### マシンクラッシュ後のリカバリは悪くなかった

クラッシュした時点で並列エージェント 3 体 + 自分の編集が混在していて、どこまで進んだか一見わからなかった。`git status` + `git diff` でファイル単位に現状を読んで、どの変更が前日セッションから引き継がれたもので、どれが今回セッションのエージェントが書いたものかを識別できた。エージェントの完了報告が失われても差分が正であれば再現可能、という当たり前の話だが実地で確認できたのは大きい。

ただし一点反省: サブエージェント側の output file が 0 bytes のままで完了通知も来ない状態を「エージェントが動いている」と解釈して待ち続けたが、結局クラッシュ後に親側で tsc を走らせたら通ったので、実際にはクラッシュ前後で並列エージェントは既に終わっていた可能性が高い。**状況が不明なときは「待つ」より「現状を計測する」を先にやるべき**だった。

### knip install で vite の peer dep が枝分かれして tsc が爆発した

knip を入れたら pnpm-lock.yaml の vite 6.4.1 の peer 付きハッシュが 2 つに分裂し、「Two different types with this name exist, but they are unrelated」のヘルメット tsc エラーが大量発生した。原因は knip が jiti を 1.21.7 → 2.6.1 に引き上げて、vite-tsconfig-paths だけが新 jiti で再解決、他の vite 参照パッケージが古いままだったこと。`pnpm.overrides` に `jiti: "2.6.1"` を追加して `pnpm install` し直したら tsc 通った。

**パッケージ 1 個追加しただけで型が壊れる monorepo、恐ろしい**。pnpm 10 の strict hoisting は依存の境界を明確にしてくれて良いのだが、peer dep の variant が分裂しやすい欠点がある。knip 導入の副作用がここまで大きいとは予想してなかった。

## 学び

- **knip の false positive 対応パターン**: (1) `/** @public */` JSDoc タグ（runtime で namespace import される drizzle relations などに有効）、(2) `knip.json` の `ignoreDependencies`（lefthook などの config file-only 参照）、(3) entry パターン追加（docs/icon-prototypes など明示的な entry 指定）。状況に応じて使い分ける
- **pnpm overrides で peer variant を固定できる**: 今回は jiti を固定したが、同様のパターンで transitive dep のバージョン分裂を強制解消できる。monorepo の依存管理トラブル対応の引き出しとして覚えておく
- **`vi.mock(specifier)` は specifier 解決に実パッケージが必要**: 前日 uuid で学んだ教訓だが、i18next でも同じ構造。「production コードで直接 import していない = 依存削除可」は test code を見ないと成立しない。packages の deps を消す前には必ず test file の mock 宣言も grep する
- **bcryptjs v3 は型定義を同梱しており `@types/bcryptjs` は不要**: v2 系では別パッケージだった型が v3 で本体に統合された。`@types/*` は定期的に本体の同梱有無を確認する価値あり
- **drizzle の `* as schema` は knip が追えない**: 名前空間 import で動的に参照される export は静的解析ツールの死角。drizzle に限らず、`import * as X from` パターンで runtime ディスパッチする設計全般に同じ弱点がある
