# 2026-03-23

## Android実機フィードバック対応 — TextInput地獄

ユーザーがAndroid実機で触ったフィードバック4件を修正するセッション。結果的にTextInputの挙動に大半の時間を費やした。

### やったこと

**元々の4件のフィードバック:**
1. 入力欄が狭い → IMESafeTextInputのAndroid用パディングがNativeWindを上書きしていた
2. モーダルがスクロールできない → ScrollViewにflexShrink追加
3. GoalやTaskのカレンダーが表示されない → OverlayPortalからModalに変更（Modal内でOverlayPortalが効かない問題）
4. NumpadModeの高さ変動 → minHeight追加

**追加で発生したTextInput問題:**
- iOS: 文字入力するとheightが変動
- iOS: defaultValueで初期テキストを入れると垂直位置がズレる
- Android: 数値入力時にTextInput高さが変動
- 全プラットフォーム: 入力欄のサイズがバラバラ

### 最終的な解決策

`IMESafeTextInput`に以下を集約:
- `defaultValue`をuseRefで初回のみ固定（再レンダー時のprop変更を防止）
- 単一行: `height: 44`固定（膨張も縮小もしない）
- iOS: `paddingTop: 0, paddingBottom: 7`（defaultValueテキストの下ズレを非対称パディングで補正）
- Android: `paddingVertical: 8`（対称パディング）
- 全TextInputをIMESafeTextInputに統一
- className: `text-base`に統一

### 反省

**最初から全体を設計すべきだった。** パディング削除 → iOS壊れた → iOS用パディング復活 → 今度はActivity編集が壊れた → h-*をpy-*に → まだ直らない → defaultValue ref化 → まだiOSズレる → height固定 → 非対称パディング復活... と7回OTAを重ねた。

根本原因は最初の調査で見抜けたはず:
1. IMESafeTextInputのインラインstyleがNativeWindのclassNameを上書きする
2. iOSのdefaultValueは垂直テキスト配置が異なる
3. 素のTextInputとIMESafeTextInputが混在している

これら3つを最初に把握していれば、「height固定 + iOS非対称パディング + 全TextInput統一」を1回で出せた。

**ユーザーの「根本的に別のフォームとかに出来ない？」「全部統一していいから」という提案が正しかった。** 自分はインクリメンタルに直そうとして何度もOTAを無駄にした。ユーザーの方がアーキテクチャ判断が的確だった。

### その他

- カレンダーのOverlayPortal→Modal変更は綺麗に決まった。CalendarCardを分離したのも良かった
- ModalOverlayのスクロール修正（flexShrink: 1）も一発で解決
- TaskActivityPickerの抽出でTaskCreate/EditDialogの重複コードを解消できた
- CreateLogDialogのActivitySelectOverlay抽出でファイルサイズも改善

TextInput以外の修正は全部スムーズだったのに、TextInputだけで8回OTAを出した。React NativeのTextInputはプラットフォーム差異の塊で、NativeWindとの相互作用も複雑。次回同様の問題が来たら「まず固定height + プラットフォーム別インラインstyle」から入るべき。

## Android autoFocus キーボード問題

### やったこと

Android実機で`autoFocus`を設定してもキーボードが表示されない問題を修正。IMESafeTextInputで`autoFocus` propをインターセプトし、Androidではネイティブの`autoFocus`を渡さず、代わりに`useEffect`で200ms後にプログラム的に`.focus()`を呼ぶようにした。iOSは従来通り。

修正は`IMESafeTextInput.tsx`の1ファイルのみ。ManualMode、TaskCreateDialog、CreateActivityDialog、CreateApiKeyDialogの4箇所全てが自動的に修正される。前回のセッションでIMESafeTextInputに挙動を集約した判断が早速活きた。

### 調査で学んだこと

ユーザーに「今もそうなの？」と聞かれてWebで調べた。React Nativeコア側では2020年にAndroidネイティブautoFocus実装が入っているが、モーダル内・画面遷移時など「Windowフォーカスが安定しないタイミング」では依然としてキーボードが出ない。2025年時点でも未解決で、遅延`.focus()`が標準的なワークアラウンド。

最初「既知の問題です」と断言したが、ユーザーの「今もそうなの？」は正当な疑問だった。自分の知識カットオフが2025年5月なので、その後に修正されている可能性はあった。調べて裏を取ったのは良かった。知識に自信があっても最新状況の確認を怠らないようにしたい。

### 前回セッションからの教訓の連続性

昼のセッションで「IMESafeTextInputに全TextInputの挙動を集約」した設計判断のおかげで、今回のautoFocus修正が1ファイルで済んだ。各コンポーネントに個別ワークアラウンドを書く必要がなかった。共通化の運用価値がすぐに実証された形。
