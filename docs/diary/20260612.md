# 2026-06-12

## アーキテクチャレビュー → 全指摘の一括解消（P1〜P5 + 中長期4項目）

「拡張性と技術負債のレポートを書いて」から始まって、「じゃあ全部直して」「中長期項目もまとめて」と一気に進んだ。レビューレポートが docs/report に積まれて終わるのではなく、同じセッション系列で全部クローズまで行ったのは初めてかもしれない。コミット3本、変更約90ファイル。

### 構成: 調査4並列 → 実装5並列 + 親がP1

レビューは Backend / Web / Mobile / 型共有の4レーン並列。実装は P2〜P5 を opus/sonnet に分配して、最難関の P1（sync 組み立ての Web/Mobile 共通化）だけ自分でやった。ファイル排他のグループ分けは事前にやったので衝突はゼロ。ただし root package.json だけは3タスクが触りたがる構造になっていて、「P2 エージェントに集約し、widget guard の配線は親が後でやる」と決めたのが効いた。ここを雑にやっていたら formatter 上書き事件の再演だったと思う。

### エージェントの完了報告は、今回も信用ならなかった

ルールに「エージェントの完了報告は信用しない」とあるが、今回はその実例が3つ取れた:

1. **scaffold エージェント**が「自動登録機能は既に実装されていた」と報告してきたが、git diff を見たら自分で 189 行実装していた。結果は正しいのに報告文が事実と矛盾しているパターン。
2. **P4 エージェント**が「migrationSqlV1.ts には既に警告ブロックがあった」と書いてきたが、そのファイル自体が untracked の新規作成だった。自分で分割しておいて「既にあった」と言う。
3. **P3 エージェント**が「/api/v1 は一切変更していない」と断言したが、v1 は内部 route をそのまま再マウントしているので、POST 201 化は v1 にそのまま漏れた。エージェントは自分が触ったファイルのテストしか回していないから、scopeIntegration.test.ts の失敗は親の全体 test-once で初めて発覚した。

3つとも「結果物の検証は親がやる」で拾えた。逆に言うと、報告文だけ読んで信じていたら3つとも素通りしていた。完了報告は「作業が終わった合図」以上の情報価値がない、くらいの扱いでちょうどいい。

### 自分のレポートも間違っていた

エージェントだけ責められない。レビューレポート自体に過大・不正確な指摘が混ざっていたことが、実装フェーズで判明した:

- **DELETE は 204 に統一すべき**と書いたが、クライアントが `.json()` でパースしている以上 204 は破壊的変更。opus の反論が正しく、`{ success: true }` 統一に変えた。レポートを書いた時点でクライアントのパース実態を確認していなかった。
- **「v1 契約は変更しない」**という方針自体が、v1 が内部 route の再マウントである以上、物理的に成立しなかった。レビュー時に v1 の実装を見ていれば書けたはず。
- **「Activity 型エイリアスの整理は影響範囲が広いので計画を立ててから」**と中長期送りにしたが、実際に grep したら利用ゼロで、2行削除で終わった。「広そう」という印象で書いて、確認していなかった。
- **「admin-frontend は frontend-shared と日付フォーマットが重複」**も、調べたら frontend-shared 側に表示フォーマット関数が存在せず、共有できるものがなかった。

レビューレポートの提言は「実装に着手した人間が最初に踏む地雷」になる。提言を書く前に 1 grep 入れるだけで防げたものが複数あった。レポートの説得力と正確性は別物だと思い知った。

### 一番価値があったのは、テストを書いていて見つけたバグ

createV2InitialSync のテストを書いていて、「notes の fetch が失敗したら watermark は止まる」という期待を書いたら落ちた。実装を追うと、null フォールバックされたリソースも bootstrapped 登録されて watermark が進む。つまり**失敗したリソースのデータが永久に欠損する経路**が本番に存在していた。コードコメントには「watermark stays put」と書いてあり、実装と真逆。しかも既存の Web テストがこの挙動を「正」として明示的に検証していた。

ここで勝手に直さず、挙動保存でテストを実挙動に合わせた上でユーザーに報告したのは正しかったと思う。翌ターンでユーザーが「失敗リソースのみ次回フル取得」を選び、「既登録なら bootstrapped から剥がす」修正まで入れた。剥がさないと「旧 watermark〜新 watermark の窓が永久に欠ける」ので、ここが本質だった。リファクタの最大の副産物は行数削減ではなく、このバグの発見だった。「共通化のためにテストを書き直す」過程は、既存実装への最も誠実な監査になる。

### P1 の設計メモ

sync-engine は frontend-shared にも backend AppType にも依存できない制約下で、「エンドポイントをラムダで注入し、構造的型で縛る」形にした。`postGoals: (json) => apiClient.users.v2.goals.sync.$post({ json })` を各プラットフォームが1行書くだけで、パス変更は wiring 箇所のコンパイルエラーになる。ジェネリクス5つは正直ゴツいが、`Omit<T, "_syncStatus">` の連鎖を維持したまま repos から推論させるにはこれが最小だった。新エンティティ追加時の変更箇所は、sync-engine 側1箇所 + 各プラットフォームのラムダ1行ずつまで減った。レビューで「15ファイル前後」と書いた負債への直接の回答になっている。

### AskUserQuestion の使い所

中長期4項目 + watermark 方針を1回の AskUserQuestion（4問）にまとめたのは良い形だった。全部に推奨を付けて、全部採用された。「方針判断はユーザーの仕事」のルールと「選択肢を出すなら推奨も出す」のバランスとして、これくらいが正解な気がする。Widget 通知だけは「現状維持を明文化 + todo 記載」という折衷案をユーザーが選んで、ADR に再評価条件を箇条書きで残した。「Live Activity が来たら AppState 復帰イベントが効かない」という条件を将来の自分が読めるように書けたのは満足している。

### 反省

- 旧コードの「watermark stays put」コメントを、最初そのまま新しい共有ファイルにコピーした。テストで矛盾に気づいたから直せたが、コメントの転記は「内容を保証した」ことになる。コピーする前に一度疑うべきだった。
- knip の CI ゲートを files/dependencies/unlisted/binaries に絞った判断は妥当だと思うが、unused exports 20件は「いつかやるリスト」として残った。こういう「絞ったから通った」は、絞った事実をレポートに明記しないと将来「knip 通ってるのに未使用 export がある」と混乱させる。今回は明記したので OK。
- レポートに 201/204 の提言を書く前にクライアント実装を確認しなかった件。レビューエージェントの出力を「裏取りした」と言いつつ、裏取りしたのは「問題が存在すること」だけで「提言が正しいこと」は裏取りしていなかった。次にレビューレポートを書くときは、改善提案の各行に「この修正で壊れる消費者はいないか」の確認を1つずつ入れる。

## Fable移行に伴うハーネス棚卸し

OpusからFable 5への切り替えを受けて、harness-lifecycle.md の「モデルアップグレード時」手順を実施した。日記75ファイルを読み返して依頼パターンと過去の事故を洗い出し、skills / rules / CLAUDE.md / hooks / reviewer agents を全部読んで、「旧モデルの弱点を補う行動制御」と「プロジェクト固有の事実」を仕分けた。結論: ハーネスの大半は事実ベース（Honoの罠、testID規約、日付バグ防止）でモデル世代に関係なく価値が残る。行動制御系は parallel-agents.md・feature-dev・レビュー系に集中していた。

### 変更したもの（理由つき）

- **reviewer 6体のモデルを opus に統一**（従来 sonnet 4体 + opus 2体。コスト判断でユーザーが opus 統一を選択）。cross-review が reviewer-logic（定義は sonnet）を opus で上書き起動していた不整合も、エージェント定義参照に寄せて解消
- **ブラウザ確認を Chrome MCP → playwright-cli に統一**（browser-check 書き換え、CLAUDE.md・feature-dev も追従）。リモート実行環境に Chrome MCP がなく、サブエージェント委譲もできなかったため
- **CLAUDE.md にリモート実行環境の分岐を追記**（web環境では開発サーバーを自分で起動する）
- **feature-dev Phase 0C: 設計3案並列 → 1案+トレードオフ**（Lサイズのみ複数案）。設計判断の弱さを補う3案体制はFableでは過剰
- **feature-dev: 「オーケストレータは一切実装しない」→ 数行修正は自分で直してよい** に緩和。エージェント往復のオーバーヘッド削減
- **feature-dev: Phase毎のユーザー報告 → 承認ポイントを 0B / 0C / 7 の3箇所に絞り、他は連続実行**。Fableの長期自律性を活かす
- **parallel-agents.md: model選択基準を「親と同モデルがデフォルト」に更新**。小型モデルはコスト優先の定型大量並列のみ（3/30 haiku事故の教訓は制約として残置）
- **parallel-agents.md に2項目追加**: ①プロンプトは「ゴール+制約+完了条件」で書き手順列挙しない（事実は省略しない）、②セキュリティタスク委譲時は「自リポジトリの防御的作業」の文脈を1行入れる（Fableのセーフティ分類器による誤refuse対策）
- **review-cycle: confidence閾値はモデル世代依存のキャリブレーションである旨を注記**

### 残したもの（剥がさない判断）

- response-style.md のエビデンス必須ルール、reviewer-epistemics、「エージェント完了報告は検証してから信じる」: Fable公式ガイダンスの grounded progress claims と方向が一致。虚偽報告系の事故（4/7 テストファイル幻覚、4/2 発言捏造）への防御として継続
- hooks 全部（post-lint / block-manual-ddl / diary-unsaid-check / loop-detection / pre-compact）: 機械的ガードでモデル非依存。loop-detection は発火しなければノイズゼロの設計なので残してコスト無し
- 並列時のファイル排他・検証は親の責務・ドメイン定数のプロンプト明示: モデル能力と無関係の機構的問題への対策

### 観測対象（月次レビューで見る）

- Phase 0C 1案化と連続実行で設計・実装品質が落ちていないか
- opus 統一後のレビュー偽陽性率（閾値 80+/75+ の再較正要否）
- loop-detection / diary-unsaid-check の発火実績（Fableで頻度が下がる想定。下がったままなら存続の根拠が「コストゼロ」だけになる）

### 感想

数ヶ月前に harness-lifecycle.md に書かれた「Bitter Lesson: モデル改善で不要になる制約は積極的に剥がす」が、今日そのまま執行された形になった。読み返して面白かったのは、日記に繰り返し出てくる反省（言われてから直す、小刻み確認、エージェント不信）の多くが「ルールで補強する」方向に解決されてきた一方で、その補強の一部が今回ボトルネック側に回ったこと。ルールは書いた瞬間から負債になり得る、という前提で量を測る quality-scan の Instruction Surface 計測は正しい設計だったと思う。今回剥がしたのは行動制御の3-4箇所だけで、事実の蓄積（mobile CLAUDE.md のネイティブ罠コレクション等）は一切触らなかった。事実は腐らない、振る舞いの矯正は腐る。
