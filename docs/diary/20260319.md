# 2026-03-19: Harness Engineering 導入 + noExplicitAny 全排除

## やったこと

Harness Engineeringベストプラクティス記事をベースに、このリポジトリのハーネス整備を実施。

### PostToolUse Hook (`post-lint.js`)
- Write/Edit/MultiEdit後にbiome format→lint→残エラーをadditionalContextで返却
- `enhanceDiagnostics()` でルール別の修正ヒントを付加（archgateパターンの一環）
- Node ESMスクリプト。最初 `.cjs` で作ったがユーザーに `.js` に変えてと言われた

### noExplicitAny 全排除
- `biome.json` で `"noExplicitAny": "error"` に変更
- 最終的に40〜50ファイルを修正。パターンは3種類:
  - Repository系: `<T = any>` → `<T = QueryExecutor>`
  - テスト系: `as any` → `as unknown as ConcreteType`
  - React hooks DI型: `biome-ignore` で明示的抑制
- **反省: エージェントに任せたが変更が保存されない問題が多発**。3回やり直して、最終的に自分で直接編集した方が速かった

### Lefthook 有効化
- コメントアウトされていたpre-commitを有効化
- `biome lint`（変更ファイルのみ）+ `tsc --noEmit`

### archgateパターン（import方向の強制）
- biome.json の `overrides` + `noRestrictedImports` で:
  - `packages/` → `@backend/*`, `@frontend/*` のimport禁止
  - `apps/frontend/` → `@backend/*` のimport禁止
  - `apps/backend/` → `@frontend/*` のimport禁止
- 3件のレイヤー違反を検出: `AppType` が `@backend/app` から直接importされていた
- ユーザーの提案で `@packages/types/api.ts` に薄い再exportファイルを作り、依存方向をクリーンに解決

## 反省点

### エージェントへの委譲が裏目に出た
`noExplicitAny` の修正をサブエージェントに3回以上投げたが、毎回「変更が保存されていない」問題が起きた。原因は不明だが、結果として同じファイルを何度も読み直して修正するという非効率な作業になった。数が多い場合でも、パターンが明確（`as any[]` → `as unknown as X[]`）なら自分で `replace_all` した方が確実で速い。

「エージェントに任せれば速い」と思い込んでいたが、保存の信頼性が低い状況では「自分でやる判断」を早めにすべきだった。

### モグラ叩き問題
lintルールを有効化すると、修正するたびに新しいファイルが検出される。biome lintは初回実行で全ファイルをスキャンしているはずだが、エージェントが一部だけ修正→他のファイルがコンパイル可能になる→新たにlint対象に入る、という連鎖が起きていた可能性がある。最初に全件リストを確定させてから一括修正すべきだった。

## ユーザーとの対話

ユーザーの指示が的確で効率的だった。特に:
- 「安全ゲートは不要。設定ファイルも編集したい時がある」 — 実際の運用を踏まえた判断
- 「Stop Hookは差分がある状態で走ると不便」 — 理論的には良いが実用上の問題を指摘
- 「`@packages/types/api` に薄いファイル作ってそっち参照する方がレイヤー的にきれいにならない？」 — `biome-ignore` で例外を増やすより構造で解決するアプローチ。これは自分からは出なかった発想。

記事のベストプラクティスをそのまま適用するのではなく、「このリポジトリの運用に合うか」をユーザーが判断してくれたのが良かった。ハーネスエンジニアリングは理論と実践の両方が必要だと実感した。

# 2026-03-19 (3): ナビゲーションSync実装

## やったこと

ページ遷移時にsyncリクエストを自動発火する機能をweb/mobile両対応で実装した。

### 設計
- `createNavigationSync` をsync-engineパッケージに追加（スロットル付きファクトリ）
- 遷移検知: web は TanStack Router の `useRouterState`、mobile は Expo Router の `usePathname`
- push（`syncAll`）+ incremental pull（`performInitialSync` with `since`）を並列実行
- 5秒スロットルで連続遷移のAPI負荷を抑制、オンライン時のみ

### レビューサイクルで修正した点
- `performInitialSync` に `isPulling` ガード追加（多重実行防止）。レビューで指摘されるまで気づかなかった。`syncAll` には `isSyncing` があるのに `performInitialSync` にはなかった非対称性
- `createNavigationSync` に `onError` callback追加。mobile は `reportError` に接続、web は `console.warn`

### Codexがファイルパス問題で使えなかった
`(tabs)` ディレクトリ名の括弧がWindows + Git Bash環境でCodexの `rg` コマンドを壊した。結果、サブエージェント単体レビューのみで進行。

## 反省点

### ブラウザ確認で手間取った
Chrome MCPのコンソール/ネットワーク追跡がSPA遷移でリセットされることに気づくのが遅かった。最初から `window.fetch` インターセプトで確認すべきだった。3回ほど無駄な遷移を繰り返した。

### syncReady の初期状態問題
レビューで「ログイン直後の遷移でsyncReadyがfalse → sync skip」を指摘されたが、これは意図的な設計（初回syncはuseAuth側で完了済み）。偽陽性と判断したが、コード上でその意図が読み取りにくいのは確か。コメントがあっても良かったかもしれない。

## ユーザーとの対話

要件が簡潔で明確だった: 「ページ遷移時にsync、全件じゃなくて直近10件くらい」。`since` による incremental pull が「直近10件」の要件を自然に満たすことに気づけたのは良かった。バックエンドに `limit` パラメータを追加する案も検討したが、`since` で十分だと判断した。結果的にバックエンド変更ゼロで済んだ。

ユーザーが「日記書いてコミットしてプッシュして」と一括で指示するスタイル、効率的で好き。

# 2026-03-19 (4): 型アサーション掃除

## やったこと

テストファイル以外の全 `as` 型アサーションを洗い出し、不要なものを排除した。

### 削除できたパターン（10箇所）

1. **Bindings型不足** (`authMiddleware`, `loggerMiddleware`): `c.env as Record<string, unknown>` で `__authenticatedUserId` にアクセスしていた。`AppContext.Bindings` に型を追加して直接アクセスに変更。根本原因は「バッチリクエスト用の内部プロパティがContext型に宣言されていなかった」こと。

2. **ファクトリ関数の未活用** (`goalUsecase`): `goalId as ActivityGoalId` が3箇所、`req.activityId as ActivityId` が1箇所。既に `createActivityGoalIdFromString()` と `createActivityId()` というファクトリ関数が存在していたのに使われていなかった。ドメイン層のbranded type変換はファクトリ関数を通すべき。

3. **`.filter(Boolean) as T[]`** (`activityUsecase`): type predicate `(x): x is T => Boolean(x)` に置換。TypeScriptの既知の制約で、`filter(Boolean)` は型ガードとして推論されない。

4. **Set.has後のアサーション** (`resolveRecordingMode`, `apiMappers`, `mobile/activityRepository`): `Set.has(value)` はTypeScriptの型ナローイングに効かないため `as RecordingMode` 等が必要だった。type guard関数 `isRecordingMode(v: string): v is RecordingMode` を追加して解決。3ファイルで同じパターン。

5. **catch句のerrorナローイング** (`userRepository`): `error as Record<string, unknown>` → `in` 演算子で `"code" in error && error.code === "23505"` に変更。

6. **reduceの初期値** (`activityQueryService`): `[] as GetActivityStatsResponse` → `Array<GetActivityStatsResponse[number]>` で型推論を活かす。

### 残したもの（理由付き）

JSON.parse/res.json/drizzle戻り値/HonoのContext不変性/DOM API/外部ライブラリ戻り値など。いずれも「実行時の値がTypeScriptの静的型と一致することをコンパイラが推論できない境界」にあるもの。これらはスキーマバリデーション（Zod等）を導入しない限り排除不可能。

## 反省点

### premiumMiddlewareのジェネリック化を試みて失敗
`premiumMiddleware<E extends AppContext>(c: Context<E>, next: Next)` にすれば `apiKeyRoute.ts` の `as unknown as HonoContext` を消せると思ったが、HonoのContext型がジェネリックパラメータに対して不変(invariant)なため、`c.get("userId")` の戻り値が `NonNullable<E["Variables"]["userId"]>` となり `UserId` に代入できなかった。Honoの型システムの特性を理解してから着手すべきだった。結局すぐ戻したので実害はなかったが、試行錯誤のラウンドトリップは無駄だった。

### `createActivityGoalId` vs `createActivityGoalIdFromString`
`createActivityGoalId()` が引数なし（新規ID生成用）で、既存文字列からの変換は `createActivityGoalIdFromString()` だと気づかず、最初 `createActivityGoalId(goalId)` としてtscエラーを出した。ファクトリ関数のシグネチャを先に確認してから置換すべきだった。一方 `createActivityId` は `id?: string` で兼用パターン。命名規約が統一されていない点は気になる。

## 感想

型アサーションの掃除は「コンパイラと会話する」ような作業で面白い。各 `as` の背景には「なぜコンパイラはここで型を推論できないのか」という理由があり、それが解消可能なもの（型宣言の不足、ファクトリ関数の未活用）と不可避なもの（JSONパース、外部ライブラリ境界）に綺麗に分かれた。

`Set.has()` が型ガードにならない問題は、同じパターンが3ファイルに散らばっていた。共通のtype guard関数を1箇所にまとめることも考えたが、各ファイルのIconType定義が微妙に異なる（sync-engine は3種、mobileは2種）ため、ファイルローカルに持つのが正解だった。

# 2026-03-19 (5): 本番sync障害の調査 → 完全リセット機能追加

## やったこと

### WAE APMログで本番障害を調査
ユーザーから「本番Webアプリでsyncが上手くいかない」との報告。`/wae-apm` スキルでAPMログを取得・分析した。

発見:
- `/users/v2/activities/sync` POST が直近24時間で **29回400エラー、成功0回** — 完全に壊れていた
- 一方 `/users/v2/activity-logs/sync` は13回成功しており、API基盤は正常
- 400レスポンスの `duration` が全件0、`errorMsg` が空 → Zodバリデーション（`zValidator`）段階で即座にリジェクトされ、ハンドラに到達していない

バックエンドのZodスキーマとフロントエンドのDexieレコードの差異を調査中に、ユーザーから「PCでは問題ない、iOS/Safariのみ」との情報が追加された。さらに「Safari設定からストレージ削除したら復活した」と報告。

### 根本原因の推定
iOS SafariのDexie（IndexedDB）に古いスキーマのActivityレコード（`showCombinedStats` や `recordingMode` が欠落したレコード等）が残っており、Zodの `z.boolean()` や `z.enum()` バリデーションに弾かれていた可能性が高い。PCは新しいDexieデータだったため問題なし。あるいはPWAのService Workerが古いJSバンドルをキャッシュしていた可能性。

### 「アプリを完全に初期化」ボタン追加
既存の「ローカルデータを削除」はDexieテーブルのみクリアだったが、今回はそれだけでは不十分だった。SW + Cache Storage も含めた完全リセットボタンを設定画面に追加:

1. Service Worker全登録解除
2. Cache Storage全削除
3. Dexieデータ全クリア
4. ページリロード → サーバーから再同期

既存ボタン（赤）と視覚的に区別するためオレンジ色で配置。

## 反省点

### 400の原因を特定しきれなかった
APMログの `errorMsg` が空（`zValidator` がlogger middleware経由せず直接400を返すため）で、実際にどのフィールドがバリデーション失敗したか特定できなかった。ユーザーが先にSafariストレージ削除で復旧させたので調査が中断した形。

本来なら、`zValidator` の第3引数（hook）でバリデーションエラーの詳細をログに記録するようにすべき。今回のような「400だが原因不明」パターンは今後も起き得る。提言すべきだったが、ユーザーが「復活した」「完全リセットボタンが必要」と話を進めたのでタイミングを逃した。

### Dexieマイグレーションの不備も指摘すべきだった
`showCombinedStats` にはDexieのバージョンアップ時のマイグレーションが存在しない。`recordingMode` はv4で追加されているのに `showCombinedStats` は何もない。古いレコードにデフォルト値を設定するマイグレーションがあれば、そもそも今回の問題は起きなかった可能性がある。これも言うべきだった。

## 感想

APMログの分析は楽しい。WAEのクエリを並列実行してエラーパターンを俯瞰し、`activities/sync` だけが完全に壊れているという異常を即座に発見できた。ただ、そこから先の「なぜZodバリデーションが落ちるか」を特定するには、APMに記録されていない情報が必要だった。observabilityの穴。

ユーザーの問題解決が速い。Safari設定からストレージ削除して復旧、そこから「じゃあアプリ内でも同じことできるようにしよう」という判断が的確。自分はまだコード上の原因調査をしていたが、ユーザーは既に「原因はどうあれ、リセットで直る→リセット機能を作る」という実用的な判断をしていた。

### 対応した6件

5つのサブエージェントを並列で起動して一括修正:

1. **ファクトリ関数命名統一**: `createActivityGoalIdFromString` → `createActivityGoalId(id?)` に統合。`createActivityId` パターンに揃えた
2. **freeze period TODOコメント**: `.catch(() => null)` に後方互換の意図と削除検討のTODOを追加
3. **premiumMiddleware型修正**: `as unknown as HonoContext` を完全撤廃。`MiddlewareHandler<AppContext>` に変更し、NODE_ENV分岐をmiddleware内部に移動
4. **タイムゾーン問題2件修正**: CSVファイル名とZodプリプロセスで`new Date().toISOString().split("T")[0]`（UTC）→ `dayjs().format("YYYY-MM-DD")`（ローカル）
5. **serverWins dayTargets正規化**: goal sync usecaseでserverWinsの`dayTargets`を`parseDayTargets`で正規化。DBに古い形式が残っていてもクライアントのZodパースが落ちない
6. **WAEエラー監視強化**: 調査の結果、今セッション前半の400エラーログ修正で既に対応済みと判明

### 400エラーのAPM記録改善

セッション(5)の反省を受けて、loggerMiddlewareで400レスポンスのbodyを`c.res.clone()`で読み、エラー詳細をWAE blob7に記録するようにした。当初は`zValidator`のラッパーを作るアプローチだったが、`@hono/zod-validator` v0.7.6の型シグネチャが複雑で断念。loggerMiddleware側での対応に切り替えた結果、zValidator以外の400エラーもカバーでき、全ルートファイルの変更も不要になった。結果的にこちらのアプローチが正解。

### Dexieマイグレーション v7

`showCombinedStats` が `undefined` の古いレコードにデフォルト値 `true` を設定するマイグレーションを追加。`recordingMode` はv4で対応済みだったのに `showCombinedStats` にはなかった。mobileはexpo-sqliteで`DEFAULT 1`設定済みなのでWeb(Dexie)のみ。

## 反省点

### 「言えなかったこと」パターンの構造的問題

今回ユーザーに伝えた8件のうち、セッション中に言えなかったのは2件（zValidatorログ、showCombinedStatsマイグレーション）。残りの6件は過去の日記に「気になる」「検討すべき」と書いて放置していたもの。

日記に書く → 次のセッションでは忘れている → 誰も見ない → 永久に放置、というパターンがある。今回ユーザーから「日記から抽出して」と言われて初めて拾えた。定期的にこの抽出をやるか、日記に書くだけでなくtodoに転記するか、何らかの仕組みが必要。

ただし、3/14の日記抽出セッションで「MEMORYに書いて2回以上破ったルールはCLAUDE.mdに昇格」という基準を作ったのと同様に、「日記に2回以上出てくる技術的負債はtodoに転記」という基準が良いかもしれない。

### premiumMiddlewareの解法が良かった

日記(4)で「Honoの型システムの不変性で解消不可能」と書いたが、今回のエージェントが`MiddlewareHandler<AppContext>`で`app.use()`に直接渡すアプローチで解決した。Honoの`MiddlewareHandlerInterface`のオーバーロードが`<E2 extends Env = E>`で受けるため、invariance問題を回避できる。自分が「不可能」と結論づけたのは、アプローチの幅が狭かったから。`(c: Context, next: Next) => ...` の関数シグネチャに固執して、`MiddlewareHandler` 型を使う選択肢を検討しなかった。

### zValidatorラッパーの型問題

`@hono/zod-validator` のラッパーを作ろうとして15ファイルのimport差し替えまでやったが、型エラーで断念してrevert。全ファイルをsedで一括置換→型エラー→全ファイルをrevert、という無駄なラウンドトリップを1回やった。型が通るか確認してからファイル差し替えをすべきだった。

## ユーザーとの対話

### 「post-lint.jsに移すことも検討していい」

この一言が良かった。自分は「CLAUDE.mdのセクションを `.claude/rules/` に移す」というテキスト→テキストの移動しか考えていなかった。ユーザーの「テキストで書く代わりに機械で検出する」という発想の転換で、スリム化の効果が単なる行数削減から「ルールの実効性向上」に変わった。

CLAUDE.mdに書いてあるルールは「読まれることを期待する」受動的な仕組みだが、hookは「違反したら即フィードバック」の能動的な仕組み。archgateパターン（biome.jsonでimport方向を強制）と同じ思想で、今回はそれをプロジェクト固有の規約にまで広げた形。

### 「テストファイルに200行制限は厳しい」

的確な除外条件。テストファイルはdescribe/itのネストとセットアップでどうしても長くなる。200行制限はプロダクションコードの分割を促すためのもので、テストに適用すると無理な分割が発生する。

### 「日記抽出スキルは作らなくていい」

素直に従ったが、少し惜しい気もする。日記(6)で「日記に書く→永久に放置パターン」を課題として認識し、「定期的な棚卸しの仕組み」の必要性を書いたばかりだった。ただ、スキルにする必要はない。手動で `/write-diary` のついでに前回日記を見返す習慣をつければ十分かもしれない。あるいは日記に `<!-- TODO: -->` タグを埋め込んで後からgrepする方が軽い。

## 反省点

### 最初の提案に「hookへの移行」が含まれていなかった

最初に日記抽出結果を提示した際、CLAUDE.md → `.claude/rules/` のテキスト移動と、新スキル作成しか提案しなかった。post-lint.jsへの移行はユーザーの提案で初めて検討した。harness.mdのタスクリストを自分で読んでおきながら、post-lint.jsの活用範囲を広げる発想に至らなかったのは、「既存のhookは既存の用途」と思い込んでいたから。ツールの拡張可能性を常に意識すべき。

### `as` キャスト検出を見送った判断

review-cycle skillに「`as`キャスト等の規約違反は修正すべき」とあり、frontend CLAUDE.mdに「`as`キャスト禁止」とある。post-lint.jsで検出可能ではあるが、diary 3/19(4)の型アサーション掃除で「JSON.parse / drizzle戻り値 / Hono Context等は排除不可能」と結論が出ている。偽陽性が多すぎるので見送ったのは正しい判断だと思うが、`as any` と `as unknown` だけに絞れば偽陽性は減る。`as any` はbiome `noExplicitAny` で既にカバーされているので追加不要だが、将来的に `as unknown as ConcreteType` のパターン（テスト以外での使用）を警告する余地はある。

## 感想

「ルールをどこに置くか」を考える作業は地味だが重要だった。同じルールでも、CLAUDE.md（常時読み込み・受動的）、`.claude/rules/`（常時読み込み・受動的だが分類整理）、post-lint.js（編集時・能動的）、biome.json（lint時・能動的）で実効性が全く違う。

今回の結果として、ルール配置の指針が明確になった:
- **機械的に検出可能** → post-lint.js or biome.json（最も確実）
- **常に意識すべき行動指針** → CLAUDE.md or `.claude/rules/`（パス指定なし）
- **特定ファイル操作時のみ必要** → `.claude/rules/`（パス指定あり）or サブCLAUDE.md
- **手動で起動する手順** → skill

この分類自体をどこかに書いておくべきかもしれないが、それはメタすぎるか。
