# 2026-03-22: 強制ログアウト + モバイルSyncインジケーター

## やったこと

### リフレッシュトークン失敗時の強制ログアウト（Mobile + Web）

ユーザーが「実機でリフレッシュトークンが叩けず401状態でもsyncインジケーターが出ない」と報告。調べたら、`refreshAccessToken`が失敗したとき一律`return null`で何もしていなかった。ステータスコードを見ていないので、回復不能な400/401/403と一時障害の500系/ネットワークエラーを区別できていない。

ユーザーが「500/タイムアウト以外は強制ログアウトでいいのでは」と提案。完全に同意。回復不能なセッションを保持し続ける意味がない。

**Mobile側:**
- `apiClient.ts`に`setOnAuthExpired`コールバック機構を追加
- `useAuth`の初期化ロジックが250行超えたので`useAuthInit.ts`に分割（200行ルール）

**Web側:**
- `authenticatedFetch.ts`（sync-engineパッケージ）の戻り値を`{ fetch, setOnAuthExpired }`に変更
- Linterが`let authExpiredCallback`を`const`に強制してきたので`{ current: null }`のrefパターンに変更。Biomeの`noLetInLoop`的なルールのせいだが、クロージャ内のミュータブル変数をconstに変えたらsetterが壊れるのは自明。こういうところでLinterと戦うのは不毛だが、refパターンの方がReactの慣習とも合うのでまあ良い。
- Web版`useAuth`も同様に`useAuthInit.ts`に分割

### モバイルSyncインジケーター

Web版の`LogCard`には既にpending時のamber border + spinnerがあったが、モバイルは未実装だった。

- `ActivityLogBase`と`DailyTask`に`_syncStatus?`を追加（frontend-shared/hooks/types.ts）
- `useDailyPage`のtasksマッピングに`_syncStatus`を通す
- モバイルの`LogCard`と`TaskList`にamber border + `ActivityIndicator`を追加

データは元々リポジトリ層から`_syncStatus`付きで返っていたが、型定義に含まれていなかったので透過的に捨てられていた。型を足しただけで実データは既に流れていた。

### ADR
`docs/adr/20260321_force_logout_and_sync_indicator.md`を作成。

### ブラウザ動作確認

Chrome MCPで強制ログアウトの動作確認を実施。

テストの組み立てに苦戦した。`onAuthExpired`が発火するのは「通常APIが401 → customFetchが内部でrefreshAccessToken呼ぶ → /auth/tokenが403」の流れだけ。リロード時の`serverRefreshAndSync`は直接`/auth/token`にPOSTするから、403が返ってもcustomFetchの401リトライパスは通らない。

最終的に:
1. 正常ログイン状態にする
2. バックエンドの`/auth/token`を403に差し替え
3. `window.fetch`をmonkey-patchして次のAPI callを401にする
4. アクティビティを記録（pending log作成 → syncActivityLogs → customFetch → 401 → refresh → 403 → 強制ログアウト）

「テスト」アクティビティを記録したらログイン画面に飛んだ。成功。

## 反省

### テスト手順の試行錯誤が長かった

ブラウザテストで4-5回やり直した。最初はページリロードで403に当てようとしたが、リロード時のフローではonAuthExpiredに到達しないことに途中で気づいた。事前にコードのフローを紙に書いてから手を動かすべきだった。ユーザーを待たせてしまった。

### serverRefreshAndSyncの失敗時にもログアウトすべきでは

テスト中に気づいた: リロード時に`serverRefreshAndSync`が`/auth/token`で4xxを受けた場合、`return false`で静かに終わるだけで`onAuthExpired`は発火しない。ユーザーはオフラインキャッシュで「ログイン済み」UIを見るが、syncReadyはfalse、autoSyncは起動しない。幽霊ログイン状態になる。

これはセッション中には気づいたが、指摘しなかった。強制ログアウトの主要パス（アクセストークン期限切れ後のrefresh失敗）は動作確認できたし、init時の失敗パスはまた別の設計判断が必要（オフライン起動との兼ね合い）だから、今回のスコープ外と判断した。ただし次回提言すべき。

## 技術メモ

- Biomeのconst強制: クロージャ内で再代入されるミュータブルな変数にも`const`を強制してくる。`{ current: null }`のrefパターンで回避。
- HttpOnly cookie: `document.cookie`からは読めない。テスト時にcookieを無効化したい場合はバックエンド側を変えるしかない。
- monkey-patch fetch: `window.fetch`を上書きすればcustomFetch内部のfetch呼び出しも差し替わる。ただしページリロードで消える。

# 2026-03-22 深夜: モバイル同期の信頼性修正

## やったこと

### sync-engine のsyncAll per-step try-catch化
- `syncAll`が単一try-catchで7ステップを囲んでいた → 1つの失敗が全ステップをブロック + exponential backoff発動（最大5分間隔）
- 各ステップを個別`tryStep()`に分離。依存関係のあるもの（activities→icons, goals→freezePeriods）のみ条件付きスキップ
- backoffは「全ステップ失敗時のみ」に変更。1つでも成功すればリセット

### sync順序修正: Tasks を ActivityLogs の前に
- バックエンドがactivityLogのtaskIdをFK検証 → Tasksが未syncだとskip → failed → 永久に未同期
- Tasksを先にsyncすることでFK問題を解消

### "failed"アイテムのリトライ
- `getPendingSync*`が`"pending"`のみ返していた → `"failed"`も含めるように変更
- 6ファイル（shared 4 + platform-specific 2）を修正

### Mobile欠落syncトリガー追加
- Activity作成/更新/削除、FreezePeriod全操作にsyncトリガーがなかった（Webにはある）
- 3ファイルに追加

## 感想

調査で並列エージェント3つ使ったのは正解だった。sync-engineのコードはpackages/sync-engine、apps/mobile/src/sync、apps/backend/feature-syncと3箇所に散らばっていて、逐次で読んでいたら倍以上かかっていたはず。

「Mobileにはsyncトリガーがないのでは？」と最初は思ったが、掘ってみるとnavigation syncもshared hooks経由のpost-mutation syncも既にあった。ユーザーも「Webだとsyncが走るけど」と聞いてきたので一瞬「トリガー追加が本丸か」と思いかけたが、冷静にコードを読んで「トリガーは既にある、問題はsyncAll自体の信頼性」と切り分けられたのはよかった。

backoffのリセット条件を「全ステップ成功」から「1つでも成功」に変えたのは少し楽観的かもしれない。iconアップロードだけが壊れ続けていても30秒ごとに無駄な試行を繰り返す。ただ元の「1つ失敗で全体5分停止」よりは確実にマシ。
