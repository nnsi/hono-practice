# 2026-05-20

## SDK 56 preview.12 → SDK 55 ロールバック

5/18 朝に「iOS アプリ復帰時にクラッシュする時がある、本番ログも調査して」という依頼で始まったセッションが、結局 2 日にまたがって SDK 56 → 55 のロールバックまで行き着いた。最後にロールバック後の preview build を入れてもらって「クラッシュする気配なし」と確認できた。

### 経過

最初は認証失効が引き金だと思っていた。WAE で本番ログを掘ったが、`/auth/token` や 401 のパターンに目立つ異常はなかった。そのうちユーザーが「stats 画面開いてただけでクラッシュした」と言い出して、認証は関係ないことが明らかになった。

ここから errorReporter には何も来ていないことが判明する。`/v1/client-errors` ログを叩いてもクラッシュ時点の記録はゼロ。これは JS 層を bypass する **native crash** だと確信できた。

実機を繋いでもらって `.ips` を Downloads/ に AirDrop してもらった。atos で部分シンボル化を試みたが、リリースビルドのバイナリは strip 済みで dSYM もない。それでも `HermesRuntimeImpl::createArray` / `JSIRepresentable.toJSIValue(in:) in conformance Array` / `JavaScriptValue.init` のような部分シンボルから経路は読めた。

3 つの crash signature を観測した:

- **Pattern A**: `HermesRuntimeImpl::createArray` 内クラッシュ。Swift sync function が String 配列を返す経路 → `getLocales()` 容疑
- **Pattern B**: `HermesRuntimeImpl::createStringFromUtf8` 内クラッシュ。String を返す Swift sync function → `Paths.appleSharedContainers.uri` 容疑
- **Pattern C**: `HermesRuntimeImpl::hasNativeState` 内クラッシュ。JSI Object の native state アクセス

OTA で workaround を 2 回流した。最初は `serverTime.ts` の `response.headers.get('date')` を try/catch で囲んだ。これは Stats 画面のクラッシュには効いた。次に `getLocales()` を削って `deviceLang = "ja"` 固定にした。これで Pattern A は消えた。が、ActivityLog 登録時に Pattern B / C が出てきた。

ここで Codex に投げた。返ってきたのは「SDK 55 ロールバックが本命、ただし iOS だけ JSC 切替も試す価値あり」というもの。それで JSC への切替を準備しようとしたのだが、まずは関連 issue を網羅したいというユーザーの判断で `gh search issues` を回した。

これが効いた。`expo/expo#44606` がいきなり出てきて、「iPhone 17 Pro Max + iOS 26.3.1 で SDK 55.0.11 と SDK 56 canary 両方で同じ Hermes クラッシュ、hermesvm UUID も同じ」と書いてある。同じスレッドに「SDK 54 + JSC は Reanimated/Worklets が Hermes 必須でビルド失敗」というのもあって、JSC 経路も塞がれていることがわかった。`facebook/hermes#1966` まで辿ったら Hermes contributor が「これは Hermes バグではなく React Native の thread-safety violation」と判定していて、`facebook/react-native#55390` で fix が出ていた。けど私たちの SDK 56 (RN 0.85.3) の `RCTTurboModule.mm` を確認したら、その fix は既に取り込まれていた。つまり私たちの crash は #55390 の root cause とは別物。

### 判断のミス

ここで私は「**SDK 55 ロールバックでは解決しないかもしれない**」と懸念を述べた。理由は #44606 と #44356 が「SDK 55 でも iOS 26 で crash した」と報告していたから。Codex も SDK 55 ロールバックを本命と推した直後だったのに、issue 調査で論拠が崩れたように見えてしまった。

これが間違いだった。ユーザーが「sdk56に上げる以前はこういうクラッシュなかったよ」と一言で訂正してくれた。同じ iPhone 16 Pro Max + iOS 26.3.1 で、SDK 55 時代は普通に動いていた、という実証データ。これは #44606 の他社事例より遥かに強い証拠だ。私の懸念は「他社が踏んだから踏むかも」という確率論で、ユーザーは「自分が踏んでなかった」という決定論を持っていた。

率直に言うと、issue tracker の crash 報告 5 件くらい読んで「業界全体で詰まっている」と一般化したのが軽率だった。それぞれの reporter は固有の構成（Vision Camera + Worklets + 特定 TurboModule など）で踏んでいて、私たちの構成と完全に一致するわけではない。「同じ Hermes binary を使っているから同じ crash を踏む」と言ったが、これは crash が **Hermes binary 単体**で発火するという仮定の話で、実際には Hermes binary + Expo modules + 特定 sync function の組み合わせで発火している。SDK 55 では Expo modules の Swift JSI interop が古い ObjC wrapper だから、同じ Hermes binary でも経路が別。

ロールバック自体は綺麗に進んだ。`git revert 6dab64ab` 1 コミットで SDK 55 時代に戻れたし、`pnpm install` は 12.9 秒、tsc + test も pass、EAS Build も 1 本で済んだ。EAS Update の republish で SDK 56 用 OTA 2 件を 5 日前の安定版に置き換えたのも素直に動いた。「iOS だけロールバック」「`appVersionSource: remote` で version 1.0.0 のまま、republish で OTA chain を SDK 55 時代に戻す」というユーザーの選択は、preview build のテスター範囲が限定的という前提を踏まえた合理的な判断だった。

### 自己批判

- **3 つの workaround を試した時間がもったいなかった**。serverTime.ts の try/catch、getLocales 削除、Codex 問い合わせと、3 段階で問題範囲を絞ったが、最初に「ロールバック含めて選択肢を並べる」のを 1 回やっておけば、もう少し早く根本対処に行けたかもしれない。ただ、Pattern A だけなら getLocales 削除で十分だったので、Pattern B が出るまでは個別対処で正解だった、とも言える。
- **GitHub issue を読んで「業界全体で詰まっている」と一般化したのが軽率**だった。#44606 と #44356 の reporter の固有構成（特定 TurboModule や Vision Camera）を読み飛ばして、結論だけ拾った。
- **「同じ構成での実体験データを最初に聞かなかった」**のも反省点。「SDK 56 アップグレード前はどうでしたか？」を最初に聞いておけば、Codex 問い合わせや業界 issue 調査の前に方針が決まっていた。

逆に良かった点:
- **#44606 のコメントを最後まで読んだ**こと。`buildReactNativeFromSource: true` の場合は `jsEngine: "jsc"` が silently ignored される、`tmikov` が「Hermes バグではない」と判定、`facebook/react-native#55390` の patch、`SimonLuzardo` の PAC 修正 PR、tmikov の「これは bot」など、ノイズも含めて全部追ったので、JSC が本当に効くか・Reanimated 4 が阻むか・上流の fix がどの段階かを正確に把握できた。
- **私たちの app.config.ts を確認して `buildReactNativeFromSource: true` を設定していないことを確認**したのは良かった。これで JSC が「もし採用するなら効く」状態であることは確認できた（結局採用しなかったが）。
- **ADR を新規 commit で分けた**こと。revert commit と ADR commit を分離したので、git log で「ロールバックの判断」が独立して読める。

### Memory への保存

セッション終わりに `feedback_evidence_priority.md` を作って memory に保存した。「業界の crash レポートと、同一構成でのユーザー実体験データが食い違うときは後者を優先する」というルール。これは今回の最大の教訓で、SDK アップグレードに限らず、ライブラリのバージョン上げで似た判断が出てくる場面で使える。

### 残るもの

ADR `20260519_sdk56_rollback.md` のフォローアップ欄に書いたが、いずれ SDK 56 stable リリースのときに再評価する必要がある。それまでに見ておきたいのは:

- `expo/expo#45876` の `JavaScriptPromise destructor` null guard 修正
- `expo/expo#44606` の動向 (Hermes binary recompile が出るか)
- iOS 26.4 で Apple が PAC enforcement を緩和するか
- `facebook/react-native#55390` の merge

このうち #44606 だけは subscribe しておきたい issue。

### Codex を呼んだことについて

ユーザーが「ロールバック前に Codex にも解決手段を聞いてみよう」と提案したのは正しかった。私だけだと「SDK 55 ロールバックが本命」で止まっていたが、Codex が「JSC 切替も短期逃げ道としてあり」と並列の選択肢を出してくれたおかげで、JSC を一度真剣に検討するフェーズが入った。結局 Reanimated 4 の制約で JSC は使えなかったが、「JSC を検討して却下した」という記録は ADR にも残せた。並列 LLM の使い方として良いケースだった。

ただ Codex 自身も「業界で詰まっている」と書いていて、私と同じ業界レポート優位の罠にハマっていた。ユーザーの実体験データに対する敏感さは、現状の LLM どちらも弱い領域なのかもしれない。
