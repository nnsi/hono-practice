これが今日の最大のミス。product-vision.md を読んで、config schema の `labels: [string, string]` を見て、「2ボタンにラベルを付ける設計」と解釈した。しかし product-vision.md にはバイナリモードの説明として「種類ボタン」「即時記録」と書いてあったはず。ActivityKinds という既存のドメイン概念を活用するのが自然な設計であるのに、config のデータ構造に引っ張られて見落とした。

## CounterMode 改修 + テスト基盤の根本修正

### やったこと

#### CounterMode のワンタッチ記録化

旧 CounterMode は「+N/-N ボタンでカウントを溜めて → メモ入力 → 保存ボタン」の3ステップだった。これを「+N ボタンのワンタッチで即記録」に変更。

変更内容:
1. **`createUseCounterMode.ts`** — ViewModel を大幅改修。`recordStep(step)` が即座に `onSave` を呼ぶ。`count`, `increment`, `decrement`, `reset`, 旧 `submit` を全削除。タブ切替（`activeTab: "counter" | "manual"`）、当日合計（`todayTotal`）、手動入力フォーム（`quantity`, `memo`, `submitManual`）を追加
2. **`CounterMode.tsx`（web/mobile）** — TimerMode のタブパターンを踏襲。カウンタータブ（ステップボタン横並び + 当日合計）と手動入力タブ（数量 + メモ + 保存ボタン）
3. **テスト全面書き換え** — recordStep のワンタッチ記録、todayTotal 算出、submitManual、steps パース、タブ初期状態

#### テスト基盤: mock useState の全廃

ここが今日の後半の本題。CounterMode テストで `submitManual`（状態変更後に submit）のテストが落ちた。原因は手製の mock `useState` がクロージャの state を正しく追跡できないこと。

最初の対応は場当たり的だった:
1. **1回目**: re-invoke パターン → 失敗
2. **2回目**: persistent state + call-index tracking → 動いたが複雑
3. **3回目**: 共有 `testHelpers.ts` に `createMockUseState` を抽出 → 4ファイル統一

ここでユーザーから「場当たり的な対応をやめて本質的に直すべき issue に集中して」と言われた。

さらに「そもそも useState の mock いる？react を install するのは悪手？」という核心的な指摘。

**答え: mock 不要。** `react` は既に frontend-shared の dependencies にあり、`@testing-library/react` の `renderHook` を使えば本物の React で hook テストができる。DI パターンは本番コード（web と mobile で React の import 元が違う）のためであって、テストでは普通に React を使えばいい。

最終的な修正:
- `testHelpers.ts` を削除
- 4テストファイル全てを `renderHook` + 本物の `useState` + `act()` に移行
- `// @vitest-environment jsdom` でテスト環境を指定
- numpad テストで「mock の制約でテストできない」と逃げていた部分が正しく動くようになった（+3テスト）

CI: 1035 tests, tsc clean, biome clean。

### 反省

#### 3回も場当たり的に mock を直した

これが今日一番恥ずかしいところ。mock useState が壊れたとき、「mock の実装を直す」という方向で3回もイテレーションした。ユーザーに言われるまで「mock 自体が不要では？」という問いを立てられなかった。

「動かないコードを直す」と「そもそもそのコードが必要か？」は全く違うレベルの問い。前者は実装の問題、後者は設計の問題。設計レベルの問いを先に考えるべきだった。

ユーザーの「場当たり的な対応をやめて」という指摘は、3回目の「共有ユーティリティに抽出」にも向けられたものだと思う。抽出は改善ではあったが、根本的にはまだ「mock を使う」前提から抜け出ていなかった。

#### DI パターンの「テスト」の意味を取り違えていた

factory パターン `createUseXxxMode({ react: { useState } })` は DI のためにある。テストでも DI 経由で mock を注入するのが「正しい使い方」だと思い込んでいた。

しかしテストの目的は「hook の振る舞いを検証すること」であって「DI の仕組みを使うこと」ではない。`renderHook` で本物の React を使えば、DI を経由しなくても（厳密には `createUseXxxMode({ react: { useState } })` に本物の React を渡して生成した hook を `renderHook` で呼ぶ）、hook の振る舞いは正しくテストできる。

手段と目的を混同した典型例。
