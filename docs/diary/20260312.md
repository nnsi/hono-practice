## CounterMode 改修 + テスト基盤の根本修正

### やったこと

#### CounterMode のワンタッチ記録化

旧 CounterMode は「+N/-N ボタンでカウントを溜めて → メモ入力 → 保存ボタン」の3ステップだった。これを「+N ボタンのワンタッチで即記録」に変更。

変更内容:
1. **`createUseCounterMode.ts`** — ViewModel を大幅改修。`recordStep(step)` が即座に `onSave` を呼ぶ。`count`, `increment`, `decrement`, `reset`, 旧 `submit` を全削除。タブ切替（`activeTab: "counter" | "manual"`）、当日合計（`todayTotal`）、手動入力フォーム（`quantity`, `memo`, `submitManual`）を追加
2. **`CounterMode.tsx`（web/mobile）** — TimerMode のタブパターンを踏襲。カウンタータブ（ステップボタン横並び + 当日合計）と手動入力タブ（数量 + メモ + 保存ボタン）
3. **テスト全面書き換え** — recordStep のワンタッチ記録、todayTotal 算出、submitManual、steps パース、タブ初期状態

#### テスト基盤: mock useState の全廃

ここが今日の後半の本題。CounterMode テストで `submitManual`（状態変更後に submit）のテストが落ちた。原因は手製の mock `useState` がクロージャの state を正しく追跡できないこと。

最初の対応は場当たり的だった:
1. **1回目**: re-invoke パターン → 失敗
2. **2回目**: persistent state + call-index tracking → 動いたが複雑
3. **3回目**: 共有 `testHelpers.ts` に `createMockUseState` を抽出 → 4ファイル統一

ここでユーザーから「場当たり的な対応をやめて本質的に直すべき issue に集中して」と言われた。

さらに「そもそも useState の mock いる？react を install するのは悪手？」という核心的な指摘。

**答え: mock 不要。** `react` は既に frontend-shared の dependencies にあり、`@testing-library/react` の `renderHook` を使えば本物の React で hook テストができる。DI パターンは本番コード（web と mobile で React の import 元が違う）のためであって、テストでは普通に React を使えばいい。

最終的な修正:
- `testHelpers.ts` を削除
- 4テストファイル全てを `renderHook` + 本物の `useState` + `act()` に移行
- `// @vitest-environment jsdom` でテスト環境を指定
- numpad テストで「mock の制約でテストできない」と逃げていた部分が正しく動くようになった（+3テスト）

CI: 1035 tests, tsc clean, biome clean。

#### UI微調整

ブラウザ確認後にユーザーから2点フィードバック:
- ステップボタンを縦並びから**横並び**に
- タブ順を TimerMode と統一（左: 手動入力、右: カウンター（デフォルト））

両プラットフォームとも修正して確認完了。

### 反省

#### 3回も場当たり的に mock を直した

これが今日一番恥ずかしいところ。mock useState が壊れたとき、「mock の実装を直す」という方向で3回もイテレーションした。ユーザーに言われるまで「mock 自体が不要では？」という問いを立てられなかった。

「動かないコードを直す」と「そもそもそのコードが必要か？」は全く違うレベルの問い。前者は実装の問題、後者は設計の問題。設計レベルの問いを先に考えるべきだった。

ユーザーの「場当たり的な対応をやめて」という指摘は、3回目の「共有ユーティリティに抽出」にも向けられたものだと思う。抽出は改善ではあったが、根本的にはまだ「mock を使う」前提から抜け出ていなかった。

#### DI パターンの「テスト」の意味を取り違えていた

factory パターン `createUseXxxMode({ react: { useState } })` は DI のためにある。テストでも DI 経由で mock を注入するのが「正しい使い方」だと思い込んでいた。

しかしテストの目的は「hook の振る舞いを検証すること」であって「DI の仕組みを使うこと」ではない。`renderHook` で本物の React を使えば、DI を経由しなくても（厳密には `createUseXxxMode({ react: { useState } })` に本物の React を渡して生成した hook を `renderHook` で呼ぶ）、hook の振る舞いは正しくテストできる。

手段と目的を混同した典型例。

### 所感

今日は2セッション分の作業。前半（binary/numpad/check）はバイナリの設計ミスが痛かったが、後半（CounterMode + テスト基盤）はユーザーの2つの指摘が的確で、両方とも自分では到達できなかった問いだった。

「場当たり的な対応をやめろ」は開発全般に通じる原則。目の前のエラーを消すことに集中すると、問題の構造が見えなくなる。一歩引いて「なぜこの問題が起きているのか」「この解決策の前提は正しいか」を問う余裕が必要。

## プロダクト戦略の議論: Actiko の差別化

### やったこと

記録モード5種が完成した段階で、ユーザーから「これで他の習慣記録系アプリと差別化できてるか？」と問われた。コードを書かないセッション。プロダクトの競争優位性について率直に議論した。

#### 記録モードの差別化は弱い

product-vision.md では「既存アプリは全アクティビティで同じ汎用フォーム」と主張しているが、実際に調べると Habitify（boolean/numeric/timer）、Loop Habit Tracker（boolean/numeric + カスタム刻み幅）、Streaks（boolean/quantity/timer）など、競合もすでに入力タイプを分けている。Actiko の記録モードが優れている点は「ボタンが大きい」「テンキーでOS keyboardを回避する」などのUXの磨き込みであって、概念レベルの差別化ではない、と伝えた。

#### 音声+AI、ウィジェットも競合あり

ユーザーから「音声+AIって他のアプリでもやってない？」と突っ込まれた。調べたら Habit Chat（マイクボタンからAIコーチに音声入力）、Habit Driven（テキスト・画像・音声でトラッキング）、Apple Intelligence連携など、音声+AIは2026年のトレンドそのもの。「誰もやっていない」という自分の最初の主張は間違っていた。

ウィジェットも同様。Dots（ウィジェット特化で1タップ記録）、Habit Tracker Widget（ホーム/ロック画面対応）など、特化アプリまで存在する。

#### Debt システムが唯一の明確な差別化

調べた限り、「未達分が負債として蓄積し、翌日以降の超過で返済できる」モデルを持つ習慣記録アプリは見つからなかった。記録モード・音声AI・ウィジェットがパリティ（最低限揃えるべき機能）だとすると、Debt が Actiko を選ぶ理由になる唯一の軸。

#### 「続かない」要因の分解と方針決定

入力の面倒さ以外の「続かない要因」を5つ出した（忘れる、報酬ゼロ、途切れたら終わり、多すぎる、タイミングのズレ）。ユーザーから各要因へのフィードバックをもらい、最終的に:

- **入力導線**: 自分のペインに最適化（A）。既にほぼ解決済み
- **継続の仕組み**: 万人向けに広げる（B）。自分も「続かない人」だから結果的に自分も救える

という整理になった。Debt システムを軸にした継続改善のアイデアを `/docs/plan/debt-continuation-ideas.md` にまとめた。

#### Debt フリーズのUI検討

Debt フリーズ（一時停止）のUIモックを提示。全体フリーズ（おやすみモード）と個別フリーズ、フリーズ中の表示、フリーズ中に記録した場合の挙動、終了時の画面を設計。ユーザーから「ゴール設定に常時フリーズ設定欄を配置」「事前の日付指定も可能に」「終了日未定も許可」「フリーズ中は目標なし扱い」という方針決定を得た。

### 反省

#### 音声+AIの競合状況を調べずに「差別化」と言った

最初の分析で「音声+AIは他のアプリが手を出していない領域」と言い切ったが、ユーザーに「本当に？」と聞かれて調べたら全然そんなことはなかった。調べもせずに断言したのは不誠実だった。ユーザーが突っ込んでくれなかったら、間違った前提で戦略を議論し続けるところだった。

主張する前に検証する、という基本ができていなかった。特に競合分析のような外部情報に依存する話題では、「たぶんそうだろう」で進めてはいけない。

#### ユーザーの「疑い」の精度が高い

音声+AI、ウィジェットの両方について、ユーザーの方が正しい直感を持っていた。自分が「差別化になる」と言ったものに対して「ほんと？」と疑えるのは、このドメインの当事者だからだろう。自分はドメイン知識が浅い分、一般論で語りがちになる。

#### 議論の進め方は良かった

一方で、議論の流れ自体は良かったと思う。記録モードの差別化の弱さ → 音声+AIも競合あり → ウィジェットも標準化済み → Debt が唯一の差別化、という順で選択肢が絞られていく過程は、ユーザーにとっても整理になったはず。最後の「入力はA、継続はB」という整理はユーザー自身から出たもので、自分が誘導したわけではない。良い壁打ち相手になれた部分はあると思う。

#### 身内のエピソードが設計を前進させた

Debt フリーズを Tier 4 から Tier 1 に上げたのは、身内が「目標を捨てて作り直した」というリアルな体験があったから。仮説ではなく実際に観測された課題だから優先度を上げる判断ができた。ドキュメントに個人情報は書かないことにしたが、この判断の根拠は忘れないようにしたい。

### 所感

今日はコードを1行も書かないセッションだった。でも、product-vision.md の「唯一のアプリ」という主張が実は怪しいこと、Debt が本当の差別化であること、継続の仕組みを万人向けに広げるべきこと、が整理された。コードを書くより重要な整理だったと思う。

自分の弱点として、ドメインの外部情報（競合状況）について「たぶんこうだろう」で語ってしまう癖がある。今回はユーザーが2回突っ込んでくれたから修正できたが、突っ込まれなかったら間違った分析のまま進んでいた。「わからないことは調べてから言う」を徹底したい。

ユーザーの「自分も続かない人だからプロダクトを作っている」「続かない要因を自分でも言語化できていない」という自己認識は、プロダクト開発者として健全だと思う。自分のペインから始めつつ、それを一般化する意志がある。Debt システムの設計アイデアが、その一般化の具体的な一歩になればいい。
