## セッション3: app.json → app.config.ts 環境変数化

### やったこと

- ユーザーから「app.jsonのbundleIdentifierがべた書きされている、公開リポジトリに載せたくない」という相談
- 3つの選択肢を提示（A: app.config.ts + env / B: gitignore + example / C: そのまま）。`bundleIdentifier`は秘密情報ではない事実を正直に伝えた上で、Aを推奨 → ユーザーがAを選択

### `source .env` の罠

最初 `source .env` で変数をセットして `eas` コマンドを実行したが `EAS project not configured` エラー。原因は `source` だけだと変数がシェル変数にはなるが `export` されないため、子プロセスの `eas-cli` に渡らなかったこと。`set -a && source .env && set +a` に修正して解決。

ユーザーの「PowerShell環境で動くかな？」という一言がきっかけで、「じゃあ実際にOTA配信してみて」という流れになり発覚した。実際にはPowerShellの問題ではなくbashの `source` の挙動の問題だったが、ユーザーが試してみてと言わなければ手順書だけ更新して終わっていた。

### 振り返り

#### 良かった点
- `bundleIdentifier`は秘密情報ではないと率直に伝えた上で選択肢を提示できた。「ユーザーがやりたいと言っているから黙って従う」ではなく、事実を示した上で判断を委ねるという対話ができた
- OTA配信で実動作を検証できた。型チェック通過だけでは見つからない問題（`source` が `export` しない）を実行で潰せた

#### 反省点
- `.env` ファイルを作る時点で `source` でのエクスポート問題を考慮すべきだった。dotenvの慣習（`KEY=value` 形式）に従ったが、シェルスクリプトとして `source` されることを想定するなら `export KEY=value` 形式にするか、手順に `set -a` を最初から書くべきだった。自分で `.env` を作り、自分で `source .env` と手順を書いたのに、その組み合わせで動かないことに気づかなかった
- archived ドキュメントのコードサンプルを実在しない定数に書き換えてしまった。archiveは過去の設計案の記録なので、`<BUNDLE_ID>` プレースホルダーにすべきだった。歴史的ドキュメントを「修正」するのは記録の改変に近い。ユーザーに「全部削除しておいて」と言われた勢いで、ドキュメントの性質を考えずに機械的に置換した

## セッション5: モバイル日付フィールドにカレンダー選択UI追加

### やったこと

#### OverlayPortalのスタック対応
- Task/Goalダイアログの日付フィールドをカレンダー選択可能にする際、根本的な問題に当たった: `OverlayPortal`がシングルスロット設計で、ModalOverlay（ダイアログ）の上にCalendarPopover（カレンダーモーダル）を重ねられない
- 最初はCalendarPopoverの代わりにインライン展開するCalendarInlineコンポーネントを作って回避したが、ユーザーから「Webみたいにモーダルが重なる形にならない？」とフィードバック
- OverlayPortalをスタック対応に改修: `useId()`で各ポータルを識別し、配列で管理して順番に重ねて描画する方式に変更。これで複数のOverlayPortalが共存可能になった
- CalendarInlineは不要になったので削除

#### 日付フィールドのUI統一
- **DatePickerField（必須日付用）**: 矢印（< >）スタイル → カレンダーアイコン + タップでCalendarPopoverが開くスタイルに変更
- **OptionalDatePickerField（任意日付用）**: 新規作成。「未設定」表示 + カレンダー選択 + クリア（×）ボタン
- Task Create/Edit: `dueDate`をIMESafeTextInput → OptionalDatePickerFieldに変更
- Goal Create/Edit: `endDate`をIMESafeTextInput → OptionalDatePickerFieldに変更
- ユーザーの「フォーム統一して」というフィードバックで、開始日と期限/終了日のスタイルも統一

### 振り返り

#### 良かった点
- **OverlayPortalのスタック化は正しい根本対処だった**: インラインカレンダーはワークアラウンドであり、今後他の場所でもモーダル重ねが必要になったとき再び問題になる。ユーザーの「Webみたいにモーダルが出る形にならない？」という指摘が、OverlayPortalの設計改善を促してくれた。CLAUDE.mdの「ワークアラウンドより根本原因を潰す」にも合致する
- **ブラウザ確認で問題を即座に発見できた**: CalendarPopoverが最初に動かなかった（ModalOverlayを置き換えてしまった）問題は、コードを読むだけでは気づきにくかった。実際にクリックして確認したから30秒で問題を特定できた

#### 反省点
- **最初にOverlayPortalの設計を確認すべきだった**: CalendarPopoverがOverlayPortalを使うことはコードから明白だったのに、ModalOverlay内で使った場合の競合を事前に考慮しなかった。OverlayPortalのコメントに「Only one overlay is shown at a time (single-slot portal)」と書いてあるのだから、読んでいれば1ステップ目のインライン回避は不要だった
- **CalendarInlineを作ってから捨てた無駄**: OverlayPortalの制約に気づかず、まずインライン展開で実装 → ユーザーにモーダル方式を要望される → OverlayPortalを直す → CalendarInlineを削除、という二度手間になった。最初からOverlayPortalのコードを読んでいればストレートに解決できたはず
