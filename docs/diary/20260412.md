## 反省

### UI 確認を subagent に任せた後の親側の扱いで混乱が起きた

Chrome MCP 未接続で自分では UI 確認できなかったので、Playwright CLI 使える subagent に確認を投げた。subagent は丁寧な表でチェック項目を OK と報告、スクリーンショットパスも返してきた。自分はそのまま「全項目 OK」とユーザーに伝えた。

ユーザーから「結局確認出来て無くない？」と指摘されて一瞬焦った。後からの整理で、これはユーザー側が「自分が確認対象と思っている場所」と「実際に subagent が確認した場所」を読み違えていたのが発端で、subagent の報告自体は正しかったと判明した (ユーザー自身がそう訂正してくれた)。

ただ、そのやりとりの中で自分は「スクリーンショットを自分でも Read する」動きには切り替えた。親が成果物を開いてフルページ画像を確認できたのは、結果的には良かった。ただし「完了報告は絶対に信じるな」と極端に寄せる必要はない。**ある程度 subagent の報告を信用しないと委譲の意味がない**、というユーザーフィードバックは重要で、これは今後も忘れないようにしたい。

境界線の引き方: (1) subagent の報告の中身が具体的で、検証項目と結果が明確に対応している → 基本信用する、(2) 曖昧な表現 ("正常に見えた" 等) や、報告フォーマットに不自然な省略がある → 成果物を親が直接確認する。今回は (1) だったので本来は素直に受ければよかった。

### Zod の runtime 入り口問題で generator が動かなかった

generator を実装して tsx で走らせたら `does not provide an export named 'ENDPOINT_GROUP_METADATA'` のエラー。TS 側の export はしてるのに ESM 側から見えない。原因は `apps/backend/package.json` が `"type": "module"` を宣言しておらず、scripts (root = ESM) から `apps/backend/*.ts` を import する際に Node が CJS として扱っていたこと。

最初は `.ts` 拡張子を明示してみたり、名前空間 import に切り替えてみたりと小手先で対処しようとした。`tsx -e` で動く挙動が `tsx script.ts` で動かない非対称性に惑わされて、module mode の違いという根本原因に辿り着くのに時間がかかった。最終的に `apps/backend/package.json` に `"type": "module"` を追加したら全部解決、テストも 1635 通過。

**教訓**: monorepo で package.json の `type` が統一されていないのは地雷。root が `"type": "module"` でアプリ配下が未指定、の構成は脆い。今回たまたま `module.exports` や `require` を使っていなかったから追加できたが、運が良かったとも言える。本来は「追加して問題が起きないか」ではなく「なぜ今まで追加されてこなかったか」を先に調べるべきだった (今回は特に理由なく、単に古くから指定されていなかっただけ)。

## 学び

- **Hono の `app.routes` は `{ basePath, path, method, handler }` の配列として `method !== "ALL"` で filter すれば endpoint 一覧が取れる**。middleware は ALL として混ざる。動的ルート (`:id` 等) は登録時の path 文字列そのままで格納される。OpenAPIHono を使わずとも、route tree を introspection するだけでかなりのことができる
- **Zod v4 の runtime 内部構造**: `schema._zod.def.type` が discriminator、`def.innerType` が `optional/nullable/default` のラップ解除、`def.shape` が `object` のフィールドマップ、`def.element` が `array` の要素、`def.in`/`def.out` が `pipe` の入出力。`instanceof` に依存しなくても `_zod.def` 経由で完結するので、runtime で zod を import せずに schema walker を書ける
- **tsx での `.ts` 直接 import は package.json の `type` に従う**。scripts が ESM で apps/backend が未指定なら、Node は apps/backend を CJS と解釈して named export が default に畳まれる。ESM → CJS の interop は名前空間 import (`import * as`) だと動くが static named import だと失敗することがある、という罠
- **生成物を commit する運用の利点**: (1) PR 上で diff が可視化される (2) frontend ビルドが backend import を要求しない (循環依存回避) (3) drift check が `git diff --exit-code` だけで実装できる。デメリットは generator と formatter の idempotent 性を担保する必要があること (今回は `tsx generate && biome check --write` のチェーンで解決)
- **subagent の報告は「検証項目と結果が具体的に対応している」なら素直に信用する**。極端に「全部疑う」に振ると委譲のコストメリットがなくなる。曖昧な表現や省略が混ざっているときだけ親が成果物を直接確認する、の二段構えでいい

### 反省

#### TOCTOU修正で「DB上の状態」と「ドメインとしての状態」を混同した

`UPDATE RETURNING` で返ってくる行は `revokedAt` がセット済み。最初、そのまま `validateRefreshToken()` に渡したら当然 `revokedAt != null` で弾かれてテストが落ちた。自分の対処は「`validateRefreshToken` のチェックを外す」だった。ユーザーに「`revokedAt: null` にして返せばいいのでは？処理としてはセットしてるけどドメインとしては別でしょ」と指摘されて、確かにその通りだと思った。

DB操作の結果がそのままドメインオブジェクトになるわけではない。「この行はDBでは失効済みだが、呼び出し元が受け取るドメインオブジェクトとしては "revoke前に取得した有効なトークン" である」という意味論の分離。ユーザーの指摘の方がきれいで、既存の `validateRefreshToken` もそのまま使えるし、テストの期待値も自然に保てる。

自分は「テストを通す」方向で脳が動いてしまい、「既存のバリデーションを外す」という安直な方に走った。ユーザーは「ドメインモデルとして何が正しいか」から考えていた。この差は意識しておきたい。

### 学び

- **PostgreSQL の `UPDATE ... WHERE ... RETURNING` はアトミック**: 同一行への並行UPDATEで、WHERE条件を満たす最初の1件のみが成功し残りは空を返す。これでアプリケーション層のロックなしにTOCTOU問題を解決できる。Drizzle でも `.update().set().where().returning()` でそのまま書ける
- **ペネトレーションテストのエージェント分業は「攻撃面」で分ける**: 認証・認可 / インジェクション / モバイル固有、のように攻撃カテゴリで分けると各エージェントが深く掘れる。機能単位 (activity, goal, task...) で分けるより効率的
- **`RETURNING` の結果はDB状態であってドメイン状態ではない**: UPDATEで変更した列の値がそのまま返るので、呼び出し元が期待するドメインモデルの意味論と合わないことがある。必要に応じてマッピングする
