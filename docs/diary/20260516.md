## 自己評価のいい部分

### useBottomTabBarHeight() の罠を実機で踏んで素早く切り替えた

paddingBottom 対応で、最初は `useBottomTabBarHeight()` を `(tabs)/settings.tsx` から prop で渡す形にした。型チェックは通る、unit test も通る、iOS upgrade flow も通った。**ところが iOS sync-offline flow を rerun したら app launch 直後に app crash**。前セッションでは sync-offline 通っていたのが、今セッションの変更だけが原因で死んだ。

スクリーンショット見たら splash 画面で app switcher card のみ。これは React Hook が throw して ErrorBoundary すら起きてない症状。`useBottomTabBarHeight` の実装を頭で再現したら「`BottomTabBarHeightContext` が undefined だと `throw new Error()` する」とわかった。CustomTabBar は `tabBar={(props) => <CustomTabBar {...props} />}` で props を受けるだけで height を navigation に通知していない。

→ 固定値 (`CUSTOM_TAB_BAR_CONTENT_HEIGHT=68 + insets.bottom`) に切り替えて SettingsPage 内で `useSafeAreaInsets` のみ使う形に修正。リビルドして再実行で通った。

**Hook が「context 必須だが暗黙に依存する」設計は本来かなり危険**。useBottomTabBarHeight は CustomTabBar との互換性が明示されていない (Tabs ナビゲータの default tab bar 前提)。次に類似 hook 使う時は、ソース読んで「どこで Context.Provider が貼られているか」を先に確認する。

### /cross-review の prompt ファイルが古いまま残って Codex が refresh token rotation の話を始めた

Round 2 の Codex 起動で、`/tmp/prompt-cross-review-r2.txt` を Write しようとしたが PostToolUse hook で "File has not been read yet" と stop された。なのにファイルは存在していて、`cat` で見ると **前回 (5/14) の review-cycle で書いた refresh token rotation の prompt がそのまま残っていた**。

それに気づかず Codex に渡し、Codex は素直にその古い prompt を解析して **「rotation consume と successor token 発行が transaction 境界で分断」みたいな Round 2 結果を出力**。私は最初「Codex がキャッシュで誤動作?」と思ったが、`grep -c "devRoute" /tmp/prompt-*.txt` で 0 件と判明。ファイル内容が完全に別物だった。

教訓:
1. **`/tmp/` の prompt ファイル名にセッション ID or timestamp を含める**。固定ファイル名だと前セッションのキャッシュと混同する
2. **Codex 起動前に `head -5` で内容確認**。一度ハマったら必ずやる癖をつける
3. **Write が "File has not been read yet" で止まった時はファイルの状態を疑う**。Read 必須 hook で stop された場合、ファイルが古い内容で残っている可能性大

最終的には Bash の heredoc で `/tmp/cross-review-r2-devroute.txt` に書き直して回避。確実性は heredoc が一番高い。

## 学び / 次回への持ち越し

### 大きな学び: Hook が context 必須なら、必ず Context.Provider の出処を最初に確認する

useBottomTabBarHeight の事故は「型は通るのに実行時 throw」という最悪のパターン。React Hook 一般に言えるが、特に React Navigation 系は Context.Provider の貼られ方が tab bar 実装によって変わるので危険。

ルール案 (`apps/mobile/CLAUDE.md` 候補):
- `@react-navigation/*` の `useXxxHeight` / `useNavigationState` 等の hook を使う前に、その Context.Provider が現在の navigator (custom tab bar 含む) で正しく動作するか確認する
- 不明な場合は固定値計算でフォールバック

### EAS local build × Maestro は並列禁止 (5/14 同様、再発)

これは **2 セッション連続で踏んだ**。明確にルール化すべき。

提案: `apps/mobile/CLAUDE.md` または `.claude/rules/` に新規ファイル:
- **EAS local build (iOS / Android) と Maestro 実行は並列禁止**
- 理由: build が CPU を占有して Maestro 操作 (XCUITest / Android emulator) が応答せず、UI 自体が ANR (Android) や splash hang (iOS) になる
- 例外: build が IO 待ち phase (Pods install / Gradle download) なら並列可、ただし判定は手間なので原則直列

### `/cross-review` の prompt 渡しに `--prompt-file` オプションが欲しい

`cat /tmp/prompt.txt | codex exec -` というパイプ方式は、ファイル名衝突 / Write hook stop / 内容確認漏れの三重リスクがある。Codex に prompt を渡す確実な方法を整理する:

選択肢:
1. **Bash heredoc で書く**: 確実、ただし長文はインデント崩れリスク
2. **ファイル名にセッション ID を含める**: 例 `/tmp/cross-review-${session_id}-round${N}.txt`
3. **Codex CLI が `--prompt-file` を提供**: そもそも CLI 側で fix されるべき

短期は (2)。次回 /cross-review 修正案として skill SKILL.md にファイル命名ルールを追記したい。

### `/write-diary` の存在を毎セッション意識する

ユーザーが「日記」「コミット」「プッシュ」「レビュー」のような **定型ワード** を投げてきたら、対応する skill が `.claude/skills/` に居る可能性を 100% 疑う。今回の `/write-diary` 見落としは、初動で `ls .claude/skills/` を確認していれば防げた。

session 開始時に `ls .claude/skills/` で skill 一覧を mental load しておく習慣を作る。CLAUDE.md にもこれを書いてもいいが、これは私の運用問題なので memory に書く方が妥当。
