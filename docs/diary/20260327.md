### pentest結果の鵜呑み

.envのCritical判定を「3者の報告だからそうだろう」と受け入れた。ユーザーが「.envはgitignore入ってるよね？stg/prodでどうやって見るの？」と突っ込んで初めて再評価した。**ペネトレーションテストの重大度判定は攻撃経路の現実性を含めて自分で評価すべき**。localhost環境でファイルシステムにアクセスできるのは前提であって脆弱性ではない。

2体同時にChrome MCP操作させたら両方詰まった。ユーザーが「片方Playwright使わせて」と提案。Chrome MCPは1セッション1体が安全。複数体でブラウザ操作する場合はPlaywrightとの併用が現実的。

### goalFreezePeriodの適用漏れ

fix-sinceエージェントが4ルート中1つだけ適用漏れ。テストファイルは作ったのにルートは未修正。エージェントのテスト報告は「自分が作ったテストが通った」であって「全テストが通った」ではない可能性がある。統合テスト（pnpm run test-once）を最後に自分で回す運用は正しい。

## 複数手法のセキュリティ監査で学んだこと

### 手法ごとに発見できる脆弱性が違う

- **Codexの静的分析でしか見つからなかった**: sync APIの所有権検証不足、serverWinsの漏洩、refresh tokenのrace condition。コードの構造を読まないと見えない

## セッション2: refresh token replay windowの検討

### やったこと

午前のペネトレーションで検出された refresh token replay window について、ユーザーと対策を検討した。

#### 3つの選択肢を提示

| 案 | 概要 | パフォーマンス影響 |
|----|------|-------------------|
| A. revokedAt grace period | 失効後10秒間はreplayを許可 | ゼロ |
| B. KV キャッシュ | 初回rotation結果をKVに保存、replay時はキャッシュ返却 | +1-2ms/req |
| C. rotation頻度を下げる | N日以上経過時のみrotate | ゼロ |

Aを推奨として提示した。

#### ユーザーとの議論で結論が変わった

ユーザーが核心を突いた:「トークン生成と失効が同時に行われるから、攻撃者がfetchとrotateの隙間に入って攻撃用トークンを作ることは出来ないのか」

これは正しい。現設計では:
1. `fetchRefreshToken` で旧トークンを検証
2. `rotateRefreshToken` で新トークン作成 + 旧トークン失効を `Promise.all` で同時実行
3. 攻撃者が隙間にreplayしても、その時点で旧トークンは失効される

つまり **grace period はセキュリティを下げるだけで、得るものがない**。多重リクエスト対策はクライアント側のmutexで解決すべき問題。

#### クライアント側を確認 → 既に対策済み

Web (`sync-engine/http/authenticatedFetch.ts`) と Mobile (`apps/mobile/src/utils/apiClient.ts`) の両方で `refreshPromise` を使ったmutexパターンが実装済み。テストカバレッジもあった。

**結論: サーバー側もクライアント側も対応不要。現状維持。**

### ドキュメント更新

`docs/knowledges/auth_flow.md` に「トークンローテーション」セクションを追加:
- セレクタパターンの仕組み
- ローテーションフロー（fire-and-forget含む）
- grace period を設けない判断理由
- クライアント側mutexの実装箇所

### 反省

**最初の提案が間違っていた。** grace period（案A）を推奨したが、ユーザーの一言で「grace periodは攻撃面を広げるだけ」と気づいた。Auth0やFirebaseが採用しているパターンだから正しいだろう、という権威への依存があった。

自分が見落としていたのは「現設計のPromise.allによるcreate+revokeの同時実行が、すでにタイミング攻撃を防いでいる」という点。既存コードの設計意図をもっと深く読むべきだった。ユーザーの方がコードの設計意図を正確に理解していた。

### 反省

#### SameSite=None を本番にも適用してしまった

最初のエージェントが`sameSite: "None"`を常時設定する修正をした。元のコードでは本番のみ`SameSite=None`だったのを、「全環境でNone」にしてしまった。ユーザーに「本番で弱くなるからNoneに戻してるの？」と突っ込まれて気づいた。

プロンプトに「localhostのみ」と書いたつもりだったが、エージェントへの指示が「シンプルに常にSameSite=Noneにすればいい」と書いてしまっていた。**セキュリティに関わる変更はエージェントに丸投げしない**、というCLAUDE.mdのルールがここでも効く。自分のプロンプトが間違っていた。

さらにその後の修正でも`secure: true`にしただけで`sameSite: "None"`を全環境に残してしまい、2回目の指摘を受けた。焦って修正を重ねると間違いが連鎖する。一度立ち止まって元のコードの意図を理解すべきだった。

#### Chrome MCPでのログイン操作に苦戦

Reactのcontrolled inputに値を設定するのに30分近くかかった。`form_input`ツールはDOMのvalueは設定するがReactのstateは更新しない。最終的にJSの`Object.getOwnPropertyDescriptor(HTMLInputElement.prototype, 'value').set`でネイティブsetterを呼ぶワークアラウンドで解決。これは今後のブラウザ確認で再利用できるパターン。

#### セッション2のSameSite=Laxを今回Noneに変えたが本番影響を考えていなかった

午前のセキュリティ監査で`SameSite=Lax`に修正したばかりなのに、午後のQAテストで`SameSite=None`に変更した。**同じ日の別セッションで矛盾する変更をしている**。会話コンテキストが切れると前のセッションの判断を忘れる。日記を読めば気づけたかもしれない。
