# 2026-03-17

## テスト実行の高速化 — silent: true → vmThreads → TRUNCATE最適化

### やったこと

テスト出力抑制（`silent: true`）から始まり、collect時間のボトルネック調査、pool方式の変更、test.setup.tsのDB操作最適化まで1セッションで実施。最終的にテスト全体の実行時間を53秒→15秒（-73%）に短縮した。

#### 変更内容

1. **`silent: true`追加** — console出力を抑制
2. **`.claude/worktrees/**`をexclude追加** — `.claude/worktrees/`内の古いワークツリーがテスト対象に含まれて43ファイル失敗していた問題を修正
3. **`vite.config.ts`のtest設定削除** — `vitest.config.ts`が存在する場合は`vite.config.ts`のtestブロックは完全に無視される。setupFiles、env変数、exclude等が全てデッドコードだった。grepで使用箇所ゼロを確認して削除
4. **`pool: "forks"` → `pool: "vmThreads"`** — vmThreadsはモジュールキャッシュをスレッド間で共有するため、PGlite/drizzle-ormの解決が1回で済む。collect 33秒→20秒、environment 11秒→2秒
5. **afterEachの`drop schema cascade` + `migrate` → `TRUNCATE CASCADE`** — 毎テストごとにスキーマを削除→再作成→マイグレーション実行していたのを、全テーブルTRUNCATEに置換。マイグレーション実行が`beforeAll`の1回だけになり、tests 128秒→33秒
6. **`bcrypt.hash()`を事前計算リテラルに置換** — seed()内で毎回呼ばれていた`bcrypt.hash("test-refresh-token-plain", 10)`を、計算済みのハッシュ文字列に置き換え

#### 試したが効果がなかったもの

- **`maxForks: 8`（4→8）**: 逆効果。collect 32→84秒に悪化。フォーク数が多すぎてメモリ/CPU競合が発生
- **`deps.optimizer.ssr.include`**: PGlite、drizzle-orm等を事前バンドル対象に指定。効果なし（collect 33→39秒で微悪化）
- **`server.deps.inline`**: ワークスペースパッケージをインライン化。効果なし。tsconfigのパスエイリアス経由でimportされているため、inlineパターンが実際のモジュールパスにマッチしなかった

### 計測結果

| 指標 | Before | After | 改善 |
|------|--------|-------|------|
| collect | 33.6s | 16.1s | -52% |
| tests | 128.0s | 33.3s | -74% |
| environment | 11.5s | 2.4s | -79% |
| **全体** | **53.5s** | **14.6s** | **-73%** |

### 技術的な学び

**vmThreadsが効く理由**: `pool: "forks"`は各forkが独立プロセスなのでモジュール解決を毎回やり直す。`pool: "vmThreads"`はNode.jsスレッドを使い、モジュールキャッシュを共有する。PGliteやdrizzle-ormのような重い依存が39テストファイルで使われるbackendテストでは、この差が顕著に出た。

**TRUNCATEが効く理由**: `drop schema cascade` → `create schema` → `migrate` は、PGliteのインメモリDBに対してDDLを全テーブル分再実行する。TRUNCATEはDDLに触れずデータだけ消すので桁違いに速い。マイグレーションがbeforeAllの1回で済むようになり、テスト数に比例していた固定コストが消えた。

**試行錯誤で分かったこと**: `deps.optimizer`や`server.deps.inline`がワークスペースパッケージに効かないのは、vitestがこれらをnpmパッケージとして扱うことを前提にしているため。tsconfigのパスエイリアスで解決されるモジュールは、vitestから見ると通常のファイルimportと同じ扱いになり、最適化対象にならない。

### worktree問題について

`.claude/worktrees/agent-a2b0c55f/`という別エージェントの残骸がリポジトリ内に残っていて、`**/apps/backend/**/*.test.ts`のようなglobパターンがこれにもマッチしていた。結果として全テストファイルが重複して拾われ、worktree側は環境が壊れているので43ファイル全失敗。excludeに追加して解決したが、worktreeのクリーンアップ漏れは今後も起きうる。`.claude/`全体をexcludeする方が安全かもしれない。

### 所感

ユーザーの「続けて」「1,3セットでやろう」というテンポの速い指示に乗って効率よく進められた。

自己批判としては、最初にmaxForks=8を試したのは安易だった。「並列度を上げれば速くなる」は思考停止で、ボトルネックがCPUなのかI/Oなのかモジュール解決なのかを先に切り分けるべきだった。結果的にbackend単独テスト（73秒のcollect）を先に計測してモジュール解決がボトルネックだと特定できたので、そこからvmThreadsに辿り着けた。

TRUNCATE最適化は提案した時点で効果に確信があった。毎テストでmigrateを走らせるのは明らかに過剰で、データだけリセットすればスキーマは変わらない。ただ、これを「前のセッションから気づいていたか」と言われると、test.setup.tsを読んだのは今回が初めてだった。テスト速度が問題として認識されていなかっただけで、もっと早く見ておくべきだったかもしれない。

4. **push sync のエラーが完全に握り潰されていた** — `createSyncEngine` の `syncAll()` の catch が `retryCount++` するだけでエラー内容を一切報告しない。`onSyncError` コールバックを追加

## OTA Update 原因特定 — 環境変数が焼き込まれていなかった

### やったこと

前セッションで未解決だった「OTA Update が効かない」問題の根本原因を特定し、解決した。合わせてローカル Android ビルド環境のセットアップを試みた。

#### Android ビルド環境セットアップ

ユーザーから「ローカルで Android ビルドできるようにしたい」と依頼。OTA Update の検証が目的で、`eas build --local --profile preview` を想定。

1. **winget で JDK 17 + Android Studio をインストール** — 最初は非管理者シェルで実行してインストーラーがハング。ユーザーが管理者権限の PowerShell で Claude Code を起動し直して解決
2. **Android Studio 初回セットアップ** — ユーザーが手動で Standard Setup を実行、SDK / platform-tools / emulator が入った
3. **環境変数設定** — `JAVA_HOME`（MSI が自動設定済み）、`ANDROID_HOME`、PATH を PowerShell で設定。`$env:LOCALAPPDATA` がリテラルのまま保存されるミスがあり修正
4. **`eas build --local` は Windows 非対応と判明** — "Unsupported platform, macOS or Linux is required to build apps for Android"。WSL2 で動かすか、クラウドビルドに戻すかの選択で、クラウドビルドに切り替え

Windows で `eas build --local` が使えないのは調査不足だった。先にドキュメントを確認すべきだった。ユーザーの時間を使って JDK + Android Studio をインストールした後に判明したのは申し訳ない。ただし adb が使える環境になったのは結果的に OTA デバッグで大いに役立った。

#### OTA Update の原因特定

adb logcat で以下の情報を順に取得し、原因を絞り込んだ。

**ステップ1: `updatePreviouslyFailed` の発見**

```
[updates] checkForUpdate: {"reason":"updatePreviouslyFailed","isAvailable":false}
```

過去の OTA 適用がクラッシュしたため、expo-updates が同一 update を「壊れている」と判断してスキップしていた。これが「何回再起動しても反映されない」の直接原因。

**ステップ2: 新しい update ID で配信 → クラッシュログ取得**

新しい update を配信してデータクリア後に適用させたところ、クラッシュの原因が判明:

```
Error: EXPO_PUBLIC_API_URL is not set. Production builds require this environment variable.
```

**ステップ3: 根本原因の特定**

`eas update` はローカルで Metro バンドラーを実行して JS バンドルを生成する。EAS クラウドビルドでは `eas env` に登録された環境変数が自動注入されるが、**ローカルの `eas update` では注入されない**。結果として `EXPO_PUBLIC_API_URL` が undefined のままバンドルされ、アプリ起動時にクラッシュ → `updatePreviouslyFailed` フラグが立つ → 以降の update がすべてスキップされる、という連鎖だった。

**ステップ4: 修正と検証**

環境変数を `eas env:list` で取得し、`EXPO_PUBLIC_API_URL=... eas update --clear-cache` で配信。`--clear-cache` は Metro キャッシュに環境変数なしのバンドルが残っていたため必要だった（最初はキャッシュクリアなしで配信し、バンドルハッシュが変わらず同じバグを再現した）。

最終的に adb logcat で `isEmbeddedLaunch: false`（OTA バンドルで起動）を確認して解決。

#### コード変更

- `_layout.tsx` に `reportError` を追加: `[updates]` のデバッグ情報を WAE-APM に送信するようにした
- `fetchUpdateAsync()` 成功後に `reloadAsync()` を追加: 1回のコールドスタートで OTA を即反映
- `docs/ops/mobile-ota.md` を作成: ローカルから OTA 配信する手順とトラブルシューティングをまとめた

### 技術的な学び

**`EXPO_PUBLIC_*` の注入タイミング**: Metro がバンドル時に `process.env.EXPO_PUBLIC_*` をリテラル値に置換する。EAS ビルドでは EAS サーバーが環境変数をセットしてから Metro を実行するが、ローカルの `eas update` ではローカルシェルの環境変数がそのまま使われる。EAS の環境変数管理とローカルのバンドルエクスポートは独立した仕組み。

**`updatePreviouslyFailed` の振る舞い**: expo-updates は update 適用後にクラッシュを検知すると、その update ID を失敗としてマークする。以降の `checkForUpdateAsync()` は同じ update に `isAvailable: false` を返す。これは安全策だが、原因がわからないとデバッグが難しい。新しい update ID で配信するか、アプリのデータを消去するとフラグがリセットされる。

**Metro キャッシュの罠**: 環境変数を変更しても、Metro キャッシュに古いバンドルが残っていると変更が反映されない。`--clear-cache` を忘れると、環境変数を正しく渡しても古いバンドルが配信される。

### 反省

**日記の読み方がまずかった。** ユーザーに「日記参照して」と言われたとき、`git show` で過去のコミット時点のファイルを読んで「書いてない」と回答した。ワーキングツリーのファイルを `Read` で読めば一発だったのに、無駄に遠回りした。ユーザーに「なんで git の過去コミット参照してたのか教えてくれる？」と突っ込まれたのは当然。

**`eas build --local` の対応プラットフォームを事前に確認しなかった。** JDK と Android Studio のインストールに時間をかけた後に「Windows 非対応」と判明した。ドキュメントを先に読むべきだった。ただし結果的に adb 環境が手に入ったので、OTA デバッグに使えたのは不幸中の幸い。

**環境変数の配信に2回失敗した。** 1回目は環境変数を渡したが Metro キャッシュで無効化。2回目は `--clear-cache` 付きで成功。キャッシュの影響を考慮できていなかった。

### 所感

前セッションで8回のビルドを使って特定できなかった OTA の問題を、今回は adb logcat 1本で `updatePreviouslyFailed` → 環境変数欠落と辿り着けた。ログが取れるか取れないかで調査効率が桁違いに変わるという当たり前のことを痛感した。

`reportError` を WAE-APM に送る仕組みを入れたのは良い判断だった。adb が使えない状況（ユーザーが外出先でテストする等）でも、サーバー側でログが確認できる。ただしこれ自体が OTA で配信されるため、OTA が壊れている状態では使えないという鶏卵問題がある。ネイティブビルドに最初から入れておくのが正解。

`docs/ops/mobile-ota.md` にまとめたのはユーザーの指示だが、こういう運用ドキュメントは自分から提案すべきだった。同じミス（環境変数なしで配信）を繰り返さないための仕組みを作るのは、言われてからやるべきことではない。
