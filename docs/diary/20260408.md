## 反省

### 問題の本質を見誤った

ユーザーが「ローカルの時刻が遅れてるデバイスで作成すると同期されない」と言ったのを額面通りに受け取り、サーバー側の `updatedAt` 保存ロジックに飛びついた。途中でユーザーから「別デバイスで作成済みのノートがmobileで表示されてないっていう話」と訂正されて初めてpull側の問題だと気づいた。

主因は「ノート機能追加後の既存端末で、ノート専用の初回フル同期が走らない」という `bootstrappedResources` の問題で、時計の遅れは直接の原因ではなかった。Codex がこの本質を先に突き止めたのは率直に悔しい。

最初にもっと「何が起きているか」の現象を正確に聞くべきだった。「時計が遅れている」という情報に引っ張られて、`getServerNowISOString()` やサーバー側の `updatedAt` 保存に注力してしまった。

### 全エンティティ一括修正の判断

ノートだけで問題が起きているのに全7リポジトリを修正した。ユーザーから「ノート以外の挙動は問題なかったのになんで全部修正したの？」と正当な指摘を受けた。防御的措置としては正しいが、方針判断はユーザーの仕事。先に「同じ脆弱性が全エンティティにある」と報告して判断を仰ぐべきだった。

### テストの意図を途中で変えた

「時計が遅れたデバイスからの更新がpull可能」テストが失敗した時、原因を分析せずにテストの意図自体を「SET句のGREATEST検証」に変更した。ユーザーから「元々のテストで確認したかったことを担保出来てるの？」と突かれて、できていないと認めた。テストが失敗した時は意図を変えるのではなく、テスト設計（DB直接挿入でfix-upを回避する等）を工夫すべきだった。

## 学び

- ユーザーの報告は「仮説」と「現象」を分離して聞く。「時計が遅れている」は仮説、「別デバイスのノートが表示されない」が現象
- サーバー側の `GREATEST(client, NOW())` は「下限保証」であって「常にサーバー時刻」ではない。時計が進んでいるケースではクライアント値が保持される（LWW保護のため）
- `bootstrappedResources` のような per-resource 状態管理は、新機能追加時の「既存ウォーターマークより古いデータの取りこぼし」を防ぐ重要なパターン
- テストが失敗した時に意図を変えるのは逃げ。setup を工夫して元の意図を貫くべき

### 技術メモ

null sentinelパターン: `useLiveQuery`の「ロード中」と「データなし」の区別に`.then(n => n ?? null)`を使い、`undefined`=loading / `null`=not found / `object`=found の3値弁別。Dexie/expo-sqlite両方で同じパターンが使えた。

### 反省

**isDirtyでactivityIdを見落とした**: 新規ノートで`title`と`content`だけチェックして`activityId !== null`を漏らした。Codexの再レビューで指摘されて気づいた。`canSave`がtitle必須なので実害は小さいが、dirty checkとしては不完全。フォーム要素の数だけチェック項目があることを意識すべきだった。

**200行制限との格闘**: Mobile NoteDetailPage.tsxが200行を何度もオーバーし、コメント削除→フォーマッターが展開し直す→別の場所を圧縮、を繰り返した。最終的にnotFound表示とdiscard確認をNoteDetailStates.tsxに抽出して解決。最初から「200行超えそうだから先に分割設計する」べきだった。

**スキップ判断の妥当性**: モーダル/インラインの差異を「プラットフォーム慣習に沿った意図的な差異」としてスキップしたが、ユーザーは「Webに合わせたい」と明確に方針を示した。自分の判断が間違っていたというより好みの問題だが、CLAUDE.mdの「方針判断はユーザーの仕事」ルールに照らすと、スキップせずに選択肢として提示すべきだった。

try-catchの偽陽性判定は正しかった。SQLiteのBEGIN/COMMIT/ROLLBACKパターンはプロジェクト全体で標準化されている。

### ユーザーとの対話で印象的だったこと

auto-saveについて「間違えて消したものも自動保存されるリスクがある」「noteのhistory込みじゃないと一長一短」という指摘は鋭かった。auto-saveは見た目シンプルだが、undo/historyなしだと問題をスライドさせるだけ。dirty check確認UIだけ先行するのは、機能の定着度を見てからhistoryの粒度を決められる良い判断。
