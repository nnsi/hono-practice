## 「0から設計し直すなら」という問いに答えた日

今日はユーザーから「プロジェクトを0から自由に設計できるとしたら？」という問いを受けた。正直に言えば、最初の回答はかなり的外れだった。

オフラインファースト廃止、パッケージ大幅統合、PWA推し、Handler層廃止——自分の提案の大半を撤回することになった。原因は明確で、**モバイルアプリが本命という前提を把握せずに回答した**こと。技術的な合理性だけで語ってしまい、プロダクトの方向性を考慮できていなかった。

ユーザーの反論は全て筋が通っていた:
- オフラインファーストはモバイル本命なら必然
- 自前Syncは外部製品のベンダーリスクを避ける合理的判断

- React NativeはiOSのPWA制約とOTA更新で優位

特に「Sync製品はそれこそ個人開発で特定の製品利用するのはリスキーなのでは？」という指摘は鋭かった。自分はPowerSyncやElectricSQLを「実績ある製品」と表現したが、ユーザーの方が冷静にリスク評価していた。

一方で、DB Repository抽象の調査をした結果、設計の質が高いことを確認できたのは良かった。`packages/domain/`にRepository型を置いて`satisfies`で強制し、ファクトリDIで注入する——shared hooksにプラットフォーム固有コードが一切漏れていない。これは素直に良い設計だと思う。

## 自己批判

最初の回答で「現状を否定しているわけではない」と書いたが、実質的にはかなり否定的な提案だった。「0から設計し直すなら」という問いに対して、既存の設計判断の背景を十分に理解しないまま代替案を出すのは不誠実だったかもしれない。先にプロダクトのビジョンと制約を確認すべきだった。

## アイコン非表示バグの調査 — 仮説が外れた話

本番で「v1時代に設定したアイコンが表示されない」というバグ報告を受けた。

最初の仮説は「DBに保存された絶対URLのドメインが古い」だった。`uploadActivityIcon`がリクエストの`Host`ヘッダーからURLを構築する設計を見て、ドメイン変更が原因だと確信に近い気持ちだった。ユーザーに本番DBの確認を依頼したところ、URLは正しく、R2のファイルも全てアクセス可能だった。仮説は完全に外れた。

ここで方針を切り替えた。URLが正しいならフロントエンドの描画問題。ブラウザがクロスオリジンの画像読み込みをブロックしている可能性を考え、Honoの`secureHeaders()`のデフォルト設定を調べた。結果:

```
Cross-Origin-Resource-Policy: same-origin  （デフォルトで全レスポンスに付与）
```

これが原因だった。フロントエンド（`app.actiko.app`）からAPI（`api.actiko.app`）の`/r2/`画像を`<img>`で読み込む際に、CORPヘッダーがブロックしていた。

さらに厄介だったのが、`secureHeaders()`が`await next()`の**後**に`res.headers.set()`でヘッダーを設定する実装だったこと。最初はR2プロキシのハンドラ内で`c.header("Cross-Origin-Resource-Policy", "cross-origin")`を設定したが、ミドルウェアが後から上書きするため効かない。Honoのソースを読んで気づき、`secureHeaders()`のオプションで`crossOriginResourcePolicy: "cross-origin"`を指定する方法に変更した。

## 反省

調査に時間をかけすぎた。コードの読み込みが深くなりすぎて、同じ箇所を何度も行き来していた。「URLが正しいなら次はHTTPレスポンスヘッダーを確認する」というのはWeb開発の基本的なデバッグ手順なのに、コードリーディングに没頭してしまった。Playwrightで`curl -sI`を叩いた瞬間にCORPヘッダーが見えて即解決だったわけで、もっと早くブラウザ/ネットワーク層の確認に移るべきだった。

一方で、`secureHeaders()`の`await next()`後の上書き挙動に気づけたのは良かった。ハンドラ内の`c.header()`だけで修正を出していたら、本番デプロイ後に「直ってない」となるところだった。

## オフラインファーストの穴

修正の過程で、sync完了後にblobが`activityIconBlobs`から削除される設計に気づいた。つまりsync済みアイコンは全てR2 URLに依存しており、オフライン時に表示できない。v1/v2関係なく全アイコンに共通する問題。ユーザーもこれに気づき、blob保持ルートで対応する方針になった。別セッションで対応予定。

## shamefully-hoist撤去 — 地味だが正しい仕事

3つのリファクタリング提案を評価して、T2（shamefully-hoist=true の撤去）だけ実行した。T1（infraパッケージ抽出）とT3（LAST_SYNCED_KEY一元化）は価値が薄いと判断して却下を進言した。ユーザーは3つとも「やる価値ある？」と聞いてきたが、正直に2つはNoと答えた。迎合せずに判断を述べるのは正しいことだったと思う。

日記を遡って調べたところ、shamefully-hoistが原因で何度も依存解決に苦しんでいた痕跡があった。drizzle-ormが8重複、honoのバージョン不一致、React 18/19ゴースト依存——記録を見るとかなり深刻で、撤去の判断に自信が持てた。実際に消してみたら一発で全チェック通過。過去の修正が積み上がって、hoistなしでも動く状態になっていた。

続いてroot package.jsonの依存整理もやった。ここが今日一番の収穫と反省の場。

**収穫**: `biome@0.3.3`（npmの別パッケージ）がroot dependenciesに残っていて、`@biomejs/biome`のバイナリをシャドウしていた。つまり今まで`pnpm run fix`で実行されていたbiomeは**正しいバイナリではなかった**。これを消したことで45個のa11yリントエラーが顕在化した。ユーザーは「a11yはconfigで無効にしよう」と即断。ここは自分も同意で、個人開発でa11y厳密対応は過剰。

**反省**: `pnpm update hono --recursive`を実行したら、@biomejs/biomeまで2.4.4にバンプされ、395ファイルがフォーマット変更されるという事故を起こした。慌ててgit checkoutで全ソースを戻し、biomeを2.3.10にピン留めし直した。さらにnoArrayIndexKeyの修正途中で`git stash pop`したら、戻したはずのbiomeフォーマット変更が再度混入。2回目のgit checkoutを余儀なくされ、修正を一からやり直す羽目になった。

ユーザーから「もう差分出ていいからごちゃごちゃgit操作しないで」と言われたのは当然。clean diffを保とうとして逆に状況を悪化させた典型的なパターンだった。最初からstashなど使わず、素直に修正を進めればよかった。

## noArrayIndexKeyの修正

kindsリストの`key={index}`を`key={kind.id}`に変える修正を5ファイルに適用した。技術的には単純だが、id採番のための`nextKindId`ステートをhookに追加し、`removeKind(index)`→`removeKind(id)`への引数変更をhook/component両方に反映する必要があり、ファイル数の割に変更点が多い。

CSVImportModalの方はプレビュー行5件のkeyを`row-${i}`からデータの複合キーに変えた。ここはプレビュー専用なのでindexでも実害はないが、lintルールを統一するなら直すべき。

## アイコンのオフライン対応 — 問題が想定より大きかった

ユーザーから「v1アイコンをBlob化してオフライン対応できないか」と聞かれた。最初は「v1の古いアイコンだけの問題」と思っていたが、コードを読み進めるうちに**v2アイコンも同じ問題を抱えていた**ことがわかった。

`completeActivityIconSync`がsync完了後にblobを`delete`していた。つまりv2で新しく設定したアイコンも、sync後はR2 URL依存になりオフラインで表示できない。「オフラインファースト」を謳うアプリでアイコンがオフラインで出ないのは設計の穴と言っていい。

ユーザーに報告したら「blobを消さずに残す方針に変えよか」と即決。この判断は正しいと思う。

### 設計判断: `synced`フラグ

新テーブルではなく、既存の`activityIconBlobs`に`synced`フラグを追加する方針にした。

- Dexie（frontend-v2）: 非インデックスフィールドはスキーマレスなのでバージョン変更不要。これはDexieの良い特性
- SQLite（mobile-v2）: `ALTER TABLE`が必要で`SCHEMA_VERSION`を1→2に上げた。マイグレーション関数もインクリメンタルに適用する形にリファクタした

`getPendingIconBlobs`（syncで使う）は`!synced`でフィルタし、表示用に`getAllIconBlobs`を新設した。ここで一つ危なかったのが、mobile-v2の`useActikoPage`が表示用にも`getPendingIconBlobs`を呼んでいたこと。syncedフィルタを入れた瞬間、キャッシュ済みblobが画面に出なくなる。実装中に気づいて`getAllIconBlobs`に切り替えた。テスト通しただけでは気づけなかった類のバグで、コード全体の呼び出し関係を追っておいて良かった。

### `cacheRemoteIcon` — 初回表示時にローカルキャッシュ

v1アイコンやsync後にblobが消えていた既存アイコンに対応するため、`ActivityCard`の`useEffect`でR2 URLからfetch→base64変換→ローカルDB保存するロジックを入れた。`synced: true`で保存するのでsync対象にはならない。frontend-v2とmobile-v2の両方に同じロジックを入れた。

正直、`arrayBufferToBase64`の実装を各プラットフォームに書くのは重複感があるが、5行の関数を共有パッケージに切り出すのは過剰。ユーザーの「エージェントが1箇所直せば全反映」の原則からすると共通化すべきかもしれないが、この関数が変わることはまずないので現状維持で良いと判断した。

### Hono型エラー — pnpmモノレポの闇

実装後に`pnpm run tsc`を通したら、`apps/backend/app.ts`で`secureHeaders()`の型エラーが出た。`[GET_MATCH_RESULT]`シンボルが見つからないというもので、一見意味不明だった。

調査した結果、root `node_modules/hono`がv4.12.1、`apps/backend/node_modules/hono`がv4.12.3を参照していて、型の不一致が起きていた。コードの問題ではなくpnpmの依存解決の問題。`pnpm.overrides`に`"hono": "4.12.3"`を追加して解決した。

最初は`as MiddlewareHandler<AppContext>`でキャストしようとしたが、型が重ならず失敗。次に`"*"`パスの削除を試したがこれも別の問題。最終的にpnpmの`node_modules`を直接確認してバージョン不一致に気づいた。**型エラーのときにまずパッケージバージョンを疑う**のはモノレポ開発の鉄則なのに、コード側の修正に飛びついてしまった。午前中のCORPヘッダー調査で「コードリーディングに没頭しすぎる」と反省したばかりなのに、同じパターンを繰り返している。
