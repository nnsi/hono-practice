# 2026-06-11

## ノートUX刷新セッション（mobile → Web統一）

「mobileのノート、どうにも使い勝手が良くないんだけどいまいち言語化出来ない」という依頼から始まったセッション。言語化できない不満をヒアリングで構造化する、という普段とは逆向きの進め方で、これがかなりうまくいった実感がある。

### ヒアリングについて

AskUserQuestionを2ラウンドに分けたのが正解だった。1ラウンド目は「どの場面でストレスか（複数選択）」「保存方式」「リッチ必要か」「利用シーン」という診断的な質問。2ラウンド目は1ラウンド目の回答を踏まえて「新規を開いた瞬間の体験」「Webで書いたものの表示」「紐づけUIの置き場所」という設計判断の質問。最初から設計の質問をしていたら、ユーザー自身が言語化できていない段階なので答えようがなかったはず。先に現状実装を調査してから質問を組んだのも効いた。「WebViewエディタが本文入力のもたつきの根本原因だろう」という仮説を持って質問できたから、選択肢が具体的になった。

結果として「書き始めるまで・入力中・保存時」の全部が痛いという回答で、編集体験の全面刷新になった。Saveボタン+破棄確認+設定パネル+FABという構成は、個々のパーツは真っ当でも、合計すると「メモを書く」という行為に対して重すぎた。自動保存にしたら破棄確認もSaveボタンも丸ごと消せる。UIの問題は足し算ではなく引き算で解決することが多い。

### 技術面の学び

- **IMESafeTextInputのdefaultValue設計**: 値をrefに固定してremountで初期化する設計なので、(1) 既存ノートはロード完了まで入力欄をマウントしない、(2) E2E用のプログラム的なsetContentはkey(editorEpoch)でremountする、という2つの回避が必要だった。コンポーネントのコメントに「ダイアログでremountされる前提」と書いてあったのを読んでいたから事前に対処できた。既存コードのコメントはちゃんと読むものだ。
- **markdownのRN描画は自前実装で足りた**: リッチエディタが出力するmarkdownのサブセット（h1/h2/bullet/ordered/blockquote/bold/italic/code）が分かっていたので、依存追加なしで100行弱のパーサ+ビューで済んだ。パーサをfrontend-sharedに置いてvitestでテストできる形にしたのは我ながら良い分割。
- **Webの自動保存で踏んだ競合**: 新規作成後にURLを実IDへreplaceするeffectが、「戻る」のnavigateを上書きして一覧から詳細に引き戻すバグ。E2Eを実行して初めて発覚した。leavingRefで抑止して解決したが、「flushの副作用としてsetCreatedId → effectでnavigate」という間接構造にした時点で、ナビゲーション同士の競合可能性を考えるべきだった。navigateを呼ぶ場所が2箇所以上あるなら必ず順序を疑う。
- **E2EがUIバグを実際に捕まえた**のは気持ちよかった。書き換えたE2Eが単なる追随ではなく検証として機能した証拠。

### 発見した既存の問題

E2E用のViteサーバー（PGlite + programmatic createServer）では、ログイン画面含め全ページでTailwindユーティリティが生成されておらず、未スタイルのままE2Eが走っていた。機能セレクタはclass名に依存しないので全テスト通ってしまう。つまりこのプロジェクトのWeb E2Eは視覚的検証能力がゼロのまま運用されてきた。今回スクリーンショットを撮ろうとしなければ誰も気づかなかったと思う。ユーザーには報告済み。`watch: null` とTailwind v4プラグインの相性を疑っているが深追いはしなかった。

### 反省・言い残したこと

- スクリーンショット撮影を2回やり直してからログイン画面で切り分けた。最初の1枚が未スタイルだった時点で「自分の変更外のページで再現するか」を先に確認すべきだった。切り分けの基本動作が後手に回った。
- mobile側は新規ノート作成後もルートパラメータが "new" のまま（Webだけ実IDにreplace）。mobileにはリロード概念がないので実害はないと判断したが、プラットフォーム間の非対称として記録しておく。後で誰かが「なぜWebだけ？」と思うかもしれない。
- mobileのMaestro E2Eはこの環境にシミュレータがなく未実行のまま。testID変更があるのでartifact再ビルド必須。マージ済みなので、次にmobile E2Eを回した時にcreate-note/edit-noteフローが落ちたら、まずartifactの古さを疑ってほしい。
- タイトル自動導出（未入力なら本文1行目）は、ユーザーが意図的にタイトルを空にしたいケースと衝突する仕様。Apple Notes等の前例があるので採用したが、「空にしたのに勝手に埋まる」というフィードバックが来る可能性は頭の隅に置いておく。

### 意見が分かれた点（自分の立場）

「Webとmobileは基本的に同じ体験をさせたい」と言われた時、文字通りに受け取ればWebのWYSIWYGエディタも捨てて生markdown編集にすべきだった。でもmobileのプレーン化はWebViewのもたつき対策であって思想ではない、と判断して「操作モデルだけ揃えてエディタは維持」を推奨し、ユーザーも選んだ。markdownショートカット周りのE2E資産を含めて捨てるには惜しすぎる機能だったし、「同じ体験」の本質は「開く→書く→勝手に保存される」というメンタルモデルの一致であってエディタ実装の一致ではない。この判断は正しかったと今も思う。
