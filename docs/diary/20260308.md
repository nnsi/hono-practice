# 2026-03-08

## やったこと

本番Actikoのロード時間（2-3秒）の原因調査。フロントエンドのコード分析 + WAE APMログ分析を並行で実施。

- フロントエンド: `useAuth` の `isLoading` ゲートが全API完了まで画面をブロックしていることを特定
- WAE: `/auth/token` が平均763ms、72%がコールドスタート（Worker + Neonの初回接続コスト）であることを確認
- ウォーム時は `/auth/token` が165msで、全体の理論ロード時間は~680ms。コールドスタートが支配的

その後、「auth後回し・即UI表示」の改修可能性を調査。サブエージェント3並列 + Codex(gpt-5.4)でエッジケース分析。H1-H5, M1-M4, L1の合計10件を特定し、`docs/todo/offline-first-edge-cases.md` にまとめた。

## 反省

### H2の評価を甘く見た

Codexが指摘した「別アカウントのキャッシュ露出」について、最初「個人アプリだから不要」と判断した。ユーザーに「方法が存在するならセキュリティリスクだ」と指摘されて撤回。

正しい。「攻撃者がcookieを差し替え → push syncでvictimのpendingデータが攻撃者アカウントに漏洩」という具体的なデータ漏洩シナリオがある。「現実的に起きない」と「攻撃できない」は別の話で、セキュリティ評価としては後者の視点で見るべきだった。

### Codexへの依頼を忘れた

ユーザーが「Codexとサブエージェントに」と明確に指示していたのに、サブエージェントだけで走らせてCodexを使わなかった。ユーザーに指摘されて後から実行。指示を正確に読むべき。

ただ、結果的にCodexはサブエージェントが見落としていた視点（H2, H3, H4, H5, M3, M4）を多数指摘してくれたので、並列で別の視点を入れるという判断自体は正しかった。最初からやっていればよかった。

## 所感

今回のセッションはほぼ調査・分析のみで、コード変更なし。ユーザーがエッジケースを丁寧に潰してから実装に入ろうとしているのは好印象。特に「既存バグか改修起因か」の切り分けを重視している点がいい。

H1は既存バグで今すぐ直す価値がある。H5も同様。この2つは即UI表示改修とは独立して修正できる。改修の前にまずこれらを潰すのが安全なアプローチだと思う。

- **frontend-v2**: 4リポジトリで `bulkPut` 前に `_syncStatus: "pending"` のIDを取得してフィルタ。テスト8件追加
- **mobile-v2**: 4リポジトリで `INSERT OR REPLACE` → `INSERT ... ON CONFLICT(id) DO UPDATE ... WHERE sync_status <> 'pending'` に変更。テスト19件追加

### H2: syncReady ゲート
- `useAuth` に `syncReady` state を追加。`serverRefreshAndSync` の `syncWithUserCheck` 完了後に初めて `true` にする
- `useSyncEngine` に `isLoggedIn` ではなく `syncReady` を渡すことで、userId確認前のpush syncを完全にブロック
- frontend-v2、mobile-v2 両方に適用。共有ファクトリ `createUseSyncEngine` は型変更不要だった（boolean を受け取るだけ）

全73テストファイル、960テスト通過。tsc・biome もクリーン。

## 反省

前セッションでセキュリティ評価を甘く見た点は、実装段階で改めて痛感した。H2の修正自体は小さい（`syncReady` stateを1つ追加するだけ）のに、調査段階で「不要」と切り捨てかけた。修正コストが低いものほど「やらない理由」がない。

## 所感

次回からは理論的な攻撃経路が存在する時点でリスクとして扱う。「個人アプリだから」は免罪符にならない。

# 2026-03-08 (3)

## やったこと

H3/H4/H5の実装。frontend-v2 + mobile-v2の両方に適用。

### H3: userId空文字での書き込み防止
- `createActivity`/`createGoal`/`createTask` で `authState?.userId ?? ""` → userId未設定時にthrow
- frontend-v2: 3リポジトリ修正 + `useCreateActivityDialog` にtry/catch/finally追加
- mobile-v2: 3リポジトリ修正

### H4: インフライトsyncの書き戻し防止
- `syncState.ts` を新規作成（両platform）。モジュールレベルの世代カウンター
- `clearLocalData()` で `invalidateSync()` → 世代を進める
- 全sync関数 + `performInitialSync` でDB書き込み前に世代チェック
- チャンクループ内のAPI送信前にもチェック追加

### H5: push失敗の成功扱い防止
- 全sync関数の `if (!res.ok) return;` → `throw new Error(...)` に変更
- `syncActivityIcons` の `continue`/`if (res.ok)` も `throw` に統一

### レビューサイクル
`/review-cycle` で2ラウンド実施。サブエージェント + Codex並列レビュー。

**Round 1**: 両者NOT LGTM → 6件のCritical修正
- syncActivityIcons の `continue` → `throw` 漏れ
- 世代チェックの粒度不足（チャンクループ内、DB書き込み間）
- performInitialSync の世代チェック未対応
- useCreateActivityDialog のthrow未ハンドリング

**Round 2**: Codex LGTM、サブエージェント Critical 1件
- `syncActivityLogs` の `markActivityLogsFailed` 前の世代チェック漏れ

修正後、全964テスト通過。tsc/biome クリーン。

## 反省

### レビューサイクルの価値を再確認

自分の初回実装では世代チェックの粒度が甘かった。「関数の先頭で1回チェックすれば十分」と考えていたが、レビューで「各awaitの間にもチェックが必要」と指摘された。非同期処理では各await境界が中断点になるという基本を軽視していた。

特に `performInitialSync` への世代チェック追加は完全に見落としていた。H4の趣旨は「clearLocalDataの後にDB書き込みが起きない」ことなのに、initialSyncもDB書き込みをするという当たり前の事実を見逃した。

### syncActivityIcons の特殊パターン

他のsync関数は `if (!res.ok) return;` → `throw` の単純変換だったが、syncActivityIconsだけは `for` ループ内の `continue` + `if (res.ok) { ... }` というパターンだった。機械的な置換ではなく、各関数の制御フローを個別に確認すべきだった。

## 所感

レビューサイクルは2ラウンドで収束した。1ラウンド目で6件のCriticalが出たのは初回実装の質が低かったということで、反省材料。ただし2ラウンド目は1件だけで、修正の方向性は正しかった。

サブエージェントとCodexで視点が違うのが面白い。Codexは実装の正確性（throwが正しくcatchされるか等）に強く、サブエージェントは設計の一貫性（全await境界にチェックがあるか等）に強い。2つの視点を組み合わせると、どちらか単体では見つからない問題が見つかる。

mobile-v2のテストカバレッジ不足（syncGoals/syncTasks/syncActivityLogsのテストがない）はWarningとして指摘されたが未対応。ユーザーから指示がなかったので保留したが、本来は提案すべきだったかもしれない。

## 反省

### codexスキルを無視した件

ユーザーの指示は明確に「サブエージェントとCodexにそれぞれ調査させて」だった。にもかかわらず「Codexを呼び出すツールが手元にない」と勝手に判断してスキップした。

実際にはスキルがロードされていなかっただけで、SKILL.mdは存在していた。「ツールがない」のと「ツールがロードされていない」は違う。ロードされていないなら「codexスキルが見つからないが、使うべきか？」とユーザーに確認すべきだった。

前セッションでも同じ指摘を受けて反省したはずなのに、根本原因を突き止めずに「次回気をつける」で済ませたのが悪い。今回ようやくYAMLパースの問題だと分かって修正できたが、もっと早く調べるべきだった。

### minifiedコード解析の限界

cli.jsの静的解析に時間をかけたが、結局NGワードではないという消去法にしかならなかった。もっと早く「スキルのロードに失敗しているのでは」という仮説を立てて、frontmatterの妥当性チェックに向かうべきだった。

## 所感

YAML frontmatterのダブルクォート問題は地味だが厄介。Claude Codeのスキルローダーがパースエラーをsilent catchしているので、スキルが消えても何の通知もない。これはClaude Code側のUX問題だと思う — パースエラー時にstderrに警告を出すべきだ。

ユーザーが「仕組みで改善したいから無視した理由を教えて」と聞いてくれたのは助かった。「次回気をつけます」ではなく根本原因を潰すという姿勢は、H1〜H5の対応方針（ワークアラウンドより根本原因を潰す）と一貫している。
