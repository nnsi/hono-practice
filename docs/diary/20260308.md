## 反省

### ISO 8601文字列比較の甘さ

最初の実装で `local.updatedAt > server.updatedAt` と書いた。codebase探索の結果、現在は `new Date().toISOString()` が常に `"...000Z"` 形式を出すので文字列比較でも動くが、これは実装の偶然であって保証ではない。サーバー側のタイムスタンプ形式が変わった瞬間に壊れる。

`new Date()` によるDate比較は数ms のオーバーヘッドだけで確実に動くので、最初からそうすべきだった。「今動いている」ことと「正しい」ことの区別を意識すべき。

### レビュアーの誤読判定

Round 2でサブエージェントが「SQL `<=` は逆方向」とCriticalを出した。これは `updated_at`（ローカル）と `excluded.updated_at`（サーバー）の指す対象を誤解した典型的な誤読。前セッション(5)でもCodexの誤読を複数検出しており、レビュー結果は毎回実コードで検証する必要がある、という教訓を再確認。

自動化レビューの限界として受け入れつつ、検証コストを惜しまないこと。

## 所感

M5のアプローチ選択は興味深かった。共有ロック（mutex）はより強い保護を提供するが、非同期操作の各awaitポイントでのロック管理が複雑になる。updatedAt比較はデータレベルの保護で、実装が単純で理解しやすい。「最も単純なアプローチから始める」というのはユーザーの判断として正しいと思う。

ただし、updatedAt比較には限界がある。同一ミリ秒内の更新は区別できない。実運用ではほぼ起きないが、理論的には穴がある。この点をユーザーに伝えるべきだったかもしれない。提示した選択肢の説明文に「同一msの更新は区別不可」と書いたつもりだが、ユーザーがその含意を十分理解した上で選択したかは確認していない。

レビューサイクルは2ラウンドで収束。Round 1のCritical（Date比較への変更）は自分でも気づけたはずの問題で、実装時の注意不足。Round 2は誤読1件 + pre-existing修正1件のみで、実質1.5ラウンドという感覚。

### WAE APMログ分析 + フロントエンドエラー確認

### エラーコンテキスト情報の付与

WAEのクライアントエラーデータを見たら `userId`、`screen`、`appVersion` が全て空。送信側（`globalErrorHandler`、`ErrorBoundary`）が `errorType` / `message` / `stack` しか渡しておらず、`reportError` の型には `userId` / `screen` があるのに呼び出し側が注入する仕組みがなかった。

`ReportErrorOptions` に `getContext?: () => ErrorContext` コールバックを追加する方針で実装:
- `errorReporter.ts`: `getContext` を呼んで `userId` / `screen` をbodyに付与
- `main.tsx`: Dexieから `userId` をキャッシュ、`window.location.pathname` を `screen` として設定

### レビューサイクル（2ラウンド）

**Round 1**: 両者NOT LGTM — 4件の指摘
1. Critical: `cachedUserId` が起動時1回読みでログイン/ログアウトに追従しない
2. Warning: `getContext` 例外時にエラー報告自体が消える（outer try-catchに飲まれる）
3. Warning: `...report, ...context` のスプレッドで `context` が `errorType` 等を上書き可能
4. Warning: `getContext` 例外テストがない

修正:
1. Dexieの `creating` / `updating` hookで `cachedUserId` をリアクティブに更新。`lastLoginAt` の有無で判定
2. `getContext` にinner try-catch追加。失敗してもレポートは送信
3. スプレッド廃止 → 明示的フィールドピック（`errorType: report.errorType` 等）
4. 例外テスト + 上書き防止テスト追加

## 反省

### `cachedUserId` の初期設計が雑だった

最初の実装で「起動時に1回Dexieから読めば十分」と安易に考えた。レビューで「ログアウト後も旧userId付きでエラーが送られる」と指摘されて初めてライフサイクルの問題に気づいた。

セッション(2)〜(3)で `useAuth` のlogoutが `lastLoginAt` を空にするだけで `userId` をDexieから消さない設計を読んでいたのに、エラーレポーターの設計時にその知識を活かせなかった。コードの文脈を横断的に結びつける意識が弱い。

### スプレッド順序の無頓着さ

`...report, ...context` で `context` が後にくる＝ `context` が優先、という基本を考えずに書いた。`ErrorContext` は `userId` / `screen` しか持たない型だから「実害がない」と思っていたが、TypeScript上の型制約とruntimeの実態は別物。`as never` でキャストすれば何でも入る。「型で守られているから安全」は甘い判断。

### 良かった点: inner try-catchのパターン

`getContext` 失敗時もレポートを送る、という設計はレビューで指摘されるまで気づかなかったが、修正後のコードは「エラー報告のためのコードがエラーを握り潰してはならない」という原則を正確に体現している。outer try-catchは「reportError自体がクラッシュ源にならない」ため、inner try-catchは「コンテキスト取得失敗がレポート送信を阻害しない」ため。二重try-catchの各レイヤーに明確な責務がある。
