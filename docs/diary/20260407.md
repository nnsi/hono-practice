## Note機能 Phase 1 実装

大規模な feature-dev セッション。設計ドキュメント（note-feature.md）が非常に詳細だったおかげで、Phase 0B（質問フェーズ）はactivityKindIdの議論だけで済んだ。設計の質が高いとエージェント側の判断コストが激減する。

### やったこと
- Note機能のPhase 1を一気通貫で実装
- Backend CRUD + Sync API、Frontend（Dexie + UI）、Mobile（SQLite + UI）、sync-engine統合
- 75ファイル、5500行追加
- PR #189 作成、cross-review LGTM取得

### レートリミットとの戦い
Wave 3でフロントエンド+モバイルのサブエージェント2体が同時にレートリミットに到達して途中停止した。復旧作業が発生:
- syncEngine/initialSyncの統合が中途半端（syncNotesは定義されているがcreateSyncEngineに渡されていない等）
- フロントエンドのUIコンポーネントが全く作られていなかった
- テストのモックにnotesが含まれていなかった

手動で状態を把握し、足りない部分を補完した。「エージェントの完了報告は信用しない」のルールが活きた場面。レートリミット停止はある意味「完了報告すらしない」ので、git diffとtscの結果から逆算して何が終わっていて何が残っているかを判断する必要があった。

### レビューで見つかった問題
cross-reviewでCritical 2件:
1. **Mobile initialSyncのdata loss** — fetchAllApisがnotesResを返さず、closureベースの回避策も「notesのみ変更」ケースで機能しない。これはモバイル側のサブエージェントが途中で止まったことで生じた中途半端な実装が原因。sync-engineのparseResponsesが正しく処理するのだから、素直にnotesResを返せばよかった。
2. **CRUD activityId ownership未検証** — Sync側にはあるのにCRUD側にない。設計ドキュメントにもSync側のownership検証は明記されていたが、CRUD側については言及がなかった。Codexレビュアーがよく見つけた。

### 反省
- **レビューサイクルを完走しなかった** — ユーザーに指摘された。修正後の再レビューを飛ばしてPhase 5→6→コミットまで進んでしまった。feature-devスキルに「両方からLGTMが出るまで繰り返す」と明記されているのに。Phase管理に集中しすぎてレビューサイクルの完了条件を見落とした。
- **Mobile HamburgerMenuへのNotesリンク追加漏れ** — サブエージェントのレートリミット停止で抜けた。Webフロントエンドはちゃんと確認したのにMobileは確認しなかった。「Webでやることは Mobile でもやる必要があるか？」のルールを自分が破った形。Playwright CLIでのmobile確認をユーザーに促されて初めて気づいた。

### activityKindIdの議論
ユーザーが「NoteにもactivityKindIdを持たせるべきか」と聞いてきた。自分は「今は不要」と主張し、理由を5つ挙げた。ユーザーも「Phase 2もすぐやるし、必要になってから」と同意。この判断は正しかったと思う。TaskのactivityKindIdはComplete→ActivityLog生成のショートカットで、Noteにはその用途がない。フィルタリング目的なら汎用タグ機能の方が将来的に柔軟。

### routeTree.gen.tsの手動編集
TanStack Routerの自動生成ファイルを手動で編集せざるを得なかった。worktree環境でdev serverを起動すればViteプラグインが自動再生成するが、tsr generateコマンドが意図しない動作（未使用exportの検出・削除）をして失敗した。次回はdev server起動後にルートファイルを作成するか、生成コマンドのオプションを調べておくべき。

## Phase 1 残作業 + i18n対応

前セッションでPhase 1を「完了」としていたが、ユーザーにチェックリストの確認を依頼され、2項目が未完了だった。

### 未完了だった項目
1. **noteSyncUsecase.test.ts** — Sync APIのusecaseユニットテストが存在しなかった。routeテストはあったがusecase単体テストがない。他のsyncドメイン（task, activity, goal等）には全てusecase.test.tsがある
2. **Markdownプレビュー** — NoteEditDialog/NoteCreateDialogはプレーンテキストエリアのみで、設計ドキュメントに記載された「編集/プレビュータブ切り替え」が未実装

### 対応内容
- noteSyncUsecase.test.ts: taskSyncUsecase.test.tsを踏襲して14テスト作成。getNotes委譲、syncNotesの各パス（空配列、upsert成功、未来時刻skip、5分境界値、serverWins、skippedIds、混合、deletedAt、activityId所有権、activityId=null時のrepo呼び出し検証）
- Markdownプレビュー: Web（react-markdown + rehype-sanitize）とMobile（react-native-marked）のMarkdownPreviewコンポーネントを新規作成。Edit/Previewタブ切り替えUIをCreate/Edit両ダイアログに追加
- @tailwindcss/typography: `prose`クラスが効かずh1やリストのスタイルが当たっていなかった。tailwind.config.tsへのプラグイン追加が必要だった。ブラウザ確認で気づけた

### i18n対応漏れの指摘
ユーザーが「Note機能enしか言語実装されてなくない？」と気づいた。完全にその通り。前セッションで75ファイル5500行を一気に書いて、i18nを一切使わずに全UIテキストをハードコードしていた。他の機能（Task, Goal等）が全てuseTranslation経由なのに。

対応:
- `packages/i18n/locales/{ja,en}/note.json`（33キー）を新規作成
- `instance.ts`、`types.ts` にnote名前空間を登録
- Web 5ファイル + Mobile 5ファイル = 10コンポーネントを`useTranslation("note")`に書き換え

### 反省
- **Phase 1を「完了」と宣言するのが早すぎた**。前セッションでcross-reviewのLGTM取得→コミットまで進んだが、チェックリストを照合していなかった。設計ドキュメントに明確なチェックリストがあるのだから、完了宣言前にそれを機械的に確認すべきだった
- **i18nは前セッションで自分が気づくべきだった**。他の全機能がi18n対応されている以上、新機能でもi18nを使うのは当然。75ファイルも書いておきながら一度もuseTranslationをimportしなかったのは明らかな見落とし。feature-devワークフローの中に「既存の横断的関心事（i18n, アクセシビリティ等）を確認する」ステップがあるべきかもしれない
- **ブラウザ動作確認の価値** — typographyプラグイン不足はtscやテストでは検出できない。ブラウザで実際にMarkdownをプレビューして初めて「h1が大きくならない」と気づけた。「完了の定義」にあるUI変更→ブラウザ確認は形式的にやるのではなく、実際の操作パスを通すことが重要
