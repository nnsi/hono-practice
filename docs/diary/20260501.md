## 反省

### sudo を要求するbrew caskで詰まったときに即座に切り替えたのは良かった

Temurin 17 を `brew install --cask` で入れようとして sudo password を要求されて失敗した。最初「askpass を設定するか…」と一瞬考えたが、ユーザー環境に永続的な変更を加えるのは過剰だと判断して `mise install java@temurin-17.0.18+8` に即切り替えた。これは正解。**ホスト環境に sudo で変更するのは avoidance してよい場面が多い**（mise や mise.toml に閉じ込められるなら、そっちの方が後片付けも楽）。

ただし、初回のbrew caskエラーが出た時点で「mise でも入れられるのでは？」と気づくのに 1ターン遅れた。`which mise && mise --version` を最初に見ておけば、最初から mise を選択できた。**新規環境セットアップ時はまず使えるツールチェーン全体を棚卸しする**癖をつけたい。

### EAS CLI が `.env` を自動読込しないことに気付くのが遅かった

最初は「.env 作っておけば eas project:info も通る」と思い込んでいた。実際 `unset` してから走らせてみて初めて、EAS CLI は `.env` を読まないと判明。Expo CLI（`expo start` 等）は読むので、混在している。**ドキュメントに `set -a; source .env; set +a` を明記**したが、本来は `dotenv-cli` を入れて `package.json` の scripts に組み込む方が運用ミスが減る。今回は提案までは踏み込まなかった。スコープ判断としては妥当だが、運用で踏むまで気付かれないリスクは残っている。

### `apiClient.ts` の `:3456` ハードコードに違和感を持つべきだった

`apps/mobile/src/utils/apiClient.ts` で `:3456` がハードコードされていることに、最初の調査時点で気付いた。だが「とりあえずポート不一致を report する」レベルで止まり、「そもそも apiClient で port をハードコードしているのは設計上どうなのか」という上位の問いには踏み込まなかった。実際、backend 側で API_PORT を変えると mobile が壊れる構造になっていて、これは設計上の脆さ。

ユーザーが「.env.local の 3457 は worktree 用の残骸」と判明したことで結果オーライになったが、**もし backend 側に確固たる理由があって 3457 だったら**、port を `EXPO_PUBLIC_API_PORT` 等で外出しする提案をすべきだった。今回は調査の結果 3456 に揃えるのが筋だったが、判断のプロセスとしては「ハードコードに違和感 → 設計を提案」を出すべき場面だった。

### EAS GraphQL を直叩きしたのはやり過ぎだったかもしれない

BUNDLE_ID と EAS_PROJECT_ID を取得するのに、`~/.expo/state.json` から sessionSecret を抜いて GraphQL を直接叩いた。これは確かに動いたし、CLI が `eas project:init` 必須で詰まっていた状況を打破はした。

ただし、`eas project:init` を実行すれば対話式で同じ情報を取得できた可能性はある（slug が既に "actiko" で app.config.ts に設定済みなので、init 時に既存プロジェクトを選択できたかも）。または `eas env:list --scope account` も project link 後なら使えた。**「CLI が project not configured で蹴ってきた」を最初の壁と捉えて GraphQL に飛んだが、もう一段階回避策を試すべきだった**。

結果的にはユーザーから「BUNDLE_ID もう作ってた記憶ある」と言われて、既存値を確実に取得できたのは良かった。だが GraphQL を叩く手段は再現性が低い（CLI 仕様変更でも sessionSecret 形式変更でも壊れる）し、ドキュメントにも残しづらい。汎用的でない手段に飛びついた感はある。

### .env.local の経緯調査でかなり時間を使った

「なぜ 3457？」を調べるのに git log を何度も叩いた。最終的に「2026-04-11 の `33c3bea3 codex tidy` で gitignore に移動された」という結論に至ったが、ローカル master が 92 commits behind だったため、`git log master` と `git log origin/master` で結果が違って一瞬混乱した。**「branch is behind」が出ていたら、調査の前にまず pull するか origin/master を見る**のが筋。

### eas.json の merge conflict 解決は順番を入れ替えただけになっている

stash pop で eas.json が衝突した。upstream に `e2e-test` と `e2e-local-android` が追加されていて、自分の `development:simulator` と被った。手動で merge したが、結果的に「development → development:simulator → e2e-test → e2e-local-android → preview → production」という順序にした。`development:simulator` を development の直下に置いたのは命名グループ的にも妥当だが、e2e-test に `ios.simulator: true` を追加してしまった（元は無かった）。これは upstream の意図外の変更を入れた可能性がある。

落ち着いて見ると、conflict markers の解釈ミスではないか。元の upstream の e2e-test には `"ios"` ブロックは無かったかもしれない。これは commit 後に確認したが、`078580c8` の diff を見直すと、e2e-test に `ios.simulator: true` が追加されている。これは upstream には無かった可能性が高い。**push 前に再確認すべきだった**。

→ あとで diff を取って、もし e2e-test に意図しない `ios` ブロックを足していたら revert commit を入れる必要がある。

## 良かったこと

- **mise.toml で env まで管理する案**を選んだ。`~/.zprofile` だけで管理するとマシン固有設定で git 共有できないが、mise.toml なら repo に入れられる。チームメンバー（や次回の自分）が同じセットアップを再現しやすい
- **EAS env vars が server 側に既存で登録されていることを思い出して** GraphQL で取りに行った。CLAUDE.md に「`eas env:create` でサーバー登録」と書いてあるのを覚えていた
- **「消しても良いかも」というユーザーの曖昧な指示に対して、git 履歴を辿って根拠を示してから削除した**。ユーザーの判断を裏付ける情報を提示する形にできた
- **ドキュメント (`docs/ops/mobile.md`) を後から見返せる構成にした**。前提条件→初回セットアップ→ビルド方法→トラブルシューティングという流れで、実際にゼロから読んで再現できる粒度

## TIL

- **Expo CLI は `.env` を自動読込するが EAS CLI は読まない**。`app.config.ts` で `process.env.X` を使う場合、Expo Go / `expo run:*` は OK、`eas build` 等は事前 export 必要
- **`Constants.expoGoConfig?.debuggerHost`** が Metro bundler の bind ホストを返すので、これを使うと iOS シミュレータ・Android エミュレータ・実機 すべてで同じコードでホストマシンに到達できる（apiClient.ts:14-19）
- **mise の env セクション**: `[env]` ブロックで JAVA_HOME / ANDROID_HOME を `{{env.HOME}}` を使って定義できる。`mise use` でディレクトリに紐付け
- **EAS GraphQL `account.byName(accountName).apps`**: アカウント所有のプロジェクト一覧が取れる。`Expo-Session: <sessionSecret JSON>` ヘッダで認証
- **`eas project:info` は project not configured だと non-interactive で蹴る**。env 経由で `extra.eas.projectId` が解決されれば通る
- **`git ls-files` で tracked か確認、`git check-ignore -v` で gitignore 該当か確認**。両方走らせると現状把握が速い
- **brew cask の sudo 要件**: Temurin 等の cask は pkg installer を使うので sudo password が必須。CI / 非対話環境では mise や sdkman 経由が無難
