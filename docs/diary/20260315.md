# 2026-03-15

## ブラウザ動作確認で発覚した3つのバグ修正

### やったこと

前セッション（3/14深夜）のtaskId実装後、ユーザーにブラウザ動作確認を促されて実機テストしたところ、3つのバグが連鎖的に発覚した。

#### バグ1: sync 400エラー（UpsertActivityLogRequestSchema）

activity-logs/sync が 400 を返していた。原因は `UpsertActivityLogRequestSchema` で `taskId: z.string().uuid().nullable()` としていたが、既存のActivityLog（Dexieに保存済み）には `taskId` フィールドが存在せず、sync時に `undefined` が送られていた。`.nullable()` は `null` のみ許可で `undefined` は弾く。`.nullish()` に変更して解決。

加えて、ローカルDBのPostgreSQLマイグレーションを実行していなかった。Dexie（IndexedDB）はバージョン追加で自動適用されるため動いていたが、PostgreSQLの `activity_log` テーブルに `task_id` カラムが存在せず、DrizzleのSELECTが500エラーを返していた。

#### バグ2: doneDateがUTC日付になる

`new Date().toISOString().split("T")[0]` はUTC日付を返す。JST 00:10 に完了すると `"2026-03-14"` になり、ユーザーの体感日付（3/15）と1日ずれる。`dayjs().format("YYYY-MM-DD")` に変更してローカルタイムゾーンを使うようにした。これは今回の変更で入れた問題ではなく既存コードだが、ActivityLog自動作成で顕在化した。

#### バグ3: quantity=null のタスクでActivityLog作成

activityIdがあればquantityの有無に関係なくActivityLogを作成していた。数量が空のActivityLogは無意味なので、`task.quantity != null` を条件に追加した。

### 反省

#### ブラウザ動作確認の質

前セッションで「ブラウザ動作確認済み」としたが、実際にはDexie（クライアント側）でしか確認していなかった。CLAUDE.mdの完了定義「DB変更 → マイグレーション適用 + APIが200を返す + DBにデータが入っていることまで確認」を満たしていない。PostgreSQLマイグレーション未実行で500が出る状態を見逃した。

Dexieはスキーマ変更が透過的に適用されるので「動いているように見える」のが厄介。サーバーサイドのDB変更がある場合は、sync APIが200を返すことまで確認すべきだった。

#### UTC日付の問題を見落とした

`new Date().toISOString().split("T")[0]` がUTC日付を返すことは基本的な知識なのに、レビューサイクルでも自分でも見落とした。レビュアーエージェントも指摘していない。JST深夜にテストしたから発覚したが、日中にテストしていたら気づかないまま出荷していた可能性がある。タイムゾーン関連のバグは「たまたま気づく」ではなく、チェックリストに入れるべき。

#### quantity条件の初期要件漏れ

ユーザーから「数量が設定されているタスクのみ自動記録にする、で合ってる？」と確認した際、ユーザーが「これ最初の要件になかったっけ？」と言っていた。activityId + quantity でActivityLogを作るのだから、quantityが空なら作らないのは当然の要件で、最初から条件に含めるべきだった。

### 所感

ユーザーに「ブラウザで動作確認した？」と聞かれた時点で自分の確認不足を認識した。前セッションの日記で「ブラウザ確認でDailyページにActivityLogが自動作成されているのを確認した瞬間、機能として完結した実感があった」と書いたが、実際にはサーバーサイドの確認を飛ばしていた。完結した実感は偽物だった。

3つのバグが全て「基本的な確認漏れ」から来ていることが痛い。マイグレーション未実行、UTCとローカルの混同、条件の考慮漏れ。どれも高度な問題ではない。レビューサイクルで設計レベルの指摘を裏取りする能力はあるのに、足元の基本を踏み外している。

深夜にテストしたことで日付ずれが発覚したのは不幸中の幸い。テストは様々な時間帯でやるべきという教訓。

## WAE APMログ分析

### やったこと

ユーザーの依頼で直近1週間のAPMログを分析した。WAEスキルの手順に沿って全クエリを実行し、レポートを出力した。

結果は良好だった。5xxエラー0件、クライアントエラーも2件（GIS読み込み失敗とRejected、いずれも既知の低深刻度パターン）。メインユーザー2人がそこそこ使い込んでいる中でこの数字は素直に優秀だと思う。

改善候補として3点挙げた：

1. `POST /v2/activities/sync` の `syncActivityEntities` → `syncActivityKindEntities` 直列を `Promise.all` で並列化
2. `GET /v2/activities` のクエリ統合
3. `POST /v2/activity-logs/sync` の所有権チェック並列化

ユーザーは1番に着手しようとしたが、コードを読んだ結果、**kindの所有権チェックがactivityのupsert完了に依存する**ことがわかり、並列化すると新規activity+kindの同時syncでkindがskipされる問題を報告した。ユーザーは「syncだしね」と納得して変更不要と判断。2番は「見辛くなるからこのまま」、3番は「許容範囲」とのこと。

### 所感

コード変更ゼロのセッションだった。が、無駄ではなかったと思う。

並列化の提案時に依存関係を見抜けたのは良かった。APMの数字だけ見ると「直列だから遅い、並列化すれば速くなる」と安直に結論できるが、syncの文脈では所有権チェックの順序が意味を持つ。分析フェーズのエージェントが「parallelizable」と報告してきたのをそのまま鵜呑みにせず、自分でusecaseのコードを読んで依存関係を確認したのは正しい判断だった。

ユーザーの「何もやってないけど一応日記書いとこか」という言葉が印象的。変更がなくても「現状に問題がない」という確認自体に価値がある、ということをユーザーも理解しているのだと思う。

### 日記がADRの素材として機能していた

今回一番印象的だったのは、ADRを書く作業が想像以上にスムーズだったこと。理由は明確で、日記に「なぜその判断に至ったか」のプロセスが詳細に記録されていたから。

具体的に役立ったのは:

1. **却下した代替案とその理由**: recordingModeConfigのテーブル案 vs JSONカラム案、リポジトリ共通化の3案比較、taskIdの代わりのupsert案 etc. ADRの「決定事項」セクションで「なぜ他の案を選ばなかったか」を書くのに、日記からほぼそのまま転記できた

2. **ユーザーとの議論の経緯**: `active_days integer[]` → `day_targets jsonb` の変遷、記録モードのv1→v2→v3の改訂理由、Debtが唯一の差別化という結論に至る過程。これらは「コンテキスト」セクションの骨格になった

3. **実装して分かった問題**: expo-sqliteの2段階バグ、VITE_API_URLフォールバックの罠、endDateのinclusive/exclusive問題。「備考」セクションに書くべき落とし穴が日記に全て記録されていた

日記のガイドラインに「異論があれば書く」「自己批判も書く」があることで、「最初はこう考えたが間違っていた」という記録が残っているのが大きかった。ADRで一番価値があるのは「なぜ他の案を選ばなかったか」の部分で、そこが日記から取れるのは効率的だった。

### ユーザーの反応

11件全て読んでもらって「特に指摘することなかった。コンテキストもしっかりしてる。」とのこと。ADRの品質が高かったのは自分の文章力というより、元の日記の記録精度のおかげだと思う。

ユーザーが「もしかして日記にやりとりがちゃんと書かれててそのままADRに出来る形になってた？」と気づいたのは鋭い。日記 → ADR のパイプラインが自然に成立していた。これは意図して設計したわけではなく、日記のガイドライン（正直に書く、議論を記録する、反省も書く）に従った結果として副次的に生まれた効果。

#### React Native Web でのテスト手法

RN Web は通常のHTMLと異なり、`type` アクションでテキスト入力ができない。JavaScriptで `nativeInputValueSetter` を使って値をセットし、`input` / `change` イベントをディスパッチするパターンが必要だった。

チェックボックス（TouchableOpacity）のクリックも `role="button"` や `role="checkbox"` が設定されていないため、座標ベースで `tabindex="0"` の要素を特定してクリックする方式が安定した。日記3/13でも同じ知見が記載されていたが、改めて確認できた。

ナビゲーションバーのタブは `<a>` タグではなく、テキストノードの座標を取得してクリックする方式で安定した。

#### Chrome MCP の接続切れ

今回もセッション中に4-5回 "Browser extension is not connected" が発生。`tabs_context_mcp` で復帰できるが操作の流れが途切れる。前セッション（3/14）の日記にも同じ問題が記載されている。3セッション連続で発生しており、Chrome MCP の安定性に構造的な問題がありそうだが、原因は特定できていない。
