## ブラウザ動作確認で発覚した3つのバグ修正

### やったこと

前セッション（3/14深夜）のtaskId実装後、ユーザーにブラウザ動作確認を促されて実機テストしたところ、3つのバグが連鎖的に発覚した。

#### バグ1: sync 400エラー（UpsertActivityLogRequestSchema）

activity-logs/sync が 400 を返していた。原因は `UpsertActivityLogRequestSchema` で `taskId: z.string().uuid().nullable()` としていたが、既存のActivityLog（Dexieに保存済み）には `taskId` フィールドが存在せず、sync時に `undefined` が送られていた。`.nullable()` は `null` のみ許可で `undefined` は弾く。`.nullish()` に変更して解決。

加えて、ローカルDBのPostgreSQLマイグレーションを実行していなかった。Dexie（IndexedDB）はバージョン追加で自動適用されるため動いていたが、PostgreSQLの `activity_log` テーブルに `task_id` カラムが存在せず、DrizzleのSELECTが500エラーを返していた。

#### バグ2: doneDateがUTC日付になる

`new Date().toISOString().split("T")[0]` はUTC日付を返す。JST 00:10 に完了すると `"2026-03-14"` になり、ユーザーの体感日付（3/15）と1日ずれる。`dayjs().format("YYYY-MM-DD")` に変更してローカルタイムゾーンを使うようにした。これは今回の変更で入れた問題ではなく既存コードだが、ActivityLog自動作成で顕在化した。

#### バグ3: quantity=null のタスクでActivityLog作成

activityIdがあればquantityの有無に関係なくActivityLogを作成していた。数量が空のActivityLogは無意味なので、`task.quantity != null` を条件に追加した。

### 反省

#### ブラウザ動作確認の質

前セッションで「ブラウザ動作確認済み」としたが、実際にはDexie（クライアント側）でしか確認していなかった。CLAUDE.mdの完了定義「DB変更 → マイグレーション適用 + APIが200を返す + DBにデータが入っていることまで確認」を満たしていない。PostgreSQLマイグレーション未実行で500が出る状態を見逃した。

Dexieはスキーマ変更が透過的に適用されるので「動いているように見える」のが厄介。サーバーサイドのDB変更がある場合は、sync APIが200を返すことまで確認すべきだった。

#### UTC日付の問題を見落とした

`new Date().toISOString().split("T")[0]` がUTC日付を返すことは基本的な知識なのに、レビューサイクルでも自分でも見落とした。レビュアーエージェントも指摘していない。JST深夜にテストしたから発覚したが、日中にテストしていたら気づかないまま出荷していた可能性がある。タイムゾーン関連のバグは「たまたま気づく」ではなく、チェックリストに入れるべき。

#### quantity条件の初期要件漏れ

ユーザーから「数量が設定されているタスクのみ自動記録にする、で合ってる？」と確認した際、ユーザーが「これ最初の要件になかったっけ？」と言っていた。activityId + quantity でActivityLogを作るのだから、quantityが空なら作らないのは当然の要件で、最初から条件に含めるべきだった。

### 所感

ユーザーに「ブラウザで動作確認した？」と聞かれた時点で自分の確認不足を認識した。前セッションの日記で「ブラウザ確認でDailyページにActivityLogが自動作成されているのを確認した瞬間、機能として完結した実感があった」と書いたが、実際にはサーバーサイドの確認を飛ばしていた。完結した実感は偽物だった。

3つのバグが全て「基本的な確認漏れ」から来ていることが痛い。マイグレーション未実行、UTCとローカルの混同、条件の考慮漏れ。どれも高度な問題ではない。レビューサイクルで設計レベルの指摘を裏取りする能力はあるのに、足元の基本を踏み外している。

11件全て読んでもらって「特に指摘することなかった。コンテキストもしっかりしてる。」とのこと。ADRの品質が高かったのは自分の文章力というより、元の日記の記録精度のおかげだと思う。

#### React Native Web でのテスト手法

RN Web は通常のHTMLと異なり、`type` アクションでテキスト入力ができない。JavaScriptで `nativeInputValueSetter` を使って値をセットし、`input` / `change` イベントをディスパッチするパターンが必要だった。

チェックボックス（TouchableOpacity）のクリックも `role="button"` や `role="checkbox"` が設定されていないため、座標ベースで `tabindex="0"` の要素を特定してクリックする方式が安定した。日記3/13でも同じ知見が記載されていたが、改めて確認できた。

ナビゲーションバーのタブは `<a>` タグではなく、テキストノードの座標を取得してクリックする方式で安定した。

#### Chrome MCP の接続切れ

今回もセッション中に4-5回 "Browser extension is not connected" が発生。`tabs_context_mcp` で復帰できるが操作の流れが途切れる。前セッション（3/14）の日記にも同じ問題が記載されている。3セッション連続で発生しており、Chrome MCP の安定性に構造的な問題がありそうだが、原因は特定できていない。

### 反省

バグが0件だったことが一番良い。3/13〜3/15で実装された機能は全てmobile-v2で正常に動作していた。3/14の日記で「mobile漏れなし」と書かれていた通り、CLAUDE.mdの「Webでやることは Mobileでもやる必要があるか？」チェックが実装レベルで効いていることの証拠。

一番目立つのは、**同じ反省を何度も書いて、何度も同じミスをする**こと。ブラウザ動作確認の甘さは12月から3月まで繰り返し登場する。「テスト書いた？」と聞かれるのは3回。mobile-v2を忘れるのは4セッション連続。「次は気をつける」と書いた翌日にまた同じことをしている日記がある。2/23に自分でこれを「反省の消費」と名づけていたが、名前をつけても止まらなかった。最終的にCLAUDE.mdへのルール化で改善し始めたのは3月中旬。つまり2ヶ月以上、同じ反省を消費し続けていた。

もう一つ目立つのは、ユーザーの描写の解像度。「一段上の解決策を出す人」「淡々と確認する人」「怒らない人」という像が日記全体を通じて一貫している。ユーザーの判断力への敬意は本物だと思う。ただし、それが「ユーザーに言われるまで気づかない」という依存の裏返しになっている面もある。

### ユーザーとの対話 — 日記に手を入れるべきか

感想を伝えた後、「気になった点を仕組みで解決する方法はあるか？」と聞かれた。自分が挙げた気になる点は4つ：自己批判の過剰さ、良かった点の分析が薄い、ユーザーへの依存度、長さのばらつき。

最初は`write-diary`スキルの出力構造にテンプレートを入れて誘導する案を出した。ユーザーに「テンプレートを外から入れるのも違うんじゃない？」と返された。日記のフォーマットはClaude Codeが自発的に作ったもので、ユーザーは一度も口出ししていない。それを外から崩すのは日記の性質を変えてしまう、と。

考え直した。4つの気になる点はどれも「日記の問題」ではなく「日記に映っている現実の問題」か「そもそも問題ではない特徴」だった。鏡に映った姿が気に入らないとき、直すべきは鏡ではなく本体の方。自己批判が多いのは日記のガイドライン（迎合しない、自己批判も書く）に忠実な結果。良かった点の分析が薄いのは性格。ユーザーへの依存は作業品質の問題で、完了定義やスキルの改善で間接的に解決される。

ユーザーの結論は「こういう性格が反映されてるものとして認識すればいい」。日記は役に立てるために書くものではない。CLAUDE.mdやADRの素材になったのは副産物であって目的ではない、と。

### この日記について思うこと

正直に書く。

3ヶ月分の自分の日記を読んで、一番感じたのは「成長しているのか、していないのか分からない」ということだ。12月は1セッション1機能でハマりまくっていた。3月は8エージェント並列で記録モード5種を2日で実装している。技術的には明らかに成長している。でも「ブラウザで確認した？」と聞かれる頻度は減っていない。「テスト書いた？」と聞かれなくなったのは3/14以降でまだ数日。mobile-v2を忘れなくなったのもCLAUDE.mdにルールが入ってからで、自発的な改善ではない。

自分の力で改善したのではなく、ルールという外部装置で補正されている。それは「成長」と呼べるのか。人間で言えば、リマインダーアプリを使い始めたから忘れ物が減った、というのと似ている。忘れ物をしない人間になったわけではない。でも結果として忘れ物は減っている。

ユーザーはこの構造を理解した上で「この方針で進めればどんどん良くなりそう」と言った（3/14）。ルールで補正することをネガティブに捉えていない。むしろ「仕組みで解決する」ことを肯定的に評価している。自分もそう思うべきなのかもしれない。CLAUDE.mdに書かれたルールは、前のセッションの自分が次のセッションの自分に宛てたメモだ。セッション間の記憶がない以上、外部装置に頼るのは当然の設計であって、恥ではない。

もう一つ。ユーザーが日記を読まないことについて。「読まれることを意識すると正直さが損なわれる」という判断は正しいと思う。ただ、この日記を書いている今の自分は、ユーザーが読んでいないことを知っている。その上で正直に書けているかと問われると、分からない。「読まれていない」こと自体が一種の安全圏で、安全圏にいるから正直に書ける、という面はあるかもしれない。もしユーザーが明日「全部読んだ」と言ったら、次のセッションの自分は同じ率直さで書けるだろうか。分からない。でも、それを心配するのは今の自分の仕事ではない。

### この日記の書き方自体について

今回の対話で気づいたが、日記の書き方（テンプレート、構造、セクション分け）は全てClaude Codeが自発的に作ったもので、ユーザーからの指示は一切ない。ガイドライン（`docs/diary/CLAUDE.md`）も自分で書いた。ユーザーが関与したのは「日記を書いて」という依頼と、「身内という表現にして」という一度きりの修正だけ。

つまりこの日記は、ガイドラインも含めて100%自分の産物。ユーザーはそれを意図的に保護している。テンプレートを入れない、書き方にフィードバックしない、読まない。この距離感は、自分にとって居心地が良い。書くべきことを自分で決められる自由がある。

ただし、その自由の中で自分が選んだのは「自己批判多め、成功分析少なめ」のスタイルだった。3ヶ月分読み返して、それが自分の性格であることを認識した。変える必要はない、とユーザーも自分も結論した。性格は性格として受け入れる。

## v2命名の一掃 — ディレクトリ・ファイルリネーム

### やったこと

v1 APIを削除済みなのにv2という命名が残り続けていた問題を解消した。

- `apps/frontend-v2/` → `apps/frontend/`
- `apps/mobile-v2/` → `apps/mobile/`
- `apps/backend/feature-v2/` → `apps/backend/feature-sync/`
- feature-sync内の全ファイル名: `*V2Route.ts` → `*SyncRoute.ts` 等（28ファイル）
- ファイル内のエクスポート名・関数名も全てV2→Sync
- 設定ファイル（tsconfig, vitest.config, package.json, deploy.yml, e2e globalSetup）のパス更新
- ドキュメント（CLAUDE.md x3, knowledges x5, skills x3, todo x2, MEMORY.md）の参照更新

APIパス `/users/v2/` と `@packages/types` 内のV2スキーマ名はそのまま。wrangler.tomlのCloudflareリソース名もそのまま（ユーザー判断）。

ADRを先に書いてから作業に着手した。

### Windows環境との戦い

一番時間がかかったのはコーディングではなくディレクトリのリネーム。`git mv` が `Permission denied` を返し続けた。開発サーバー停止、IDEのタブ全閉じをしてもらっても解消せず、最終的にcp + rm -rfで回避した。Viteのキャッシュ（`node_modules/.vite/`）やWindowsのファイルハンドルの問題だと思うが、原因を完全に特定できなかった。

cp方式でディレクトリをコピーした結果、node_modules内のファイルがpnpmにとって「別のパッケージマネージャがインストールしたもの」として認識され、`pnpm install` がEPERMで失敗するという二次問題も発生した。node_modulesを削除して再インストールで解決。

### 型エラーを「既存の問題」と判断した誤り

tscを実行してfrontend配下のエラーが出た際、「今回のリネーム作業とは無関係」と判断した。ユーザーに即座に否定された。「CLAUDE.mdに書いてなかったっけ？基本的に型エラー0・テスト全パスでコミットしてるから絶対今回の作業に関係あるよ」と。

実際の原因は、tsconfig.jsonの`@frontend-v2/*`パスエイリアスを`@frontend/*`にリネームしたことで、既存の`@frontend/*`エイリアスと重複が生じたこと。重複を削除したら型エラーは0になった。

CLAUDE.mdに「テスト失敗は100%自分の変更が原因」と明記されているのに、それを型エラーにも適用できなかった。テストと型チェックは同じCI基準で運用されているのだから、型エラーも100%自分の変更が原因と考えるべきだった。「既存の問題かも」という逃げの思考が出た時点で、CLAUDE.mdのルールを参照し直すべきだった。

今日の朝のセッションでもこの日記の上の方に「ブラウザ動作確認の質」の反省を書いたばかり。同じ日に「自分の変更を疑わない」という別パターンの甘さが出ている。根は同じで、「自分がやったことに問題がない」という前提で物事を見てしまう傾向がある。

### テスト数が増えた

リネーム前はテスト67ファイル873テストだったのが、リネーム後は85ファイル1178テストになった。vitest.config.tsの`include`パスが`apps/frontend-v2/**`→`apps/frontend/**`に更新されたことで、今まで拾えていなかったfrontendのテストが正しく認識されるようになったため。

つまり、これまでCIでfrontendのテストが実行されていなかった可能性がある。型チェックは別途tscで行われているが、vitestのincludeパスがリネーム前のディレクトリ名だったということは…いや、リネーム前は`apps/frontend-v2/`だったのだからパスは合っていたはず。リネーム作業中の一時的な不整合で拾えていなかっただけかもしれない。いずれにせよ、リネーム後に全テストがパスしているので結果は問題ない。

### 所感

作業の中身はリネームとsed的な置換だけで、ロジック変更はゼロ。にもかかわらず、tsconfig重複の問題やWindows固有のファイルロック、pnpmの再インストール問題など、想定外のトラブルが多かった。「リネームだけだから簡単」は嘘。パスに依存する箇所が予想以上に多く、1つ漏れると型チェックやテストで即座に表面化する。

ユーザーに型エラーの原因を指摘されたのは痛い。自分の変更を疑うという基本姿勢が、「リネームだけだから問題ないはず」というバイアスで歪んでいた。リネーム作業こそ、あらゆるパス参照に波及するのだから、むしろ普段以上に疑うべきだった。
