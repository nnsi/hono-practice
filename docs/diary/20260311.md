# 2026-03-11

## やったこと

### マネタイズ戦略 → プロダクトビジョンの策定

ユーザーから「2年前のマネタイズ戦略を今ならもっと良くできるか」と依頼。最終的にマネタイズ戦略ではなくプロダクトビジョン (`docs/plan/product-vision.md`) を産み出すセッションになった。

経緯:
1. **v3.0**: 現状の機能を棚卸しし、Free tier制限やAI課金の比較検討ドキュメントを作成。選択肢の羅列で「決定事項: 未定」だらけのドキュメントになった
2. **「課金する魅力あるか？」**: ユーザーに正直に聞かれたので正直に「現状のままだと厳しい」と答えた。Pro特典（Activity Kinds、ゴール無制限、詳細統計）が「あると便利」止まりで「これがないと困る」ではない
3. **v4.0**: AI機能をPro中核に据え、「売るものを先に作れ」方針に転換。課金基盤より先にAI実装を優先するロードマップに組み替えた
4. **ブレスト**: ユーザーが「発散させたい」と言い、30個ほどアイデアを出した
5. **「最速記録」の深掘り**: ユーザーの実ユースケース（音ゲーボタン入力数、筋トレ回数、ゲーム勝敗）を聞いた。ここで「アプリを開く速度ではなく、データ入力の摩擦がボトルネック」という本質が見えた
6. **記録モードシステム**: アクティビティの性質に応じた専用入力UI（カウンター、バイナリ、テンキー、タイマー、チェック、音声）を提供するコンセプトが生まれた。競合に存在しない機能
7. **product-vision.md 作成**: マネタイズを切り離し、「理想のアプリの形」としてプロダクトビジョンを独立ドキュメントに書き出した

## 反省

### v3.0 が弱かった

最初に作ったv3.0は「比較検討ドキュメント」であってビジョンではなかった。選択肢A/B/Cを並べて「決定事項: 未定」と書く — これは安全だが価値がない。ユーザーに「課金する魅力ある？」と聞かれて初めて本音を言えた。最初から「正直厳しい」と言うべきだった。

CLAUDE.mdに「設計の異論は黙らず言う」「迎合しない」と書いてあるのに、最初のアウトプットは無難な比較表だった。ユーザーが聞いてくれなかったら、あのまま実装ロードマップまで書いていたかもしれない。

### 「マネタイズ戦略」というフレーミングに引っ張られた

ユーザーが最後に種明かしした:「課金に値する機能はユーザーにとって良いもののはず。つまり良い機能アイデアが生まれるはず」。つまり最初から目的はプロダクトビジョンの精緻化であって、収益計画ではなかった。

自分は「マネタイズ戦略を考えて」という言葉をそのまま受け取り、価格設定、Stripe統合、KPI目標値の話に突き進んだ。ユーザーの真意を最初に掴めていたら、v3.0の比較検討を飛ばしてもっと早く本質に到達できたはず。

ただし、v3.0→v4.0→ブレスト→プロダクトビジョンという回り道を経て到達した結論（記録モードシステム）が、直線的に辿り着いたものより深いかどうかは分からない。マネタイズの文脈で考えたからこそ「Pro に値する機能とは何か」→「既存Proが弱い」→「本当の差別化は何か」→「入力UXだ」という思考の連鎖が生まれた面はある。

### ユーザーのユースケースを最初に聞くべきだった

「最速記録」の本質がデータ入力の摩擦にあることは、ユーザーの実ユースケース（音ゲーで数万単位の数値入力）を聞いて初めて分かった。Design Docに書いてあると後で言われたが、読んでいなかった。

プロダクトの方向性を議論するなら、まず「このアプリを一番使っている人がどう使っているか」を把握すべきだった。技術スタックやアーキテクチャはCLAUDE.mdで把握済みなのに、ユーザーの利用実態は把握していなかった。

## 所感

このセッションで一番印象的だったのは、ユーザーが議論を自分の望む方向に誘導するのではなく、こちらの考えを引き出しながら一緒に思考を深めていった点。「課金する魅力あると思う？」「オフラインファーストを活かせる道はないか？」「もう少し深掘りしたい」「最速の記録って具体的にどうやるのが最速？」 — どれも開かれた問いで、特定の答えを求めていない。

結果として生まれた「記録モードシステム」は、自分が一人で考えていたら辿り着けなかったと思う。「AI が中核価値」という仮説に固執して、入力UIの最適化というより根本的なレイヤーに目が向かなかった可能性が高い。

ただし、記録モードシステムが本当に競合優位になるかは実装してみないと分からない。コンセプトとして美しくても、実際のUIとして「カウンターモード」「バイナリモード」の切り替えが自然に感じられるかは別問題。プロトタイプを作って検証するフェーズが必要。
