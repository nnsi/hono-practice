7. **product-vision.md 作成**: マネタイズを切り離し、「理想のアプリの形」としてプロダクトビジョンを独立ドキュメントに書き出した

## 反省

### v3.0 が弱かった

最初に作ったv3.0は「比較検討ドキュメント」であってビジョンではなかった。選択肢A/B/Cを並べて「決定事項: 未定」と書く — これは安全だが価値がない。ユーザーに「課金する魅力ある？」と聞かれて初めて本音を言えた。最初から「正直厳しい」と言うべきだった。

CLAUDE.mdに「設計の異論は黙らず言う」「迎合しない」と書いてあるのに、最初のアウトプットは無難な比較表だった。ユーザーが聞いてくれなかったら、あのまま実装ロードマップまで書いていたかもしれない。

### 「マネタイズ戦略」というフレーミングに引っ張られた

ユーザーが最後に種明かしした:「課金に値する機能はユーザーにとって良いもののはず。つまり良い機能アイデアが生まれるはず」。つまり最初から目的はプロダクトビジョンの精緻化であって、収益計画ではなかった。

自分は「マネタイズ戦略を考えて」という言葉をそのまま受け取り、価格設定、Stripe統合、KPI目標値の話に突き進んだ。ユーザーの真意を最初に掴めていたら、v3.0の比較検討を飛ばしてもっと早く本質に到達できたはず。

ただし、v3.0→v4.0→ブレスト→プロダクトビジョンという回り道を経て到達した結論（記録モードシステム）が、直線的に辿り着いたものより深いかどうかは分からない。マネタイズの文脈で考えたからこそ「Pro に値する機能とは何か」→「既存Proが弱い」→「本当の差別化は何か」→「入力UXだ」という思考の連鎖が生まれた面はある。

### ユーザーのユースケースを最初に聞くべきだった

「最速記録」の本質がデータ入力の摩擦にあることは、ユーザーの実ユースケース（音ゲーで数万単位の数値入力）を聞いて初めて分かった。Design Docに書いてあると後で言われたが、読んでいなかった。

プロダクトの方向性を議論するなら、まず「このアプリを一番使っている人がどう使っているか」を把握すべきだった。技術スタックやアーキテクチャはCLAUDE.mdで把握済みなのに、ユーザーの利用実態は把握していなかった。

## 所感

## 記録モードUI注入の設計セッション

### やったこと

product-vision.md で定義した記録モードシステムを、フロントエンドでどう実現するかの設計。3回の改訂を経て最終形に到達した。

1. **v1: レジストリパターン**: `Map` + `registerRecordingMode()` + `setup.ts` で実行時登録。`LogFormBody` がレジストリを参照してモードコンポーネントを描画する設計
2. **v2: 静的 Record + 3層分離**: ユーザーから「mobile は？」と指摘を受けて改訂。`Map` を `Record<RecordingMode, Component>` に変更し、型レベルでモード網羅を強制。`packages/domain` / `packages/frontend-shared` / `apps/*/` の3層に分離してモバイル対応を明記
3. **v3: ヘッドレスフック + 薄いレンダラー**: 「もっと良い設計ある？」と聞かれて提案。各モードのロジックをヘッドレスフックとして `packages/frontend-shared` に共有し、プラットフォーム側はビューモデルを描画するだけの構成に

### 反省

#### v1 で mobile を考慮していなかった

これは恥ずかしい。コードベースには `apps/mobile-v2/` があり、`packages/frontend-shared` のファクトリパターンが既に Web/Mobile 共有のために存在している。その構造を毎日見ているのに、設計案で `apps/frontend-v2/` にレジストリも型もコンポーネントも全部置いた。ユーザーに指摘されるまで気づかなかった。

「記録UIの設計」という狭いスコープに集中しすぎて、システム全体の構造を視野に入れられていなかった。

#### v3 を最初から出せなかった

ヘッドレスフック案は、既存の `createUseLogForm` パターンの自然な拡張でしかない。v1 の時点で「既存パターンに合わせるべき」と自分で書いていたのに、実際に合わせたのは v3 になってから。v1/v2 では `resolveRecordingMode` や型定義だけを共有して、ロジック（状態管理、バリデーション、ハンドラ）はプラットフォーム側に置いていた。

「既存パターンとの一貫性」を設計原則として掲げながら、その原則を自分の設計に十分適用できていなかった。ユーザーの「もっと良い案ある？」がなければ v2 で止まっていた。

#### 「打ち止めか？」への判断は適切だった

v3 の後「これが最善？もう打ち止め？」と聞かれた。検討した代替案（スキーマ駆動UI、クロスプラットフォームUIライブラリ、サーバー駆動UI、プラグインシステム）がいずれも過剰であることを説明し、「打ち止め」と回答した。ここで無理にv4を捻り出さなかったのは正しかった。

### 所感

設計の改善が「自分で気づいた問題を自分で直す」ではなく「ユーザーに指摘されて直す」パターンだったのが引っかかる。v1→v2 は「mobile は？」、v2→v3 は「もっと良い案ある？」。どちらもユーザーの問いかけがトリガー。

一方で、設計はレビューを通じて良くなるものでもある。3回の改訂で段階的に洗練されたプロセス自体は健全だったとも思う。問題は「指摘される前に気づけたはずのことを気づけなかった」点であって、「改訂が必要だった」こと自体ではない。

`modeConfig` の扱いについてユーザーが「今回は考慮外って認識でOK？」と確認してきた件。「OK」とだけ答えたが、これは適切だった。スコープを広げて「ついでに設計しましょう」と言わなかったのは良い判断。

## recordingMode + recordingModeConfig 永続化

### やったこと

前セッションで作った記録モードUIコンポーネントに対して、バックエンド永続化の全レイヤーを貫通させた。Activity が `recordingMode` と `recordingModeConfig`（JSON テキスト）を持つようになり、ローカル DB / サーバー DB / 同期エンジンのすべてで永続化される。

#### 設計議論

ユーザーから「`activity_kind_steps` みたいなテーブルを作って管理する感じかな」と提案があった。これに対して JSON カラム案を対案として出した:

- steps は Activity 単位の設定であって ActivityKind 単位ではない（命名の混乱リスク）
- 別テーブルにすると同期エンジンに新しいエンティティが増える（`_syncStatus` 管理、chunk 分割、serverWins 処理…）
- `[1, 5, 10]` 程度の配列を正規化する意味が薄い
- 将来 binary モードのラベル設定等も必要になるが、モードごとにテーブルを増やすのは辛い

ユーザーは即座に JSON カラム案を採用。また、既存の `CreateActivityRequest.quantityOption` が実はまさに steps のために作られていたが DB にもリポジトリにも対応がない死んだフィールドだったことが判明し、削除に合意。

#### 実装の流れ

1. **Phase 0-2（直列）**: Config 型定義（zod discriminated union）→ 共通型変更（8 ファイル）→ Drizzle マイグレーション生成 + backfill SQL
2. **Phase 3（並列 4 エージェント）**: Backend / Sync mapper / Frontend-v2 / Mobile-v2 を同時実行
3. **追加修正**: エージェントのスコープ外だった v1 テストファイル 2 件 + mobile-v2 rowMappers テストを自分で修正

#### 主な変更

- **新規ファイル**: `recordingModeConfig.ts`（parse/serialize/default + zod schema）、`RecordingModeSelector.tsx`（モード選択 + counter steps 編集 UI）、`apiMappers.test.ts`
- **型変更**: `ActivityRecord`, `ActivitySchema`, `ActivityBase`, sync DTOs, API request/response の 8 ファイル
- **DB**: Postgres migration `0025`（2 カラム + timer backfill SQL）、Dexie v4（upgrade handler）、SQLite v3（ALTER TABLE + backfill）
- **Backend**: v2 repo upsert に 2 カラム追加、v1 usecase に受け渡し追加
- **Sync**: `mapApiActivity` に snake/camel 両対応マッピング追加
- **UI**: Create/Edit ダイアログに `RecordingModeSelector` 配置

#### 検証

CI 全パス: 75 files, 1004 tests, tsc OK, biome OK

### 反省

#### `quantityOption` の発見が遅かった

`CreateActivityRequest` に `quantityOption: z.array(z.number()).optional()` というフィールドが既に存在していたことに、ユーザーに言われるまで気づかなかった。Explore エージェントが見つけてきてはいたが、自分がその意味を深く考えなかった。ユーザーは「それは steps の文脈で考えていたやつだ」と即座に答えた。

コードベース探索で「既にある似た概念」を見つけたとき、それが今回の設計とどう関係するかを自分で考察し、ユーザーに確認すべきだった。Explore エージェントに情報収集を任せるだけでなく、収集結果の「解釈」は自分の仕事。

#### 型変更の波及を過小評価した

`ActivityRecord` に 2 フィールドを追加し `ActivityBase.recordingMode` を required にした結果、想定以上のテストファイルで型エラーが発生した。v1 テスト 2 件 + mobile-v2 rowMappers テストはエージェントのスコープ外で、自分が手動修正する羽目になった。

計画段階で「型変更の波及範囲」を grep して、影響を受けるテストファイルを全て洗い出し、エージェントのスコープに含めるべきだった。特に `Activity` 型のモックを作っているテストファイルは確実に壊れると予見できたはず。

#### 並列エージェント運用は安定してきた

前セッションに続き、直列（インターフェース変更）→ 並列（実装）の戦略がうまくワークした。4 エージェント同時起動で、ファイル編集の重複ゼロ。全エージェントが CI パスする状態で戻ってきた（mobile-v2 テストの 1 件失敗を除く）。

プロンプトに「対象ファイル（これ以外は編集しないこと）」を明記したのが効いている。エージェントがスコープ外のファイルに手を出さないので、マージ時の競合リスクがない。

### 所感

「テーブル vs JSON カラム」の設計判断でユーザーと議論できたのは良かった。JSON カラムを推した理由（同期コスト、データの性質、拡張性）を具体的に述べて、ユーザーがすぐ納得した。CLAUDE.md の「別の案が良いと思ったなら理由を述べて議論する」を実践できた場面。

`recordingModeConfig` の型設計で discriminated union を選んだのは正しかったと思う。counter の `steps` と binary の `labels` を同じ JSON カラムに入れつつ、zod でバリデーションをかける。`parseRecordingModeConfig` が不正 JSON を安全に null に落とすので、DB からゴミが読まれても壊れない。

一方で、この永続化の「完成度」については疑問がある。DB マイグレーションを生成したが、本番環境に適用するのはユーザーの判断待ち。Dexie の version(4) upgrade handler はインラインで `isTimeUnit` ロジックを再実装しているが、ドメインの `isTimeUnit` を import するのではなく文字列リテラルをハードコードした。Dexie の upgrade callback は DB モジュール内で完結すべきという判断だが、ロジックの重複は事実。将来 `isTimeUnit` の対象単位が増えたとき、ここを更新し忘れるリスクがある（ただし upgrade は一度しか走らないので、新しい単位が追加される前に upgrade 済みの端末には影響しない）。

## CounterMode 実装 + ブラウザ動作確認 + RecordingModeSelector ステップ永続化バグ修正

### やったこと

前セッションの続き。CounterMode の headless hook + レンダラーを実装し、Web/Mobile 両方でブラウザ動作確認を行った。

#### CounterMode 実装

1. **Headless Hook**: `packages/frontend-shared/recording-modes/modes/createUseCounterMode.ts` — ファクトリ DI パターンで `useState` を注入。ViewModel は `count`, `steps`, `increment`, `decrement`, `reset`, `memo`, `submit` 等
2. **テスト**: `createUseCounterMode.test.ts` — useState mock の型合わせに苦労（`readonly` tuple の非互換、ゼロ引数オーバーロードの欠如）
3. **Frontend-v2 / Mobile-v2 レンダラー**: 並列エージェントで実装。カウント表示 + ステップボタンペア (-N/+N) + リセット + メモ + 保存
4. **レジストリ更新**: `counter: ManualMode` → `counter: CounterMode` に変更

#### ブラウザ動作確認

**Web**: Actiko → テスト腕立て（counter mode）→ Daily → カウンターUI表示。+1×3 + +5×1 = 8 で記録保存成功。編集ダイアログでステップ値を `1, 3, 5` に変更 → Daily で `-1/+1`, `-3/+3`, `-5/+5` ボタンに即反映。

**Mobile**: RecordingModeSelector が未実装だったため新規作成（Web から移植）。カウンターモードでアクティビティ作成 → Daily でカウンターUI表示。+1×3 + +10×1 = 13 で記録保存成功。

#### RecordingModeSelector ステップ値永続化バグ

ブラウザ確認中に、作成時に指定したステップ値 `1, 5, 10` が保存されず、デフォルト `1, 10, 100` のままになるバグを発見。

**原因**: `defaultValue` + `onBlur`/`onEndEditing` パターン。ステップ入力フィールドからフォーカスを外す前に「作成」ボタンを押すと、`onBlur` による `setRecordingModeConfig` と `handleCreate` の `recordingModeConfig` 読み取りが React のバッチング内で同時に処理され、`handleCreate` は古い値を読む。

**修正**: 両プラットフォームの RecordingModeSelector で `defaultValue` + `onBlur`/`onEndEditing` → ローカル `useState` + `value` + `onChange`/`onChangeText` に変更。毎キーストロークで親に即反映するため、送信時に古い値を読むことがない。

最初 `useEffect` でモード切替時のローカル state 同期を書いたが、ユーザーから「useEffect はそういうものに使う hook じゃない」と指摘され、`handleModeChange` 内で直接 `setStepsText` を呼ぶ形に修正した。

#### バイナリモード確認

ユーザーから「バイナリモードも確認して」と依頼。調査の結果、binary は型定義と config schema のみ存在し、RecordingModeSelector の VISIBLE_MODES にも含まれず、レジストリでは ManualMode にフォールバック。実装が必要な旨を報告し、次セッションに持ち越し。

### 反省

#### `defaultValue` + `onBlur` パターンの危険性を見抜けなかった

Web 版の RecordingModeSelector を最初に書いたとき（前セッション、エージェント実装）、`defaultValue` + `onBlur` で実装された。Mobile 版もそのパターンをそのまま移植した。React のバッチングで「blur のstate更新がsubmitハンドラに間に合わない」問題は、React 18/19 では基本的な知識のはずだが、実装時もレビュー時も気づけなかった。

ユーザーに「永続化できてないってことじゃん」と指摘されて初めて問題に気づいた。しかもユーザーの指摘は UI の結果（ステップ値がデフォルトのまま）を見ての気づきであって、React の内部動作を分析したわけではない。つまり自分がブラウザ確認中に「ステップ値が違う」ことに気づいて原因を追うべきだった。実際、最初の Mobile 確認時にステップ値が `1, 10, 100`（デフォルト）で表示されていたのに、「作成時のcommitタイミングの問題かもしれないが、カウンターモードの実装としては正しく動作しています」と報告してスルーした。これは完全に自分のミス。

#### `useEffect` の濫用を指摘されたのは正しい

最初の修正で `useEffect(() => { if (recordingMode === "counter") setStepsText(...) }, [recordingMode])` と書いた。ユーザーから「useEffect はそういうものに使う hook じゃないよね？」と言われた。その通り。config が変わるタイミングは `handleModeChange` だけなので、そこで直接 `setStepsText` すれば十分。useEffect で「外部変化を監視して同期する」パターンは、このケースでは不要な間接性を持ち込んでいる。

React 19 を使っているプロジェクトで、useEffect を副作用の同期以外に使うのは時代遅れだし、React の方向性にも反する。ユーザーに React バージョンを確認されたのも、暗に「React 19 なら分かるでしょ」という意味だったと思う。

#### DB マイグレーションを手動 SQL で適用しようとした件

前セッション（ログに記録あり）で DB マイグレーションを `pnpm db-migrate` ではなく手動 SQL で適用しようとして、ユーザーに「ジャーナルテーブル自作のレコード挿入で本当に問題ない？」と止められた。これは恥ずかしいミスで、Drizzle のマイグレーションジャーナルの存在意義を理解していたのにショートカットを取ろうとした。

### 所感

このセッションは「自分で気づくべきことをユーザーに指摘される」パターンが多かった。ステップ値の永続化バグ、useEffect の不適切な使用、React 19 の理解不足。どれも自分で気づけたはずの問題。

ただ、CounterMode の headless hook 自体の設計は良かったと思う。`resolveSteps` が config パース失敗時にデフォルトにフォールバックする仕組みは堅牢だし、ファクトリ DI で useState を注入するパターンはテスタブル。問題は hook の外側（RecordingModeSelector の入力管理）にあった。

バイナリモードが型定義だけで実装が空なのを確認したとき、ユーザーは「なるほどね〜、じゃ次のセッションで実装しようか」とあっさり受け入れた。スコープの切り方が上手い。自分なら「ついでに」と広げたくなるところを、ユーザーは適切に区切る。
