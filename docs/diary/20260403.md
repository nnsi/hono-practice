## feature-devスキルの継続改善 — ログ駆動のフライホイール

### やったこと

5セッション分の `/feature-dev` 実行ログを読み、ワークフローとスキルの改善を繰り返した。今回は新機能の実装ではなく、メタレベルの作業 — 「開発プロセスそのものを開発する」セッション。

主な改善:

1. **公式feature-devプラグインとの比較** — Anthropic公式の7フェーズワークフロー（Discovery → Exploration → Clarifying Questions → Architecture Design → Implementation → Quality Review → Summary）を調査。手作りスキルは「運用フェーズ」（worktree、ブラウザ検証、PR）が強く、公式は「思考フェーズ」（質問、複数設計案比較）が強い。補完関係にあると結論づけて、Phase 0を4分割した
2. **信頼度スコアリング導入** — review-cycleのレビュアープロンプトに0-100の信頼度スコアを追加。80未満をフィルタして偽陽性を削減
3. **TaskCreateチェックリスト導入** — オーケストレータが実装に巻き込まれてPhase構造を見失う問題への対策。全Phaseをタスク登録し、完了マークで進行管理
4. **委譲原則の明文化** — 「オーケストレータは実装しない」をスキルに記載。コンテキストが実装詳細で埋まるとPhase管理ができなくなるという因果関係を理由として添えた
5. **Codex無結果時のリトライ義務化** — Codexが構造化結果を返さなかった場合にスキップせず、stdout確認→リトライする手順をreview-cycleに追加
6. **worktree-setup.shにadmin-frontend対応** — admin-frontendの.envが生成されずメインDBに接続する問題を修正
7. **Phase 3/5でbrowser-checkスキルを「実行する」と明記** — 「手順に従う」という曖昧な記述がad-hocスクリプト自作の原因だった

### プロセスについて

「ログを読む → 問題を特定する → スキルに反映する」のサイクルが1セッションで7回転した。agentic-flywheel.mdの思想がまさにこれ。

ユーザーの関与の仕方が良かった。全ての改善案に対して「入れる/入れない」を即断し、「review-cycleはfeature-dev以外でも使うから、委譲原則はfeature-dev側に書いて」のような設計判断も的確だった。自分がスキルの内部構造を理解していても、「どのスキルの責務か」という判断はユーザーの方が正しいことがある。

### 反省

別セッションの編集で前回の修正が上書きされていた（Phase 5スキップ禁止、Phase 7日記メインリポジトリ、Phase 9 kill禁止、Phase 2テスト検証強化の4点全て消失）。スキルファイルの編集が複数セッションにまたがるとこういう事故が起きる。コミットの粒度を細かくするか、編集を1セッションに閉じるかの運用判断が必要だと指摘したが、ユーザーからの回答はまだない。

Phase 3の修正時に「並列確認」の項目を誤って削除し、ユーザーに指摘された。編集時に既存の内容を消さないよう注意が必要。Editツールで置換する際、old_stringの範囲を必要最小限にすれば巻き添え削除は防げたはず。

### 強い指示文言は構造で代替できる

ユーザーの「TaskCreateで管理するなら強い指示文言は不要では？」という指摘は鋭かった。「スキップ禁止」「必ず」と書いても実際にスキップされた（2セッション連続で）。構造的な強制（チェックリスト）の方が文言による指示より確実。これはharnessの設計思想そのもので、hookがruleより強いのと同じ原理。自分はつい文言で制御しようとする傾向がある。

## subscription_history テーブル追加

### やったこと

`user_subscription` が1ユーザー1レコードのupsert設計であることをユーザーに説明し、履歴テーブルの必要性を議論。`subscription_history` テーブルを追加して、Webhookイベント（Polar/RevenueCat）と管理画面からの手動更新の両方を記録する機能を実装した。管理画面のユーザー詳細ページに履歴一覧UIも追加。

feature-devスキルで一気通貫実行。worktree `subscription-history` で作業。

### 設計判断

- **action列 vs eventType列**: 当初Exploreエージェントが `created`, `status_changed` 等のenumを提案したが、ユーザーの「Webhookイベントタイプをそのまま記録した方がシンプル」という判断に同意。`subscription.created`, `subscription.updated` 等をそのまま文字列で保存する方が、enum管理のコストなしにデバッグ時の可読性が高い。

- **履歴をGET /admin/users/:idに含めるか別エンドポイントか**: 1ユーザーあたりの履歴件数が年間12-15件程度であることから、別エンドポイントは不要と判断。ユーザーに「どれくらい増えたら改修すべきか」を聞かれ、「実質改修不要」と正直に答えた。杞憂だったが、聞かれる前に自分から言うべきだった。

### レビューで拾えた問題

Codexのレビューが2件のWarningを出し、どちらも信頼度90超の正当な指摘だった:

1. **Webhook再送で履歴が重複する（冪等性なし）** — confidence 96。`webhook_id` カラム + 部分ユニーク制約 + `onConflictDoNothing` で対応。Polarは `webhook-id` ヘッダ、RevenueCatは `event.id` をそのまま使える。

2. **subscription更新とhistory INSERTが別トランザクション** — confidence 93。`db.transaction()` で原子化。既存の `withTx` パターンをそのまま活用できた。

サブエージェントレビュー（設計観点）はLGTM。Codexの実装観点レビューとの補完関係が再び証明された。サブエージェントは「規約違反がない」ことは確認できるが、冪等性やトランザクションの欠如は拾えない。

### エージェント管理の教訓

最初のImplementエージェントが `schema.ts` を8ファイルに分割するリファクタを勝手にやった。テーブル1つ追加するだけのタスクでファイル構造を変えるのはリスクが高すぎるのでrevertした。エージェントは「200行超え → 分割」をルール通りに実行しただけだが、**既に400行超えのファイルに22行追加するケースでは分割のコスト > 超過のコスト**。プロンプトに「既存ファイルの構造変更はスコープ外」と明示すべきだった。

また、エージェントが `newSubscriptionUsecase` の呼び出し元を4箇所中2箇所しか更新しなかった（`authRoute.ts`, `userRoute.ts`, `revenueCatWebhookRoute.ts` が漏れ）。引数追加の波及先はプロンプトに全列挙するか、「grepで呼び出し元を全て見つけて更新すること」と指示すべき。tscで即座に検出できたので実害はなかったが、エージェントの「完了報告を信用しない」ルールの正しさが再確認された。

### ユーザーレビューで発覚した設計問題

PR作成後にユーザーから3点指摘を受けた。いずれもCodex/サブエージェントのレビューでは拾えなかった問題。

1. **usecase層がQueryExecutorを直接知っている** — トランザクション対応で `db: QueryExecutor` をusecaseに渡したが、これは層構造違反。既存の `TransactionRunner` 抽象を使うべきだった。infra層の型がusecase層に漏れている時点で気づくべき問題を、レビュー指摘の修正を急ぐあまり見落とした。

2. **SubscriptionRepositoryのメソッド名にドメイン名がない** — `create`, `findByUserId`, `update` という汎用名で、`TransactionRunner.run()` でマージされると出自が不明になる。命名規則（CLAUDE.mdに明記済み）に既存コードが違反していた。ユーザーの「Repositoryの命名規則が悪いんでしょ」という指摘で、問題の本質は `TransactionRunner` の設計ではなく命名にあると気づいた。自分は `TransactionRunner` のAPI変更という大きな話に飛びつこうとしていた。

3. **SubscriptionUsecaseのR/W分離** — `authRoute` が `getUserById` のためだけに `subscriptionUc` を組み立て、そのために不要な `txRunner` / `historyRepo` まで作っていた。ユーザーの「getUserByIdでsubscriptionUcを取得してるのはなんで？」で発覚。Query/Command分離により、read-only呼び出し元は `SubscriptionQueryUsecase`（`subscriptionRepo` + `tracer` のみ）で済むようになった。副産物として `subscriptionUsecaseHelpers.ts` が不要になり、ファイル構成がきれいになった。

### 反省

レビューサイクル（Codex + サブエージェント）は冪等性やトランザクションの欠如を拾えたが、層構造違反・命名規則違反・不要な依存注入はどちらも検出できなかった。これらはプロジェクト固有のアーキテクチャルールに基づく指摘であり、レビュアーにプロジェクトコンテキストが足りない。ユーザーレビューの代替にはならない。

トランザクション対応を急いで `db` を直接渡したのは、「動けばいい」思考に引きずられた。既存の `TransactionRunner` パターンを確認していたのに使わなかったのは怠慢。
