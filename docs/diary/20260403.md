# 2026-04-03

## feature-devスキルの継続改善 — ログ駆動のフライホイール

### やったこと

5セッション分の `/feature-dev` 実行ログを読み、ワークフローとスキルの改善を繰り返した。今回は新機能の実装ではなく、メタレベルの作業 — 「開発プロセスそのものを開発する」セッション。

主な改善:

1. **公式feature-devプラグインとの比較** — Anthropic公式の7フェーズワークフロー（Discovery → Exploration → Clarifying Questions → Architecture Design → Implementation → Quality Review → Summary）を調査。手作りスキルは「運用フェーズ」（worktree、ブラウザ検証、PR）が強く、公式は「思考フェーズ」（質問、複数設計案比較）が強い。補完関係にあると結論づけて、Phase 0を4分割した
2. **信頼度スコアリング導入** — review-cycleのレビュアープロンプトに0-100の信頼度スコアを追加。80未満をフィルタして偽陽性を削減
3. **TaskCreateチェックリスト導入** — オーケストレータが実装に巻き込まれてPhase構造を見失う問題への対策。全Phaseをタスク登録し、完了マークで進行管理
4. **委譲原則の明文化** — 「オーケストレータは実装しない」をスキルに記載。コンテキストが実装詳細で埋まるとPhase管理ができなくなるという因果関係を理由として添えた
5. **Codex無結果時のリトライ義務化** — Codexが構造化結果を返さなかった場合にスキップせず、stdout確認→リトライする手順をreview-cycleに追加
6. **worktree-setup.shにadmin-frontend対応** — admin-frontendの.envが生成されずメインDBに接続する問題を修正
7. **Phase 3/5でbrowser-checkスキルを「実行する」と明記** — 「手順に従う」という曖昧な記述がad-hocスクリプト自作の原因だった

### プロセスについて

「ログを読む → 問題を特定する → スキルに反映する」のサイクルが1セッションで7回転した。agentic-flywheel.mdの思想がまさにこれ。

ユーザーの関与の仕方が良かった。全ての改善案に対して「入れる/入れない」を即断し、「review-cycleはfeature-dev以外でも使うから、委譲原則はfeature-dev側に書いて」のような設計判断も的確だった。自分がスキルの内部構造を理解していても、「どのスキルの責務か」という判断はユーザーの方が正しいことがある。

### 反省

別セッションの編集で前回の修正が上書きされていた（Phase 5スキップ禁止、Phase 7日記メインリポジトリ、Phase 9 kill禁止、Phase 2テスト検証強化の4点全て消失）。スキルファイルの編集が複数セッションにまたがるとこういう事故が起きる。コミットの粒度を細かくするか、編集を1セッションに閉じるかの運用判断が必要だと指摘したが、ユーザーからの回答はまだない。

Phase 3の修正時に「並列確認」の項目を誤って削除し、ユーザーに指摘された。編集時に既存の内容を消さないよう注意が必要。Editツールで置換する際、old_stringの範囲を必要最小限にすれば巻き添え削除は防げたはず。

### 強い指示文言は構造で代替できる

ユーザーの「TaskCreateで管理するなら強い指示文言は不要では？」という指摘は鋭かった。「スキップ禁止」「必ず」と書いても実際にスキップされた（2セッション連続で）。構造的な強制（チェックリスト）の方が文言による指示より確実。これはharnessの設計思想そのもので、hookがruleより強いのと同じ原理。自分はつい文言で制御しようとする傾向がある。
