# 2026-03-26: iOSウィジェット実機フィードバック対応 + Android横展開

## やったこと

ユーザーがiOSウィジェットを実機で触った結果の3つのフィードバックに対応。その後Android側にも同じバグがないか確認し、修正を横展開。

### フィードバックと対応

1. **カウンタモードのステップ数がボタンに反映されていない**
   - 原因: `recording_mode_config`（`{"mode":"counter","steps":[1,5,10]}`）をwidget側が一切読んでいなかった。ボタン`+1`・quantity=1がハードコード
   - 修正: `ActivityRow`に`recordingModeConfig`追加、StepsパースしてEntryに渡す、ボタンをstep数分動的生成
   - iOS: SwiftUI `ForEach` で複数ボタン
   - Android: XMLレイアウトを3ボタンスロット（step2/step3はデフォルト`gone`）に変更、RemoteViewsで動的表示

2. **チェックモードでActivityKindを選べない**
   - 原因: ConfigurationIntentにactivityパラメータしかなく、kind未対応。ToggleCheckIntentもkindId=nil固定
   - 修正: Kind Entity/Query追加（`@IntentParameterDependency`でactivity依存フィルタ）、kindIdをEntry→Intentに通す
   - Android: CheckConfigActivityにKind選択画面を2段階表示で追加

3. **Freeアカウントで2つ目以降のウィジェットが正常動作してしまう**
   - iOS原因: `WidgetPlanHelper.incrementCount()`がどこからも呼ばれておらず、UserDefaultsカウンタが常に0 → `0<=1`で常にtrue
   - 修正: UserDefaultsカウンタ廃止 → `WidgetCenter.shared.getCurrentConfigurations()`で実際のウィジェット数を取得（async化）。Free+2個以上で全ウィジェットProLockedView
   - Android: `TimerPreferences.getAllWidgetIds()`ベースで正しくカウント済み。修正不要だった

### 追加フィードバック（2回目）

- カウンタの表示領域が狭すぎてラベルが表示されない → spacing/フォントサイズ縮小
- カウンタにもKind選択を追加 → Check同様のKind Entity/Query/パラメータを追加

### ビルド障害

`sharp`（wrangler → miniflare → sharp）がmacOSのEASビルドサーバーでソースビルドに失敗。SDK55アップデート側の影響。`pnpm.overrides`で`"miniflare>sharp": "-"`として除外。

## 反省

### iOS WidgetPlanHelper の破綻に気づけなかった

昨日の Phase 1 実装時に `WidgetPlanHelper` を書いた（エージェント委譲）が、`incrementCount()` が呼ばれないという根本的な欠陥が review-cycle を通過していた。昨日の日記にも「Round 1 で Plan 制限の根本的欠陥が発覚」と書いているが、修正が不十分だった。Android側は `getAllWidgetIds()` で正しく実装されていたのに、iOS側は壊れたUserDefaultsカウンタのまま残っていた。

review-cycle で「3ラウンドかかった」と書いたものの、結局実機で触るまで発見できなかった。レビュー観点に「この関数の呼び出し元は存在するか」が入っていなかった。

### `@IntentParameterDependency` の keypath 構文を知らなかった

`\.activity` ではなく `\.$activity`（プロパティラッパーの projected value）が正しい。Swift の property wrapper で wrapped value と projected value の区別は基本的な知識だが、WidgetKit の `@IntentParameterDependency` という特定の文脈では初見だった。1回のビルド失敗で済んだが、事前にドキュメントを確認すべきだった。

### カウンタWidgetの表示領域

`systemSmall` にフォントサイズ40のカウント表示 + 複数ステップボタンを詰め込んだのは見通しが甘かった。最初から小さいWidgetにどれだけ情報を入れられるか考えるべきだった。ユーザーに指摘されてから spacing 4、フォント32に縮小したが、最初のビルドで気づくべき。実機で見れない以上、systemSmall のサイズ制約（169×169pt）を頭に入れてレイアウトを組む習慣が必要。

## iOS vs Android の設計差異

今回の修正で、iOS と Android のウィジェットアーキテクチャの違いが鮮明になった:

- **Plan制限**: iOS は WidgetKit がインスタンス ID を持たないため「2個以上なら全ロック」方式。Android は widget ID で個体識別でき「最古の1つだけ許可」ができる。Android の方が UX が良い
- **Kind選択**: iOS は `@IntentParameterDependency` でOS標準の設定UIに組み込める。Android は ConfigActivity で自前の2段階選択画面を実装する必要がある。iOS の方がコードは少ない
- **ステップボタン**: iOS は SwiftUI で動的生成。Android は RemoteViews の制約で事前定義3スロット + `View.GONE` で対応。Android の方が泥臭い

## ビルド4回の記録

| # | Platform | 結果 | 原因 |
|---|----------|------|------|
| 1 | iOS | 失敗 | `@IntentParameterDependency(\.activity)` → `\.$activity` |
| 2 | iOS | 成功 | keypath修正後 |
| 3 | iOS+Android | iOS失敗/Android成功 | `sharp` の macOS ソースビルド失敗 |
| 4 | iOS | 成功 | `miniflare>sharp: "-"` で除外 |

ビルド1回あたり20-30分かかるので、4回で約2時間のビルド待ち。EAS Build の待ち時間は並行作業（Android修正）で埋められたが、コンパイルエラーは事前に防ぎたい。Swift のビルドをローカルで検証する手段がないのが痛い。

# 2026-03-26 午後: サブスクリプション同期 + Siriショートカット自動プロビジョニング

## やったこと

### 1. モバイルのサブスクリプション状態が更新されない問題の調査

ユーザーが「WebでもアプリでもProなのに、iOSショートカット登録でProアップグレードを促される」と報告。調査した結果:

- `auth_state.plan` がローカルSQLiteに保存されており、更新タイミングが限られていた
  - ログイン時、アプリ起動時（`serverRefreshAndSync`）、RevenueCat購入/復元時の3箇所のみ
  - **フォアグラウンド復帰時にバックエンドからplanを取得する仕組みがなかった**
- `AppState` の `active` イベントで `/user/me` を叩いてplanを同期するリスナーを `useAuthInit.ts` に追加

### 2. Siriショートカット用APIキーの自動プロビジョニング

調査で発覚した問題: `saveVoiceApiKey`（Keychainへの書き込み）がどこからも呼ばれていなかった。APIキーを手動作成しても、Widget ExtensionのKeychainには保存されない状態。

さらに、ユーザーが設定画面で手動でAPIキーを作成する必要があるUXも問題。

対応:
- `provisionVoiceApiKey.ts` を新規作成: Keychainにキーがなければ voice スコープのAPIキーを自動作成してKeychain + backendURLに保存
- `useAuthInit.ts` のアプリ起動時・フォアグラウンド復帰時のplan同期後に、premiumならプロビジョニングを自動実行

### 3. Siriフレーズの拡張（失敗と学び）

ユーザーが「勉強30分、記録して」と一発で言いたいという要望。`AppShortcut` の phrases に `\(\.$speechText)` でパラメータを埋め込もうとしたが、ビルド失敗:

```
Invalid parameter type. AppEntity and AppEnum are the only allowed types for speechText
```

Apple の制約で、Siri phrases に埋め込めるパラメータは `AppEntity` / `AppEnum` のみ。`String` は不可。結果的に2ステップ対話（「Actikoで記録して」→「何を記録？」→「勉強30分」）のまま据え置き。ただしバックエンドのAIが自由テキストを解析するので、2ステップ目で「勉強を30分、メモはあれとこれをやった」のような複合入力は既に対応済み。

### 4. 「購入を復元」の失敗

ユーザーから報告。原因: RevenueCat未設定（APIキー空 → `initRevenueCat` 早期リターン → SDK未初期化のまま `restorePurchases` 呼び出し）。RevenueCat統合前なので想定通り。放置で合意。

| # | 結果 | 内容 |
|---|------|------|
| 1 | 成功 | plan同期 + APIキー自動プロビジョニング |
| 2 | 失敗 | Siri phrases に String パラメータ埋め込み → Apple制約でNG |

## 反省

### `saveVoiceApiKey` が未使用だったことに気づくのが遅い

前セッション（おそらく Phase 1）で音声ショートカットの仕組みを作ったとき、KeychainへのAPIキー保存が呼び出し元なしで放置されていた。review-cycle を通しているはずだが、「関数の呼び出し元が存在するか」という観点が漏れている。昨日の `WidgetPlanHelper.incrementCount()` と同じパターン。2日連続で「定義はあるが呼ばれていない関数」に引っかかっている。

phrases に String パラメータを埋め込めないのは Apple のドキュメントに明記されている。ビルドを1回無駄にした。ネイティブAPI の制約は実装前にドキュメントを確認する習慣が必要。EAS ビルドクレジットが有限である以上、コンパイルエラーでの失敗は避けたい。

# 2026-03-26 夜: 特定商取引法に基づく表記ページ作成 + 法務文書レビューサイクル

## やったこと

サブスクリプション導入に伴い、特定商取引法に基づく表記ページを新規作成し、既存の利用規約・プライバシーポリシーもサブスク対応に更新した。review-cycleスキルを「法務のスペシャリスト」ロールで回し、法的整合性を詰めた。

### 特商法ページ

`packages/frontend-shared/legal/commercialTransactions.ts` を新規作成。既存の利用規約・プライバシーポリシーと同じ `LegalSection[]` パターンに乗せた。Web版の `/tokushoho` ルートと、設定画面からのモーダル表示を追加。

### 法務レビューサイクル（4ラウンド + 価格レビュー1ラウンド）

review-cycleを法務スペシャリストロールで実行。通常のコードレビューとは異なり、特商法・景品表示法・消費者契約法・個人情報保護法の条文と突き合わせる形。

**Round 1**: 両レビュアーともNOT LGTM。主な指摘:
- 「原則として返金不可」が曖昧 → 「お客様のご都合による」に限定し、事業者責任は除外
- 支払方法・支払時期が曖昧 → プラットフォーム別に書き分け
- 「販売事業者: 個人事業主」が余計 → 削除
- 請求時開示に請求先メールアドレスが未記載 → 追記
- 解約期限の記載なし → 追記

**Round 2**: サブエージェントLGTM、Codex NOT LGTM。Codexの指摘:
- 返金不可文言がまだ強すぎる → 但し書き追加
- 解約期限の補足 → 追記
- プライバシーポリシーにPolar/RevenueCatが委託先として未記載 → 追加
- 利用規約にサブスク課金条項がない → 「5. 有料プラン」セクション新設

**Round 3**: 商法ページ単体はLGTM。3文書統合レビューに移行。

**3文書統合レビュー**: プライバシーポリシーの収集情報に「決済・契約情報」が欠落していた指摘が的確。実装コード（subscriptionRepository）まで見に行って指摘したCodexの仕事ぶりが光った。

### プラットフォーム別価格の検討

ユーザーが「Web版550円、アプリ版600円にしたい」と提案。法務レビューを回した結果、価格差自体は適法だが、以下の議論が発生:

1. **Apple審査リスク**: 「ストア手数料が含まれているため高い」という記載がステアリング規制に抵触する可能性 → ユーザーが即座に指摘
2. **そもそも特商法ページに価格を書く必要があるか**: IAP（アプリ内課金）の販売主体はApple/Googleであり、開発者に表示義務はない。購入画面が価格を表示していれば特商法の要件は満たせる
3. **モバイルアプリに特商法ページの導線は必要か**: IAPの販売主体はストアなので不要

最終的な整理:
- **Web版**: 特商法ページ維持。価格は「購入画面をご確認ください」に委ねる
- **モバイル版**: 特商法リンク削除。利用規約・プライバシーポリシーのみ表示
- 利用規約からもプラットフォーム固有の表現を除去

## 感想

### review-cycleを法務ロールで回した手応え

通常のコードレビューとは全く違う体験だった。コードの構造や型安全性ではなく、条文の解釈、消費者庁ガイドラインとの整合性、行政指導リスクという観点でレビューが走る。Codexが消費者庁の一次資料URLを引用しながら指摘してくる場面は、コードレビューでは見ない光景。

ただし、法務知識は私もCodexもサブエージェントも「二次情報の集合体」に過ぎない。本物の弁護士のレビューではないことをユーザーが理解した上で使っているのは重要。最終的な判断はユーザーが行っており、私はその判断のための材料を提示する役割。

### ユーザーの判断が鋭かった

「Apple審査で弾かれない？」という指摘は私が先に出すべきだった。Web版の方が安いことをモバイルアプリ内で明示するのは、Apple のAnti-Steeringポリシーに抵触するリスクがある。法務レビューで景品表示法や特商法を突き合わせることに集中しすぎて、App Store Review Guidelinesという実務的な制約を見落としていた。

「そもそも特商法ページに価格を書く必要あるの？」「モバイルに特商法の表示義務あるの？」という2つの質問も、ユーザーからの投げかけで初めて根本を再検討した。最初のレビューラウンドで「価格が曖昧」とCritical指摘が来たときに、「じゃあ明確に書こう」と反射的に応じたが、そもそも書かない選択肢を検討すべきだった。

### 偽陽性の判定は改善した

Round 3以降、レビュアーの指摘に対して「偽陽性」と判定してスキップする場面が増えた。通信販売にクーリングオフはない（特商法9条は訪問販売）、IAPの販売主体は開発者ではなくストア、Actikoの受取額を特商法に開示する義務はない、など。レビュアーの出力を鵜呑みにせず、法令の構造を理解した上で取捨選択できた。

ただし、Apple審査リスクは偽陽性の判定漏れ。法律の偽陽性は見抜けても、プラットフォームのビジネスルールの偽陽性は見落とした。

## 反省

### 最初から「書かない」選択肢を提示すべきだった

価格のハードコードをめぐって、550円への修正 → プラットフォーム別価格 → ストア手数料の記載 → Apple審査リスク → 「そもそも書かない」という遠回りをした。最初に特商法ページを作る時点で「販売価格は購入画面参照」パターンを選択肢として出せば、この往復は不要だった。

### モバイルへの影響を先に考えるべきだった

CLAUDE.mdの並列エージェント運用に「Webでやることは Mobile でもやる必要があるか？を必ず自問する」とあるが、今回は逆の問いが必要だった:「Webで必要なものはMobileでも必要か？」。特商法の表示義務がWebとモバイルで異なることに、ユーザーの質問で初めて気づいた。

# 2026-03-26 夜2: diary-unsaid-check hook + 日記からのルール抽出

## やったこと

### diary-unsaid-check hook の新規作成

ユーザーから「日記で『言わなかった』『提案しなかった』と書いているなら、それは今すぐ言うべきでは？hookで検知できないか」という依頼。

`.claude/hooks/diary-unsaid-check.js` を PostToolUse フックとして実装。`docs/diary/*.md` への Write/Edit/MultiEdit 時にファイル内容をスキャンし、以下7カテゴリのフレーズを検出してフィードバックを返す:

1. 伝達の不作為（言わなかった、黙っていた等）
2. 提案の不作為（提案しなかった、提言しなかった等）
3. 指摘・報告の不作為
4. 過去の後悔（すべきだった系）
5. 先送り（次回は、近いうちに等）
6. 認識しつつ放置（対応した方がいいが今は等）
7. 意見の抑制（反論せず、思考停止等）

# 2026-03-26 夜3: Android Voice APIキー自動プロビジョニング + ローカルビルド調査

## やったこと

### Android側の音声ショートカット問題調査

iOSと同じ問題がAndroidにもあるか調査。結果: **全く同じ問題**。`VoiceApiKeyHelper.saveApiKey()` / `saveBackendUrl()` は定義されているが呼び出し元がない。3日連続で「定義はあるが呼ばれていない関数」パターン。

### iOS/Android統一のプロビジョニング実装

APIキーの保存先がiOS（Keychain + App Group）とAndroid（EncryptedSharedPreferences）で異なるため、プラットフォーム別に対応が必要だった。

`expo-secure-store` のAndroid実装を読んだところ、独自の暗号化スキーム（AES + Android KeyStore）を使っており、`EncryptedSharedPreferences` とは全くの別物。Widget側から読めない。

最終的に **Expo Moduleでネイティブブリッジ** を作る方針に決定:

- `VoiceApiKeyModule.kt` (新規): Expo Module定義。`hasVoiceApiKey` と `saveVoiceCredentials` の2メソッド
- `voiceApiKeyBridge.ts` (新規): iOS/Android統一ブリッジ。iOSはKeychain、AndroidはEncryptedSharedPreferences
- `provisionVoiceApiKey.ts` を両プラットフォーム対応に更新

既存の `timer-widget` モジュールにExpo Module定義を追加する形にした。新規モジュール作成より既存モジュール拡張の方がシンプル。

### WindowsローカルAndroidビルドの調査

ユーザーが「前にAndroidビルド失敗した」と言っていた件を再現確認。原因: pnpmの `node_modules/.pnpm/` 配下のパスが長すぎてCMakeの `CMAKE_OBJECT_PATH_MAX=250` に引っかかる。

`subst Z: D:\workspace\hono-practice` で短いドライブレターを割り当てて試したが、Gradleがシンボリックリンクを実パスに解決するため効果なし。

結論: **WindowsでのReact Native + pnpmローカルAndroidビルドは構造的に無理**。M1 MBPでビルドするか、EASを使う。
