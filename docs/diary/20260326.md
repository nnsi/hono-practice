# 2026-03-26: iOSウィジェット実機フィードバック対応 + Android横展開

## やったこと

ユーザーがiOSウィジェットを実機で触った結果の3つのフィードバックに対応。その後Android側にも同じバグがないか確認し、修正を横展開。

### フィードバックと対応

1. **カウンタモードのステップ数がボタンに反映されていない**
   - 原因: `recording_mode_config`（`{"mode":"counter","steps":[1,5,10]}`）をwidget側が一切読んでいなかった。ボタン`+1`・quantity=1がハードコード
   - 修正: `ActivityRow`に`recordingModeConfig`追加、StepsパースしてEntryに渡す、ボタンをstep数分動的生成
   - iOS: SwiftUI `ForEach` で複数ボタン
   - Android: XMLレイアウトを3ボタンスロット（step2/step3はデフォルト`gone`）に変更、RemoteViewsで動的表示

2. **チェックモードでActivityKindを選べない**
   - 原因: ConfigurationIntentにactivityパラメータしかなく、kind未対応。ToggleCheckIntentもkindId=nil固定
   - 修正: Kind Entity/Query追加（`@IntentParameterDependency`でactivity依存フィルタ）、kindIdをEntry→Intentに通す
   - Android: CheckConfigActivityにKind選択画面を2段階表示で追加

3. **Freeアカウントで2つ目以降のウィジェットが正常動作してしまう**
   - iOS原因: `WidgetPlanHelper.incrementCount()`がどこからも呼ばれておらず、UserDefaultsカウンタが常に0 → `0<=1`で常にtrue
   - 修正: UserDefaultsカウンタ廃止 → `WidgetCenter.shared.getCurrentConfigurations()`で実際のウィジェット数を取得（async化）。Free+2個以上で全ウィジェットProLockedView
   - Android: `TimerPreferences.getAllWidgetIds()`ベースで正しくカウント済み。修正不要だった

### 追加フィードバック（2回目）

- カウンタの表示領域が狭すぎてラベルが表示されない → spacing/フォントサイズ縮小
- カウンタにもKind選択を追加 → Check同様のKind Entity/Query/パラメータを追加

### ビルド障害

`sharp`（wrangler → miniflare → sharp）がmacOSのEASビルドサーバーでソースビルドに失敗。SDK55アップデート側の影響。`pnpm.overrides`で`"miniflare>sharp": "-"`として除外。

## 反省

### iOS WidgetPlanHelper の破綻に気づけなかった

昨日の Phase 1 実装時に `WidgetPlanHelper` を書いた（エージェント委譲）が、`incrementCount()` が呼ばれないという根本的な欠陥が review-cycle を通過していた。昨日の日記にも「Round 1 で Plan 制限の根本的欠陥が発覚」と書いているが、修正が不十分だった。Android側は `getAllWidgetIds()` で正しく実装されていたのに、iOS側は壊れたUserDefaultsカウンタのまま残っていた。

review-cycle で「3ラウンドかかった」と書いたものの、結局実機で触るまで発見できなかった。レビュー観点に「この関数の呼び出し元は存在するか」が入っていなかった。

### `@IntentParameterDependency` の keypath 構文を知らなかった

`\.activity` ではなく `\.$activity`（プロパティラッパーの projected value）が正しい。Swift の property wrapper で wrapped value と projected value の区別は基本的な知識だが、WidgetKit の `@IntentParameterDependency` という特定の文脈では初見だった。1回のビルド失敗で済んだが、事前にドキュメントを確認すべきだった。

### カウンタWidgetの表示領域

`systemSmall` にフォントサイズ40のカウント表示 + 複数ステップボタンを詰め込んだのは見通しが甘かった。最初から小さいWidgetにどれだけ情報を入れられるか考えるべきだった。ユーザーに指摘されてから spacing 4、フォント32に縮小したが、最初のビルドで気づくべき。実機で見れない以上、systemSmall のサイズ制約（169×169pt）を頭に入れてレイアウトを組む習慣が必要。

## iOS vs Android の設計差異

今回の修正で、iOS と Android のウィジェットアーキテクチャの違いが鮮明になった:

- **Plan制限**: iOS は WidgetKit がインスタンス ID を持たないため「2個以上なら全ロック」方式。Android は widget ID で個体識別でき「最古の1つだけ許可」ができる。Android の方が UX が良い
- **Kind選択**: iOS は `@IntentParameterDependency` でOS標準の設定UIに組み込める。Android は ConfigActivity で自前の2段階選択画面を実装する必要がある。iOS の方がコードは少ない
- **ステップボタン**: iOS は SwiftUI で動的生成。Android は RemoteViews の制約で事前定義3スロット + `View.GONE` で対応。Android の方が泥臭い

## ビルド4回の記録

| # | Platform | 結果 | 原因 |
|---|----------|------|------|
| 1 | iOS | 失敗 | `@IntentParameterDependency(\.activity)` → `\.$activity` |
| 2 | iOS | 成功 | keypath修正後 |
| 3 | iOS+Android | iOS失敗/Android成功 | `sharp` の macOS ソースビルド失敗 |
| 4 | iOS | 成功 | `miniflare>sharp: "-"` で除外 |

ビルド1回あたり20-30分かかるので、4回で約2時間のビルド待ち。EAS Build の待ち時間は並行作業（Android修正）で埋められたが、コンパイルエラーは事前に防ぎたい。Swift のビルドをローカルで検証する手段がないのが痛い。
