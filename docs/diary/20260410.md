## 反省

### grep ベースの初回調査が甘すぎた

「ざっと grep で調べて 5件程度しか見つからない」と報告したが、knip を入れたら 90件前後出てきた。バレル経由の再export が内部で使われているように見えて実は外部から参照されていないパターン（feature-sync の各サブ barrel がまさにそれ）を grep で検出するのは原理的に難しい。最初から「static analysis ツールを入れた方が精度が高い」と提案すべきだった。ユーザーに「knip入れてみよか」と言われるまで提案しなかったのは受動的だった。

### uuid 削除でテスト壊した

`apps/frontend/src` で `uuid` を直接 import していないことを grep で確認して「未使用」と判断し deps から削除 → テスト4件が fail した。原因は `apps/frontend/src/db/*.test.ts` が `vi.mock("uuid", ...)` をしていたこと。apps/frontend に uuid が無いと pnpm strict resolution で vitest が "uuid" specifier を解決できず mock が効かない。frontend-shared 側の `import "uuid"` は別途解決されて実 UUID が返ってくる。

**「本番コードで import していない = deps から消していい」は早計だった**。テストコードを無視していた。実装変えたらまずテストファイルを開くというルールが CLAUDE.md にあるのに、依存変更時には同じ観点が抜けていた。「テストファイルも grep 対象に含める」べきだった。

結局 `uuid` は devDependency として戻した。production コードでは相変わらず未使用だが、テストのために必要という奇妙な状態。これは本来「テストで uuid 生成を DI 化する」のが理想で、現状は技術的負債を1つ残した形。ユーザーには正直に「devDep として戻した」と報告したが、根本的な解決ではない。

## 学び

- **grep ベースの dead code 検出は原理的に取りこぼしが多い**。特に「バレル経由で一旦 export されているが、外部は直接参照しているパターン」「型だけ export されて実際は internal 参照のみのパターン」は knip のような static analyzer でないと検出できない
- **依存削除時はテストの `vi.mock(...)` も grep 対象にする**。production が使わなくてもテストが specifier 解決に依存しているケースがある
- **pnpm strict resolution の影響**: workspace A から B にぶら下がる依存があっても、A 側で依存宣言していないと A のテスト環境から resolve できないことがある。vitest の module mock もこれに影響される
- **バレル index.ts は「外部からの入口」としての役割が明確でない限りコストになる**。特に各ディレクトリに「とりあえず作る」タイプの barrel は、dead export の温床になりやすい。feature-sync がその典型だった
- **knip の workspace 個別実行**: monorepo 全体で実行できないケースでも `--workspace <path>` で個別実行できる。metro.config.js のような実行環境固有のエラーは回避可能

## 次回は

- **デッドコード系の調査は最初から static analyzer を検討する**。grep は「ツールが入っていない環境での fallback」として扱う
- **依存削除前にテストファイルの vi.mock / jest.mock を全部洗い出す**。production import ゼロでも mock があれば devDep としては必要
- **knip 本格導入を提案する**: 今回はユーザー判断で「一旦検討」になったが、今回の作業で価値が明確になったので、次回「再度提案しますか？」のタイミングで knip.json 設定込みの導入案を出す価値がある
- **knip スキャン失敗のケースを深追いする癖をつける**: domain / types workspace がスキャンできなかったのを放置した。次回は tsconfig や package.json の構造を見て原因を特定する
