## いい判断だったと思うこと

### 「Notion寄り」と推した自分の案を、Codex の指摘で軌道修正できた

最初の私の推し: 「ビュー切替（最近順 ⇔ Activity別Board）+ List密度モード切替」だった。Codex が「**Board は note 数が少ないユーザーに空間効率悪く、activity 多いユーザーに横断しづらい**」「**ビュー切替の2軸化（順序×密度）は組み合わせ爆発で UI が重くなる**」「**検索が最も効く**」と指摘。

これは正しかった。私の最初の比較で「2(日付グルーピング) + 1(検索/フィルタ) のセットが費用対効果高い」と一度言っておきながら、3rd party (Obsidian/Notion) の事例調査で **Notion の機能リッチさに引っ張られて Board 案を推してしまった**。Codex の指摘は私の最初の感覚を後押しする形だった。

セカンドオピニオンの価値は「自分の中で揺らいだ判断を、外部の視点でリセットさせる」ところにある。盲目的に Codex に従うのではなく、「最初の自分の感覚と一致した」という確認材料として使えたのが良かった。

### Explore 委譲が効いた

ファクトリパターン (`createUseTasksPage`)、taskGrouping.test.ts のパターン、i18n キーの場所、dayjs locale の初期化状態 ... 全部 Explore 1 回で網羅できた。ルール（parallel-agents.md）の Explore 出力テンプレートに沿った指示で、影響分析 + 既存パターン + 依存 + リスクの構造で返ってきた。

特に「**Mobile 側は dayjs locale 既に対応済み**」を Mobile実装エージェントが Read で確認 → 親への報告に明記してくれた点は良い。Explore で「**未初期化なら追加するが、コストが大きそうならスキップしてよい**」と逃げ道を作ったプロンプト設計が機能した。実際は既に対応済みで何もしなくて良かった。

### Mobile エージェントの「200 行制限のため testIds.ts 分割」を refactor 越権としなかった

エージェントが `testIds.ts` から `cardTestIds.ts` に `buildTaskCardTestIds` を分離し、`TaskCard.tsx` の import を更新するという、指示外の側面修正をしてきた。CLAUDE.md「**bug fix doesn't need surrounding cleanup**」と矛盾する変更に見えた。

git diff を確認して: 元 192 行 → testID 追加で 204 行 → 200 行ルール超過。確かに分割は **必要** だった。エージェントの判断は正しい。短絡的に revert していたら 200 行ルール違反のまま commit していた。エージェントの side change は **理由を確認してから判定する** という運用が安定してきた。

### Playwright で seed → IndexedDB クリア → 再 sync まで一気にやった

最初の login 後、9 件の seed note が UI に出てこなかった。原因は frontend がオフラインファースト (Dexie + sync since) で、login 直後の lastSync timestamp が seed 時刻より新しかったため。「**しばらく待てば pull される**」と数分待つこともできたが、IndexedDB を `indexedDB.databases() → deleteDatabase()` で全削除 → reload で強制 full sync させた。これで 9 件全部表示。

オフラインファーストアプリのデバッグでは「**クライアント DB を即座にクリアできるかどうか**」が時間効率を決める。`localStorage.clear()` だけでは Dexie は消えない。`indexedDB.databases()` API は Chrome 系で使えるので playwright の `browser_evaluate` から叩ける。

## 自己批判

### Maestro flow の `assertVisible: id: notes.filterAll` で 1 ラウンド失敗した

実装は「activity 紐付き note が 1 件もないとき filter 全体を隠す」という妥当な仕様。flow を書く時に、create-note flow が **activity 未指定で note を作成する** ことを忘れていて、`filterAll` を visible で assert → 1 ラウンド目失敗。

これ自体は flow の即時修正で済んだが、根本的な問題は「**flow を書く時に被テスト UI の表示条件を確認せずに assertion を入れた**」こと。実装側の `{filterActivities.length > 0 && (<NotesActivityFilter />)}` を **Read していれば** 一瞬で気付けた。NotesActivityFilter.tsx は読んだけど、親の `filterActivities.length > 0` ガードを読み飛ばしていた。

教訓: **Maestro flow の assertion を書く前に、対象 UI の表示条件を実装側で確認する**。今後は flow 書き換えのループを 1 回減らせる。

### import 経路の不統一（Web は直接、Mobile はバレル）を最後まで放置した

Web 実装エージェント: `@packages/frontend-shared/hooks/useNoteListFilter` から直接 import
Mobile 実装エージェント: `@packages/frontend-shared/hooks` のバレル経由

動作には影響しないが一貫性が悪い。最後の報告で「**揃えるならバレル経由が標準**」と user に提示したが、user が「commit していい」と言ったので結局そのまま commit した。本来は **エージェント起動時のプロンプトで import 規約を明示** すべきだった。「import経路は既存ファイルの import を見て合わせる」とだけ書いたから、それぞれが見た既存ファイルが違うとブレる。「**バレル経由で import すること**」と明示すべき。

並列エージェント運用ルール (parallel-agents.md) に「**import 経路の規約をプロンプトに明示する**」を追加提案するか検討。

### user の「mobile も同じUIでいけそう？」に対して、すぐに案を出さず、まずファイルを読みに行った

これは効率的だったとも言えるが、user は「同じでいけるか」を 30 秒で判断したかった可能性がある。私は Mobile 側の NotesPage.tsx と CLAUDE.md を読んでから「**ロジックは共通化、View 層だけ Web は常時表示 / Mobile はアイコン展開**」と返した。

調べてから返した内容は適切だったと思うが、「30 秒で結論」を期待されていた場合は遅い。user の問いの粒度を最初に確認すべきだった。「**ファイル読んで詳細に判断するか、感覚で答えるか、どっちがいい？**」と。response-style.md の「短く」と「正確に」のバランスは難しい。

## メタ: claude-code-guide や agentic-flywheel への提案

### Maestro flow 作成のチェックリスト案

flow 書く前に最低限確認:
1. 対象 UI の **表示条件** をコンポーネントで Read（条件付きレンダリング `{cond && (<X />)}` を見落とすと assertion がいきなり落ちる）
2. testID が **本当に焼き込まれているか** を `mobileTestIds` レジストリと実装側両方で確認
3. flow を書いたら **1 回目を必ず流す**（書きっぱなしで commit せず、最低 1 回は緑を確認）

これを書きに行く価値はある。`apps/mobile/CLAUDE.md` の Maestro セクションに足すべきか user に提案する候補。

### Codex セカンドオピニオンを「自分の判断揺らぎ検出器」として使うパターン

明示的に書きたい: 「自分の中で 2 つの案で揺らいでいる時こそ Codex を呼ぶ」。今日の note 一覧の件はまさにこれ。一発で解決する単純タスクには Codex は不要。

ただ、これを rule に書くほど普遍的かは微妙。`response-style.md` の「中立・客観的」と地続きの話なので、たぶん新規 rule にせず既存に「**重要な設計判断で揺らぎを感じたら外部視点を取る**」一文足すくらいが適温。
