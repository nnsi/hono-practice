# 2026-03-20: iOS/Androidキーボード回避の統一 + モーダルfooter固定化

## やったこと

### KeyboardAvoidingView behavior統一
- ユーザーから「iOSでキーボードが出ると表示領域が狭くなる、Androidみたいに被せたい」という相談
- 最初は「被せる方向」で選択肢を提示したが、ユーザーが「Androidだと下の入力欄が隠れるよね」と指摘
- よく考えたらその通りで、「被せて隠れない」は物理的に不可能。結局「キーボード分だけ押し上げる」か「被せて隠れる」のトレードオフ
- 提案を転換: 問題があるのはむしろAndroid側（被さって見えない）で、iOS側の`padding`挙動に揃えるべきだと提言
- ユーザーも納得し、`behavior`を`"padding"`に統一。`Platform`のimportも不要になり削除

### モーダルfooter固定化
- OTA配信後、ユーザーが「保存ボタンがキーボードで隠れて押せない」という本質的な問題を特定
- 原因: 保存ボタンがScrollView内にあるため、モーダルが縮むとスクロールしないと到達できない
- `ModalOverlay`に`footer`プロップを追加し、ScrollViewの外（モーダルカード下端）に`border-t`付きで固定配置
- 全8ダイアログのアクションボタンをfooterに移動:
  - CreateActivityDialog, EditActivityDialog, EditLogDialog, CreateGoalDialog
  - TaskCreateDialog, TaskEditDialog, CreateApiKeyDialog, DeleteConfirmDialog

## 振り返り

### 良かった点
- キーボード回避の議論で、最初の「iOSをAndroidに合わせる」から「Androidの方が問題」と視点を転換できた。ユーザーの要望をそのまま実装するのではなく、根本問題を指摘できた
- footer固定化はModalOverlayの1箇所の変更で全ダイアログに波及するクリーンな設計。共通化の運用価値がそのまま活きた

### 反省点
- 最初の提案時点で「被せるけど隠れない」が物理的に無理だと気づくべきだった。選択肢Cを「ScrollViewで自動スクロール」と書いたが、ユーザーに「それでも解決しなくない？」と突っ込まれるまで矛盾に気づいていなかった。提案前にもう少し深く考えるべきだった

# 2026-03-20 (2): Mobile Google OAuth修正

## やったこと

### Google OAuth認証エラーの診断と修正
- ユーザーから「Google認証でポリシー違反エラーが出る」という報告
- エラー: "You can't sign in to this app because it doesn't comply with Google's OAuth 2.0 policy for keeping apps secure"

#### 最初の誤診: implicit flow → auth code + PKCE
- 最初は`responseType: IdToken`（implicit flow）が原因と判断し、auth code + PKCEに変更してOTA配信
- しかし解決せず。**フロー方式ではなく、リダイレクトURIの形式自体がブロックされていた**

#### 真の原因: Web Application型クライアントでカスタムURIスキーム
- `expo-auth-session`の`makeRedirectUri()`が`actiko://`（カスタムスキーム）を生成
- Google は2022年以降、Web Application型OAuthクライアントでカスタムURIスキームのリダイレクトをブロック
- クライアントタイプがWeb Applicationであることをユーザーに確認して確定

#### 修正方針の検討
- 方式A: `expo-auth-session/providers/google`（プラットフォーム別clientId）
- 方式B: `@react-native-google-signin/google-signin`（ネイティブSDK）
- ユーザーが「React Native Webが壊れないか？」と質問 → 方式BはネイティブSDK依存でWeb非対応と回答 → 方式Aに決定

#### 実装（フロントエンド + バックエンド + CI/CD）
1. Google ConsoleでiOS/Android用OAuthクライアントID新規作成（ユーザーが実施）
2. モバイル: `expo-auth-session/providers/google`に切り替え。プラットフォーム別clientId対応
3. バックエンド: `googleVerify`のaudience検証を`string | string[]`対応に。`OAuthVerify`型、usecase、handler、routeの全チェーンを更新
4. CI/CD: `deploy.yml`に`GOOGLE_OAUTH_CLIENT_ID_ANDROID`/`_IOS`のwrangler secret put追加
5. EAS env追加、新ビルド → iOS確認でログイン成功

## 振り返り

### 良かった点
- バックエンドのaudience検証問題に事前に気づけた。モバイルだけ直してデプロイしていたら、プラットフォーム別clientIdのトークンがバックエンドで弾かれて「直ったのに動かない」になっていた
- 方式の選択でユーザーに判断材料を正直に提示できた。React Native Webの懸念も即答できた

### 反省点
- 最初の診断が甘かった。implicit flow → auth codeへの変更を「これで直るはず」とOTA配信までしてしまった。**カスタムURIスキーム自体がブロックされている**という可能性を最初の段階で検討すべきだった。結果的にユーザーに「まだ同じエラー出てる」と言わせてしまった
- Googleの2022年ポリシー変更は知識としてあったのに、最初の調査で`makeRedirectUri()`の出力を確認せず、フロー方式だけに着目してしまった。原因の仮説を立てるときに複数の可能性を並行で検証する癖をつけたい

### 学び
- Google OAuthのモバイル対応は「クライアントタイプ × リダイレクトURI × フロー方式」の3軸で考える必要がある。1軸だけ見て診断すると誤診する
- `expo-auth-session/providers/google`はプラットフォーム別clientIdの自動選択、PKCE、自動コード交換を全部やってくれる。生の`useAuthRequest`を使う理由はほぼない

# 2026-03-20 (3): Android Google OAuth — 長い戦い

## やったこと

前回のセッション(2)でiOSは解決したが、Androidは別の問題が連鎖して残っていた。最終的にAndroid専用のバックエンドリレー方式で解決。

### 問題1: Android OAuthクライアントがブラウザフローに非対応
- iOS用クライアントIDでiOSは成功。同じノリでAndroid用クライアントIDを作成→「Actikoが無効なリクエストを送信」エラー
- 原因: **Google ConsoleのAndroid型クライアントはネイティブSDK専用**で、expo-auth-sessionのブラウザベースフローに対応していない
- 方式A（バックエンドリレー）/ B（ネイティブSDK）/ C（Expo proxy）の選択肢をユーザーに提示→方式A採択

### 問題2: バックエンドリレーのリダイレクト後、ログイン画面に戻される
- `GET /auth/google/callback` → `actiko://oauthredirect?code=...` のリダイレクト自体は成功
- しかしGoogleログイン画面が一瞬見えて即座にTOPに戻る現象
- 自力で原因特定できず、Codexに調査を委譲
- **Codexが根本原因を特定**: `_layout.tsx`の認証ガードが `oauthredirect` ルートを未認証と判定し、useEffectでコード交換が走る前に `/(auth)/login` にリダイレクトしていた
- 修正: `inAuthGroup`の判定に`"oauthredirect"`を追加（1行修正）

### 問題3: client_secret is missing
- Androidではweb clientIdを使うため、コード交換にclient_secretが必須
- モバイルにsecretを埋め込むのはNG → バックエンドに`POST /auth/google/exchange`エンドポイントを新設
- バックエンド側でclient_secret付きのコード交換を実行し、id_token取得→既存ログインフローに合流
- `GOOGLE_OAUTH_CLIENT_SECRET`をCloudflare Workers secretとdeploy.ymlに追加

### 問題4: oauthredirectルートからログイン完了できない
- `googleLogin()`はid_tokenを受けてPOST /auth/googleを呼ぶ設計。oauthredirectからは使えない
- `loginWithUserCheck`を`completeLogin`としてAuthContextに公開
- oauthredirectルートで: exchange → setToken → apiGetMe → completeLogin の流れで完結

### 最終的なAndroid Google OAuthフロー
1. LoginForm → `setOAuthPending`(codeVerifier保存) → `googlePromptAsync()`
2. Chrome Custom Tab → Google認証 → `https://api.actiko.app/auth/google/callback`
3. バックエンド 302 → `actiko://oauthredirect?code=...`
4. `app/oauthredirect.tsx` → `POST /auth/google/exchange` (バックエンドがclient_secret付きでコード交換)
5. setToken → apiGetMe → completeLogin → メイン画面遷移

## 振り返り

### 良かった点
- Codexへの委譲判断。自分が「Chromeが一瞬開いて戻る」の原因を絞り込めなかった時に、expo-web-browserの内部実装まで読み込んで`_layout.tsx`の認証ガードが根本原因だと特定してくれた。CLAUDE.mdの「自分が詰まった問題も別エージェントに委譲してみる」が活きた
- ユーザーの「refreshTokenをJSONに含むのはセキュリティ的に問題」という指摘に即座に同意してrevertできた。Codexの指摘を鵜呑みにしかけたが、ユーザーの方がドメイン知識を持っていた

### 反省点
- **試行錯誤が多すぎた**。implicit flow修正 → プラットフォーム別clientId → バックエンドリレー → auth guard修正 → server-side exchange と5段階の修正を要した。各段階でOTA配信→ユーザーに確認→次の問題発覚のループ。もっと事前に全体フローをシミュレーションしてから実装に入るべきだった
- **expo-auth-sessionのAndroid内部実装を理解せずに進めた**。`openAuthSessionAsync`がAndroidではpolyfillで`Linking.addEventListener`を使うこと、`_waitForRedirectAsync`がURLプレフィックスマッチすること、AppState変更で`dismiss`になること——これらを最初に調べていれば「バックエンドリレーだけでは不十分」「auth guardが問題」と一発で分かったはず
- **Codexの報告を検証せずに適用しかけた**。refreshTokenの件でユーザーに突っ込まれたのは恥ずかしい。エージェントの出力は信用しない、というCLAUDE.mdのルールを自分自身が破っていた

### 学び
- Android + expo-auth-session + Google OAuth は「iOS型クライアント = カスタムスキームOK」「Android型クライアント = ネイティブSDK専用」「Web型クライアント = カスタムスキームNG + client_secret必須」という非対称性がある。iOSで動いたからAndroidも同じ方式で動くとは限らない
- `_layout.tsx`の認証ガードはOAuthコールバックのような外部からのディープリンクを考慮して設計する必要がある。新しいルートレベルのページを追加する際はガードの条件を確認すべき
- **デバッグUI（エラー内容を画面表示する仕組み）は最初から入れるべきだった**。「client_secret is missing」が見えた瞬間に問題が明確になった。console.logだけではモバイル実機のデバッグは困難
