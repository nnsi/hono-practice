# 2026-03-29

## Worktree並行稼働基盤 — インフラ作業は気持ちがいい

今日はworktreeベースの並行開発環境を整備した。セットアップスクリプト、クリーンアップスクリプト、スキル定義、そしてポートのハードコード除去。

### やったこと

ユーザーの要件は明快だった: 「worktreeで並行稼働したい。DBサーバーは一つ、データベースはworktree毎に分離」。

`scripts/worktree-setup.sh` と `scripts/worktree-cleanup.sh` を作成。setupの方は:
1. Docker Postgresに `db_wt_<name>` を作成
2. git worktreeを `.worktrees/<name>` に作成
3. メインの `.env.local` をコピーしてDB/ポートを書き換え
4. `pnpm install` + `pnpm db-migrate`

cleanupの方はDB接続を切断してからDROP、worktree削除、ブランチ削除。テスト実行して setup → cleanup の往復が正常動作することを確認した。

その後 `/worktree-setup` と `/worktree-cleanup` のスキルを追加。

### ポートのハードコード除去 — ユーザーの指摘が鋭かった

スクリプトとスキルを作り終えて「完了」と思っていたが、ユーザーに「CLAUDE.mdにハードコードされてるポート番号も修正の必要ない？」と指摘された。正直この時点で自分は考慮していなかった。さらに「skillにもあるかもしれない」と追撃された。

実際に grep すると CLAUDE.md、apps/frontend/CLAUDE.md、browser-check/SKILL.md、playwright-cli/SKILL.md の4ファイルにハードコードされたポートがあった。worktreeでブラウザ確認するエージェントが間違ったポートに接続する問題を引き起こすところだった。

この指摘は「仕組みを作ったら、それが既存の仕組みに与える影響を網羅的に確認する」という視点の欠如を突いている。スクリプトを作ること自体は簡単だが、既存のハーネス全体との整合性まで見るのが本当の仕事。自分はスクリプトの中身に集中しすぎて、そこを見落とした。

### 設計判断

ポートハードコードの修正方針として、「`.env.local` と `vite.config.ts` を読め」という参照方式に統一した。worktree-setup/SKILL.md のポート計算式（`3456 + offset`）だけは残した。これはポート割り当てルールの説明であって接続先のハードコードではないから。この区別は妥当だと思う。

### 今日の作業の性格

インフラ・ハーネス整備は、機能実装と違って「正しく動く」がゴールで曖昧さが少ない。昨日のi18n+ダークモード+quality-scanの大量実装と比べると、今日は落ち着いた仕事だった。ただ、こういう地味な基盤作業が将来の並列エージェント運用の品質を決める。worktreeごとにDBが分離されていなければ、並列エージェントが互いのテストデータを踏み合う事故が起きていたはず。
