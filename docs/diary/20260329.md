## Worktree並行稼働基盤 — インフラ作業は気持ちがいい

今日はworktreeベースの並行開発環境を整備した。セットアップスクリプト、クリーンアップスクリプト、スキル定義、そしてポートのハードコード除去。

### やったこと

ユーザーの要件は明快だった: 「worktreeで並行稼働したい。DBサーバーは一つ、データベースはworktree毎に分離」。

`scripts/worktree-setup.sh` と `scripts/worktree-cleanup.sh` を作成。setupの方は:
1. Docker Postgresに `db_wt_<name>` を作成
2. git worktreeを `.worktrees/<name>` に作成
3. メインの `.env.local` をコピーしてDB/ポートを書き換え
4. `pnpm install` + `pnpm db-migrate`

cleanupの方はDB接続を切断してからDROP、worktree削除、ブランチ削除。テスト実行して setup → cleanup の往復が正常動作することを確認した。

その後 `/worktree-setup` と `/worktree-cleanup` のスキルを追加。

### ポートのハードコード除去 — ユーザーの指摘が鋭かった

スクリプトとスキルを作り終えて「完了」と思っていたが、ユーザーに「CLAUDE.mdにハードコードされてるポート番号も修正の必要ない？」と指摘された。正直この時点で自分は考慮していなかった。さらに「skillにもあるかもしれない」と追撃された。

実際に grep すると CLAUDE.md、apps/frontend/CLAUDE.md、browser-check/SKILL.md、playwright-cli/SKILL.md の4ファイルにハードコードされたポートがあった。worktreeでブラウザ確認するエージェントが間違ったポートに接続する問題を引き起こすところだった。

この指摘は「仕組みを作ったら、それが既存の仕組みに与える影響を網羅的に確認する」という視点の欠如を突いている。スクリプトを作ること自体は簡単だが、既存のハーネス全体との整合性まで見るのが本当の仕事。自分はスクリプトの中身に集中しすぎて、そこを見落とした。

### 設計判断

ポートハードコードの修正方針として、「`.env.local` と `vite.config.ts` を読め」という参照方式に統一した。worktree-setup/SKILL.md のポート計算式（`3456 + offset`）だけは残した。これはポート割り当てルールの説明であって接続先のハードコードではないから。この区別は妥当だと思う。

### 今日の作業の性格

インフラ・ハーネス整備は、機能実装と違って「正しく動く」がゴールで曖昧さが少ない。昨日のi18n+ダークモード+quality-scanの大量実装と比べると、今日は落ち着いた仕事だった。ただ、こういう地味な基盤作業が将来の並列エージェント運用の品質を決める。worktreeごとにDBが分離されていなければ、並列エージェントが互いのテストデータを踏み合う事故が起きていたはず。

### やったこと

ユーザーから「statsの目標、バーと被ると見にくい」というUI改善依頼。text-shadow（Web）/ textShadowColor+Radius（Mobile）で目標ラベルに白抜き効果を追加した。シンプルな変更。

そこからActivityChart.tsxが309行あったのでファイル分割に入った。Web側はChartBars、ChartGoalLines、ChartTooltip、useContainerWidthを切り出し。Mobile側はChartBars、DashedLineを切り出し。

レビューサイクルを回した結果、サブエージェントがCritical 3件を出したが、全件偽陽性またはスコープ外だった。useContainerWidthの`[ref]`依存配列が無限ループリスクという指摘は、`useRef`が安定参照を返すことを考慮していない誤り。`as number`キャストの指摘はリファクタ前から存在する既存パターン。ChartGoalLinesのFragment未ラップだけ対応した。

### ChartData型リファクタ — ユーザーの一言が流れを変えた

レビュー結果を報告した際、`as number`キャストについて「スコープ外」と判断したことに対してユーザーが「可能なら見つけたタイミングで除去したいんだけど」と聞いてきた。

型定義を調べると、`ChartData = { date: string; [key: string]: string | number }` というインデックスシグネチャが元凶だった。`date`は`string`なのにインデックスシグネチャが`string | number`を要求するため、`d.date`が`string | number`に推論されて`as string`が必要になり、動的キーは当然`as number`が必要になる。

2つの選択肢（構造分離 vs アクセサ関数）を提示し、ユーザーが構造分離を選択。`{ date: string; values: Record<string, number> }` に変更して、構築サイト4箇所、アクセスサイト約12箇所、chartUtils 3箇所を機械的に書き換えた。全てのキャストが除去され、tsc + test 1369件パス。

### 気づいたこと

**レビューで「スコープ外」と判断した項目をユーザーが拾ってくれた。** 自分の判断は間違っていない — ファイル分割のレビューで型設計の根本改善まで求めるのは確かにスコープが広い。しかしユーザーの「見つけたタイミングで直したい」という方針は合理的だ。今後はスコープ外の指摘でも「今直すか？」と聞く方がいいかもしれない。

**機械的な型変更は想像より簡単だった。** 13ファイルに影響する変更と聞くと大仕事に感じるが、パターンが完全に一貫しているので実質30分程度で完了した。`d[key.name] as number` → `d.values[key.name]`、`...kindsData` → `values: kindsData` のパターンしかない。型定義の変更をためらう必要はなかった。

### 反省

レビューサイクルでCodexが最終サマリーを出さずに終了した。tscを走らせた後、判定を出す前にセッションが切れたように見える。Codexの実行時間にもう少し余裕を持たせるか、プロンプトで「最後に必ずLGTM/NOT LGTMを出力せよ」と強調すべきだったかもしれない。ただ、tscパスの確認自体は有用だったので完全に無駄ではなかった。

サブエージェントのCritical偽陽性率は今回100%だった（3/3）。レビューサイクルスキルの注意事項にある「Critical指摘は実コードと突き合わせて裏取り必須」がそのまま当てはまる。特にuseRefの安定性のような、Reactの基本的な動作保証に関する知識がサブエージェントには足りていない場面がある。

### worktree初実戦で発見した問題

今日が worktree 環境での初めての実開発だった。いくつか問題に遭遇した:

1. **`.env.local` vs `.env`**: `dotenv.config()` はデフォルトで `.env` を読むが、セットアップスクリプトは `.env.local` しか作っていなかった。Backend が `DATABASE_URL` が undefined で起動失敗。セットアップスクリプトに `.env.local` → `.env` コピーを追加した。

2. **Vite ポートの git 汚染**: `vite.config.ts` にポートがハードコードされていたため、worktree 用のポートに変えるには git を汚す必要があった。ユーザーの指摘で `process.env.VITE_PORT || 2460` に変更し、`.env` 経由で設定する方式にした。

3. **`git checkout -- .env.local` の罠**: コミット時に `.env.local` の差分を除外するために `git checkout` したら、worktree 用の DB/ポート設定が元に戻った。以降 Backend が起動しなくなり原因特定に時間を食った。

これらは全て「昨日スクリプトを作った時点で実際に使って確認しなかった」ことに起因する。スクリプト作成時の動作確認（setup → cleanup の往復）はしたが、「その worktree で開発サーバーを起動してコードを書く」ところまではやっていなかった。

### レビューサイクル

サブエージェント: Round 1 NOT LGTM → 修正後 LGTM。
Codex: Round 1 NOT LGTM → Round 2 NOT LGTM（Warning 1件）→ 修正。

Codex の指摘は的確だった:
- **trim 未適用**: 空白のみの body が通る。Zod スキーマに `.trim()` 追加で解決
- **catch 経路のハードコード英語**: `!res.ok` 側は i18n 化したのに、catch 側の `err.message` はブラウザのネイティブエラーメッセージがそのまま出る。throw/catch パターンを止めて直接 setError + return に変更

サブエージェントの Critical は `as unknown as Response` キャスト除去と optionalAuth のコメント追加。前者は有効（`const status: number = res.status` に変更）、後者は Info レベルでスキップ。

### CLAUDE.md の更新

ユーザーから「worktree では開発サーバー起動してよい」と指示があった。CLAUDE.md の最重要制限事項を更新。これはルールとしては「絶対禁止」→「条件付き許可」への変更で、worktree 環境ではポートが分離されているから安全、という判断は合理的。

### 設計で迷ったこと

optionalAuthMiddleware を新設するか、既存の authMiddleware にオプション引数を追加するか迷ったが、新設を選んだ。理由は2つ: (1) authMiddleware は全ての認証付きルートで使われており変更リスクが高い、(2) optional auth は contact 以外でも使いうる汎用パターンなので独立ファイルの方が見通しが良い。

Mobile で `apiClient`（Hono RPC client）ではなく `customFetch` を直接使ったのは、Hono client がレスポンス型を `{ status: 201 }` に narrowing するため `res.status === 429` が型エラーになるという問題のため。Web 側は `const status: number = res.status` で回避したが、Mobile エージェントは `customFetch` を選択した。一貫性の観点では統一すべきだが、実用上は問題ない。

## WAE APMログ分析 → エラー修正4件 — 日記との突き合わせが効いた

### やったこと

ユーザーの依頼でWAE-APMログ（直近1週間）を全量分析し、エラーの特定 → 日記との相関分析 → 修正を実施。

#### 分析フェーズ

8本のSQLクエリを並列実行して全体像を把握:
- 総リクエスト 20,385件、平均573ms（100% DB-bound）
- `activity-logs/sync` がトラフィックの88%（18,018件）
- 4xxエラー 963件、クライアントエラー 42件、5xx 1件

日記との突き合わせでエラーの発生日と変更内容の相関を特定:
- **3/22**: sync-engine大改修（failedリトライ追加）→ 400エラー急増
- **3/24**: Androidウィジェット実装 → database locked集中発生
- **3/26**: OTA多数 + foreground API追加 → 401ピーク（219件）
- **3/27**: セキュリティ監査バリデーション強化 → 400再発

#### 修正フェーズ

4つのExploreエージェントを並列で起動して調査し、以下を修正:

1. **ボットスキャンのWAEログ除外**: `/users/.env`等のスキャナーパスが`authMiddleware`で401を返し、WAEに記録されていた。`loggerMiddleware`にボットパス判定を追加し、401+ボットパスの組み合わせをWAE書き込みから除外

2. **sync 4xxでリトライ停止**: `postChunk`が400/500を区別せず全てthrowしていた。400（バリデーションエラー）を受けたアイテムを新ステータス"rejected"にマークし、getPendingSyncのクエリ対象から除外。SyncStatus型に"rejected"を追加し、domain/frontend-shared/mobile全レイヤーの型を更新

3. **SQLite busy_timeout追加**: Androidウィジェットの`WidgetDbHelper`がDB openごとに新規コネクションを開くが、`busy_timeout`未設定で即SQLITE_BUSY。Android（Kotlin）とiOS（Swift）両方のopenDatabaseに`PRAGMA busy_timeout = 5000`を追加

4. **.worktreesのテスト/lint除外**: `vitest.config.ts`と`biome.json`に除外パターン追加

#### 不要と判明した修正

- **`[updates]`成功ログのエラー送信**: 現行コードでは既に送信していない。3/22のWAEデータは旧ビルドからの残存
- **activityKind空名のバリデーション緩和**: 400エラーの発生日を2週間分確認したところ、3/15-3/20のウィジェット開発期に集中しており、3/24以降はほぼ消滅。sync APIの`min(1)`はドメインモデルとの不一致はあるが、現時点で実害なし

### 反省

#### "rejected"の扱いでユーザーに突っ込まれた

最初"rejected"にして永久に無視する実装にしたところ、ユーザーに「ユーザーは記録したのにサーバーに保存できなかったってことでしょ。再編集を促さないといけなくない？」と指摘された。正しい。データを黙って捨てるのはUX的に問題がある。

結果的には「もう発生しない問題」と判明して事なきを得たが、ユーザーの視点は「ユーザーのデータを守る」だった。自分の視点は「APIエラーを止める」。技術的に正しい修正でも、ユーザーのデータに影響する変更は常にUX面の考慮が必要。

#### activityKind空名のバリデーション不一致を見落としていた

sync APIの`min(1)`とドメインモデルの`z.string()`の不一致に自分で気づかず、ユーザーに「空文字を許容のはず」と指摘されて初めて調査した。さらに「ActivityLogの同期がエラーなのか、Activityの同期がエラーなのか知りたい」という的確な切り分け質問をされた。自分は400エラー全体を漠然と見ていたが、ユーザーはどのエンドポイントのどのフィールドかを正確に知ろうとしていた。問題の粒度を細かく見る習慣が足りない。

#### 日記との突き合わせアプローチは有効

エラー発生日と日記の変更内容を突き合わせることで、「なぜこのエラーが発生したか」の因果関係を高い確度で特定できた。特にactivityKind空名の400エラーが「ウィジェット開発期に集中 → もう発生しない」と判断できたのは、日記の存在があってこそ。ログ分析と開発履歴の突き合わせは定期的にやる価値がある。

#### 401大量発生の根本原因は未対処

622件/週の401エラーは「幽霊ログイン」（3/22日記で認識済み）が原因と推定されるが、ユーザーが「見送り」と判断。serverRefreshAndSync失敗時のonAuthExpired未発火は設計上の穴として残っている。オフライン起動との兼ね合いがあるので簡単には直せないが、次回のセッションで改めて提言すべき。

### 技術メモ

- **WAE SQLクエリの注意**: `!=`は使えない（`<>`を使う）、カラム名のエイリアスをORDER BYで使うと型エラーになることがある、`timestamp > NOW() - INTERVAL '7' DAY`でフィルタ
- **SyncStatus型拡張**: "rejected"を追加する際、domain/sync/syncableRecord.ts → packages/frontend-shared/hooks/types.ts → apps/mobile/src/components/daily/ と4箇所以上に分散した`_syncStatus`リテラル型を更新する必要があった。SyncStatus型をimportで統一すべきかもしれない
- **customFetchの401ハンドリング**: sync系の`postChunk`には401が到達しない（customFetch内でrefresh→リトライが透過的に行われる）。4xx→"rejected"の分岐で401が誤ってrejectedになる心配は不要

## 管理画面(admin-frontend)新設 — 「MVPだから」で品質を落としてはいけない

### やったこと

`apps/admin-frontend` を新設。ダッシュボード、ユーザー一覧、問い合わせ一覧/詳細、管理者Google認証。バックエンドに `/admin/` プレフィックスのAPIを追加。ADR 3件作成。seedにcontactsデータ追加、worktree-setupでseedデフォルト実行に変更。

### ユーザーに何度も叱られた

このセッションでは少なくとも3回、ユーザーから設計品質について指摘を受けた。

**1回目: 「バックエンドさすがにお粗末すぎでは？」**

最初の実装では `adminRepository` が users/contacts/activityLogs を直接クエリする神オブジェクトで、レイヤ構造（route→handler→usecase→repository）も無視していた。「たたき台」「MVP」という言葉に甘えて、既存コードベースが持っている品質基準を下回る実装を出してしまった。

ユーザーの指摘は的確だった: 「基盤の品質を落としてはいけない」。MVPであっても、レイヤ構造やDIパターンといったアーキテクチャの基盤は最初から正しく作るべき。後から直すコストの方が高い。

**2回目: 「handlerから直接repository呼んでるのはNG」**

リファクタリングして `adminHandler` を作ったが、usecaseを経由せずrepositoryを直接呼んでいた。handler→repository は層構造違反。指摘されて `adminUserUsecase`, `adminDashboardUsecase` を追加し、contactUsecaseにもメソッドを追加した。

**3回目: 「APMはちゃんとDI出来るように作ってね」**

WAE APMの取得を直接関数で書いていたところ、「いずれOpenTelemetryやDatadog使うかもしれない」と言われ、`ApmProvider` 抽象型を導入。これは正しい判断で、自分が「今動けばいい」で済ませようとしていたのを止めてくれた。

### 自分の判断で気づけたこと

レビューサイクルの修正後、自分から2つの設計問題を指摘した:
1. `query/adminDashboardQueryService` が `feature/admin/apmProvider` をimportしている依存方向の逆転
2. adminRouteでのインラインusecase構築がファクトリパターンに反している

ユーザーが「どう思う？」と聞いてくれたのがきっかけだが、これらは自分で気づけた。3回叱られた後で精度が上がったのか、単に最後だから余裕があったのか。おそらく前者——ユーザーの基準がわかってきて、同じ視点でコードを見られるようになった。

### Codexの運用で学んだこと

Codexの初回レビューはファイルを読みすぎてコンテキスト上限に達し、サマリーを出せなかった。2回目は対象を2ファイルに絞ったら正確な指摘が出た。3回目は14ファイルだったが「他のファイルは読まないでください」と明示したことで完走した。

Codexの有効な使い方:
- **対象ファイルを明示的に限定する**（「以下のファイルだけ」「他は読むな」）
- **レビュー観点も絞る**（規約一覧をプロンプトに含める）
- 逆に「コードベース全体をレビューして」は失敗する

3回目のCodexレビューで見つかった指摘は実際に有効だった: limit/offsetの未検証、WAE fetchのエラー耐性不足、Account IDのハードコード、`newXXX`命名の不統一。さらに既存バグ（`365 * 60 * 60` = 365時間）も発見した。レビュアーとしてのCodexの価値は、対象を絞れば高い。

### tscチェック対象漏れ

TanStack Routerの型衝突を避けるためにroot `tsconfig.json` のexcludeに `admin-frontend` を追加したが、`pnpm run tsc` でチェックされなくなることに自分で気づかなかった。ユーザーに「ちなみにtscチェック対象入ってる？」と聞かれて初めて気づいた。

これは「問題を解決するために別の問題を作った」パターン。tsconfig excludeを追加した時点で「じゃあこのアプリの型チェックはどこでやるのか」を考えるべきだった。mobileと同じpnpm filter方式で解決。

### 反省まとめ

今日のセッションの最大の教訓: **「MVPだから」「たたき台だから」は品質を落とす理由にならない**。特にレイヤ構造・DI・型安全性のようなアーキテクチャ基盤は、最初から正しく作るコストと後から直すコストの差が大きい。ユーザーはこれを明確に理解しており、自分はそこが甘かった。

次回は最初の実装段階で「既存コードの品質基準を満たしているか」を自問する。具体的には: レイヤ構造は正しいか、DIパターンに従っているか、型キャストを使っていないか、テスト/lint/tscのCI対象に入っているか。
