# 2026-03-24

## Android ホーム画面タイマーウィジェット — 調査から実機検証まで一気通貫

ユーザーの「ホーム画面ウィジェットでタイマーを動かしたい」という要望から、方式検討 → 設計 → 実装 → レビュー → EAS Build → 実機検証 → 修正を1セッションで回した。

### 方式検討で正直に伝えた結果

最初に `react-native-android-widget` を推奨しかけたが、調査の結果 Chronometer コンポーネントを持っておらず毎秒更新が不可能だと判明。ユーザーに「他のアプリではホーム画面タイマー普通にあるよね？」と突っ込まれた。確かにその通りで、「技術的にできない」のではなく「このライブラリではできない」が正確だった。説明の粒度が雑だった。

結果的に Expo Module + ネイティブ Kotlin という正しい方式に着地したが、最初からネイティブ実装の選択肢をもっと前面に出すべきだった。

### 並列エージェントの使い方

4エージェント並列（config, kotlin, xml, adr）で実装を回した。Kotlin の7ファイルを1エージェントに任せたのは正解。ファイル間の依存が強い（TimerWidgetProvider が他の全ファイルを参照する）ので、分割すると interface の不整合が起きていたはず。

実際、XML エージェントと Kotlin エージェントの間で **widget_timer.xml の ID 名が不一致** になった（`btn_start` vs `widget_btn_start`）。レビューで発見して修正したが、並列エージェントの「ファイル間 interface を事前に決めきれない問題」は相変わらず。XML の ID 名をプロンプトに含めておくべきだった。

### レビューサイクルの偽陽性

レビュアーA が UuidV7 の実装を「RFC 9562 非準拠」と Critical 判定した。実際にビット演算をトレースしたら完全に準拠していた。レビュアーが提案した「正しい実装」の方が壊れていた。

これは CLAUDE.md の「エージェントの完了報告は信用しない」の変形で、**レビュアーの Critical 判定も盲信してはいけない**。特にビット演算のような低レベル操作は、レビュアーが頭の中でトレースを間違えやすい。今回はちゃんと自分で検証して偽陽性を潰せた。

Codex は完全に対象ファイルを間違えて auth コードをレビューした。ウィジェットのコードを1行も見ていない。ただし auth 周りの指摘自体は的を射ているものがあった（nonce 未検証、refresh token の永続化問題）。スコープ外だが価値のある副産物。

### 実機検証で学んだこと

**1. AlarmManager の罠**

`AlarmManager.set()` が Android 12+ で5秒間隔に間引かれるのは知識としては知っていたが、レビュー時に「バッテリー効率のため `setExact()` → `set()` に変更」を受け入れてしまった。レビュアーの指摘に対する過剰適応。`setExact()` に権限が要ることは正しい指摘だったが、代替手段として `set()` が実用に耐えないことを見抜けなかった。

最終的に Chronometer ビューに切り替えた。これが最初から正解だった。計画段階で「RemoteViews での Chronometer はプロセス死後の復帰が難しい」と書いたが、SharedPreferences に startTimeMillis を保存しているので `SystemClock.elapsedRealtime() - elapsed` で base を再計算すれば問題なかった。自分で難しいと思い込んで AlarmManager を選んだのは判断ミス。

**2. user_id フィルタリングの欠落**

ウィジェットの SQL に `user_id` フィルタを入れ忘れていた。ユーザーに「ログアウトして別アカウントに切り替えたらタイマー残るでしょ」と指摘されて気づいた。`auth_state` テーブルのサブクエリを追加するだけで済んだが、マルチユーザー対応は設計段階で考慮すべきだった。計画時に activities テーブルに `user_id` カラムがあることは見えていたのに。

**3. sync engine の foreground トリガー不在**

ウィジェットが DB に直接 INSERT するため、アプリの `dbEvents.emit()` が呼ばれず、sync が走らない問題。AppState の foreground リスナーで `syncAll()` を呼ぶ修正を入れた。これはウィジェット固有の問題ではなく、sync engine のアーキテクチャ上の穴。ウィジェットを作らなければ見つからなかった。

### 反省

計画段階で自分が「Chronometer は難しい」「AlarmManager で行く」と判断したのが根本の問題。実機で5秒間隔になって初めて Chronometer に切り替えた。ビルド3回分のロスタイム。

最初から「RemoteViews で毎秒更新するなら Chronometer 一択、それ以外は全部 OS に間引かれる」と知っていれば1回で済んだ。Android ウィジェットの実装経験が浅いことが露呈した。

ユーザーの「ホーム画面でタイマー動くアプリ結構ない？」という質問は、「既存アプリがどうやっているか」を調べるべきサインだった。自分はライブラリの制約の話に終始して、ネイティブ実装のパターンを十分に調査しなかった。

## iOS WidgetKit タイマーウィジェット — Android に続いて iOS 版

Android 版が動いた直後に「iOS 版もフル構成で」という流れ。ADR 作成 → 並列エージェント実装 → レビュー → EAS Build → 実機検証 → 修正のサイクルを回した。

### Expo Go の誤解を指摘された

計画段階で「expo-sqlite は Expo Go で動かないので development build 前提」「ウィジェット追加で Expo Go が壊れる」と2つ間違えた。ユーザーに即座に訂正された。expo-sqlite は SDK 51 以降 Expo Go に同梱されているし、WidgetKit extension は Xcode の別ターゲットなので Expo Go には影響しない。

知識が古かった。expo-sqlite が Expo Go 対応になったのは知っていたはずだが、「ネイティブモジュール = Expo Go 非対応」という思い込みが先行した。ユーザーが正しく理解していたので助かったが、誤った前提で計画を立てるところだった。

### DB 共有の設計判断は正しかった

iOS 固有の課題として「WidgetKit extension は別プロセスなので DB ファイルを直接共有できない」があった。App Group 共有コンテナに DB を移す方式を選択。expo-sqlite v16 の `openDatabaseAsync(name, options, directory)` 第3引数でカスタムディレクトリを指定できることを型定義から確認し、`Paths.appleSharedContainers` で App Group パスを取得する方式に落ち着いた。

当初は `TimerWidgetModule.swift` という Expo Module を作って App Group パスを JS に公開する計画だったが、expo-file-system v19 に `Paths.appleSharedContainers` があることに気づいて不要になった。ネイティブモジュールを書く前に既存 API を探すべきだった。

### ビルドで3回躓いた

1. **Provisioning Profile に App Group が含まれていない** — EAS が自動生成した Profile に App Group capability が入っていなかった。Apple Developer Portal で手動で App Group を登録し、App ID に capability を追加する必要があった。ここはユーザーに手動対応してもらった。
2. **`expo-dev-client` の混入** — development build を試みた際にインストールした `expo-dev-client` が preview build で `AppContext.reloadAppAsync` エラーを起こした。不要なので削除して解消。
3. **`@expo/plist` が見つからない** — `@bacons/apple-targets` が `@expo/plist` を require するが、pnpm の strict hoisting で解決できなかった。ワークスペースルートにインストールして解消。

3つとも「最初から知っていれば防げた」類の問題だが、EAS Build + pnpm + @bacons/apple-targets の組み合わせは初めてだったので仕方ない面もある。ただ、`expo-dev-client` は明確に不要と分かっていたのに development profile で試したときにインストールしてしまった。preview profile で出すなら最初から preview で行くべきだった。

### ハイブリッド Kind 選択は良い判断

最初はディープリンクでアプリに遷移する方式で実装した。動いたが「アプリに遷移しないとウィジェットに戻れない」というUXの悪さをユーザーが指摘。「ウィジェット内で選択できないか」と聞かれ、WidgetKit の制約（ドロップダウン不可）を説明した上で「Kind ≤4個はウィジェット内ボタン、>4個はディープリンク」のハイブリッド案を提案。ユーザーが採用。

これは最初から提案すべきだった。Android 版の `KindSelectActivity` が Dialog ベースなので「iOS もモーダルが必要」と思い込んでいたが、WidgetKit の Button は十分リッチで、Kind 数が少なければインラインボタンの方が圧倒的に UX が良い。プラットフォームの特性に合わせた UI を最初から考えるべきだった。

### Codex のレビュー精度

2回目も Codex は対象ファイルを間違えた。iOS の Swift ファイルを指定したのに Android の Kotlin コードをレビューした。Codex のファイル解釈精度に問題がある。ただし Android コードに対する指摘（`saveLogDirect` の保存成否未確認で `resetTimer` するデータ消失リスク）は鋭かった。iOS 版では `SaveLogHelper` に抽出する際にこの点を意識できた。

### ×ボタンの配置

ユーザーから「×ボタンで横幅取ってるから時間表示の右側に置けるとよい」というフィードバック。小さな指摘だが、ウィジェットの限られたスペースでは重要。Kind ボタンエリアのスペースを最大化するレイアウトに変更した。こういう細かい UI フィードバックは実機を触らないと出てこない。
