## 置き場所の変遷

最初 `packages/domain/activity/` に置いた → ユーザーに「Activityのdomainじゃないでしょ」と指摘される → `packages/domain/ordering/` に移動 → ユーザーに「domain logicでもないじゃん」と指摘される → 最終的に `packages/utils/` を新設して配置。

正直、最初から `packages/utils/` が正解だと気づくべきだった。`generateOrder` は Activity にも ordering にも依存しない純粋な文字列ユーティリティなのだから。元が `apps/backend/lib/` にあったという事実をもう少し素直に受け取っていれば、「lib = util的なもの → domain外」と最初のステップで判断できたはず。2回のやり直しでユーザーの手間を増やしてしまった。

## 既存データとの互換性

ユーザーから「デプロイしたらどうなる？」と聞かれた。ゼロ埋め数値（`"000001"`）とlexical order（`"mhqrstab"`）が混在しても辞書順ソートは壊れないが、グループが分かれる可能性はある。Dexieマイグレーションで全件再採番する案も出したが、ユーザーは「壊れないなら良い」と判断。妥当だと思う。ユーザーは1人だけなので、並び替えUIを使えば自然に正規化される。

## ActivityKindのorderIndex

ActivityKindのorderIndexもゼロ埋めのまま残っている。バックエンドではnull扱いだったりと一貫性がないが、kindの並び替えUIがないので今は触らなくてよい。ただ将来的にkindの並び替えを実装する際には統一が必要。

## PR #134 マージ後のテスト修正

前回セッションのPR #134マージ後にCIテストが全滅していた件の修正。原因は2つ:

1. `packages/utils/package.json` を作り忘れていた。前回セッションで `packages/utils/` を新設したのに `package.json` を置かなかったため、pnpmワークスペースがパッケージとして認識できず、`@packages/utils` のimportが解決できない → 芋づる式に `uuid`, `dayjs`, `zod` 等の依存解決も失敗
2. `activityRepository.test.ts` のアサーションが古いまま。`generateOrder(null, null)` はランダム8文字を返すが、テストは旧ロジックの `"000001"` を期待していた

## 自己批判: 調査手法の失敗

最初「merge起因ではない、環境依存の既存問題」と誤判断した。原因は `git checkout dc80eed -- .` でmerge前の状態を再現しようとしたこと。このコマンドはワーキングツリーのファイル内容を戻すだけで、merge後に新規追加された `packages/utils/` のファイルは消えない。つまり「merge前」の正確な再現になっていなかった。`git checkout dc80eed` でHEADごと切り替えるべきだった。

結果としてmerge前後で同じ44ファイル失敗という誤ったデータに基づいて「問題なし」と結論してしまい、ユーザーに「テスト落ちてるって言ってるのに」と再指摘させることになった。gitの基本動作を理解していながら、検証の正確性を怠った。反省。
