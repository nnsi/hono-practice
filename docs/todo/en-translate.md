# 英語圏展開計画

## 概要

Actikoを英語圏（米国・EU・UK・カナダ・豪州）に展開するための対応事項まとめ。

## 対応言語

- 英語のみ（日本語 + 英語の2言語）

## タスク一覧

### 1. UI文字列のi18n化（エンジニアリング）

- i18nライブラリ導入（react-i18next が React Native 対応もあり有力）
- 翻訳キーの namespace 設計（`auth.login`, `activity.create` 等）
- 文字列抽出・置換（~800ユニーク文字列、法務文書除く）
- 翻訳JSON作成（ja / en）
- 日付・数値のロケール対応（`dayjs().format("YYYY年M月")` 等）

#### 規模感

| 領域 | ファイル数 | 文字列数（重複除去後） |
|------|-----------|----------------------|
| Frontend (Web) | 87 TSX | ~600 |
| Mobile | 79 TSX | ~350（Webと50%重複） |
| frontend-shared | hooks + legal | ~800（うち法務文書1,600+） |
| Backend API errors | 数ファイル | ~5 |
| Zodバリデーション (packages/types) | 8ファイル | ~16 |
| ドメイン (debtFeedback等) | 1ファイル | ~10 |

- Web/Mobileで~50%の文字列が重複 → 共通化すれば翻訳は1セットで済む
- 文字列抽出はエージェント並列で機械的に進められる部分が大きい
- 動的文字列（テンプレートリテラル）が多数あり、単純置換では済まない箇所あり

### 2. 論理削除→物理削除に変更（エンジニアリング）

- 現状アカウント削除ボタンはあるが論理削除
- GDPR「忘れられる権利」は原則物理削除を要求
- Apple / Google Play のストア要件としてもアプリ内アカウント削除が必要
- 対象: user, activity, activity_log, task, goal 等の全関連データ + R2ファイル

### 3. 英語版 利用規約 + プライバシーポリシー作成（法務）

- 日本語規約の翻訳では不十分。英語圏向けに別文書として作り直す必要がある
- Termly, iubenda 等のテンプレートサービスを使えば安く済む
- 文書構成: 利用規約1つ + プライバシーポリシー1つ（各言語）
- 1文書内に地域別セクションを設ける形が一般的

#### 日本向けと英語圏向けの差分

| | 日本向け（現状） | 英語圏向け（必要） |
|---|---|---|
| 準拠法 | 日本法 | 対象地域の法律（or 選択条項） |
| 特商法表記 | 必要 | 不要 |
| GDPR条項 | なし | EU向けなら必須 |
| CCPA条項 | なし | カリフォルニア向けなら必須 |
| 仲裁条項 | なし | 米国向けではほぼ標準 |
| データ削除権 | なし | GDPR/CCPA両方で必要 |

#### 各条項の補足

- **GDPR**: EU/EEA + UK向け。データ処理の法的根拠、DPO連絡先等。UK GDPRもほぼ同内容
- **CCPA**: カリフォルニア州向け。GDPR対応していればほぼカバーできるが「個人情報を売却しない」の明示的通知が追加で必要。Actikoは広告・データ売却なしなので「売却しません」と書くだけで済む
- **仲裁条項**: 米国での集団訴訟リスク対策。紛争を裁判所ではなく仲裁機関で解決する旨 + 集団訴訟放棄。Termly等で自動生成される
- **GDPR対応でカナダ（PIPEDA）・豪州（Privacy Act）もほぼカバー可能**

### 4. プライバシーポリシーにCookie使用の記載追加（法務）

- Actikoのリフレッシュトークン保存は strictly necessary cookie に該当 → Cookie同意バナーは不要
- ただしポリシーに「認証目的でCookieを使用している」旨の記載は必要
- Google Analytics等を将来導入する場合はその時点でconsent実装が必要

## 進め方

1. **今やる**: UI文字列のi18n基盤整備 + 英語翻訳
2. **展開前にやる**: 法務文書の作成 + 論理削除→物理削除
3. **法務文書はTermly等のサービスを利用する**（自力翻訳はリスク高）

英語圏リリースは法務文書が揃うまでブロックされる。

## Timezone対応

### 現状

- `createdAt` / `updatedAt` / `deletedAt` / 課金期間などの瞬間時刻は `timestamp with time zone` で保持している
- `activity_log.date` / `task.doneDate` / `goal.startDate` などの業務日付は `date` 型で保持している
- このため、英語圏対応のために既存データを全面的にUTCへ移行する必要はない
- 一方で実装はJST固定ユーティリティ、UTCベースの処理、端末ローカル時刻ベースの処理が混在している
- `user.timezone` のような保持先はまだ存在しない

### 方針

- 業務日付は今後も「そのユーザーのローカル日付」として `YYYY-MM-DD` / `date` で扱う
- 瞬間時刻は既存どおり `timestamp with time zone` を使う
- タイマー記録については、現時点ではサーバサイドで厳密な発生瞬間を持たず、日付ベースのログとして扱う
- `recordedAt` / `startedAt` / `endedAt` のような追加基盤は現時点では YAGNI とする

### 進め方

1. 日付操作の統一
   - `dayjs()` / `new Date()` / `toISOString().split("T")[0]` / JST専用helper の混在をやめる
   - `today` / `startOfDay` / `endOfDay` / `formatDate` を共通化する
2. `user.timezone` を追加
   - DB / Domain / API に timezone を追加する
   - 既存ユーザーの初期値は `Asia/Tokyo` とする
3. 各層で timezone を使用
   - backend の日次集計・月次集計
   - frontend / mobile の「今日」「今月」「初期日付」
   - date picker や task / goal / activity log 作成時の日付決定

### 補足

- このアプリは日付選択で `ActivityLog` を追加できるため、「旅行中の発生瞬間をあとから厳密再解釈する」要件は現時点では優先度が低い
- 英語圏対応で先に解くべき問題は、絶対時刻の完全再現ではなく「どのユーザーにとっての今日か」を一貫して決めること
